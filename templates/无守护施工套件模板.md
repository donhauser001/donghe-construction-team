# 无守护施工套件模板

> 四份产物的填空骨架（规则见 `playbooks/08-无守护模式.md`，本模板不重复规则）。
> 落盘位置：启动确认 / 心跳 / 总报告 → `docs/施工日志/<今天>/`；队列 → `docs/任务卡/`。

---

## 产物 1 · 无守护启动确认（`无守护启动-HHMM.md`）

### 元信息

| 字段 | 值 |
|---|---|
| 触发时间 / 触发原话 | YYYY-MM-DD HH:MM · "（用户原句）" |
| 执行施工队 / 操作系统 | A 队 · macOS |

### 启动自检（任一 ❌ 拒绝启动）

| 检查项 | 结果 |
|---|---|
| A1 项目已接入东合 | ✅/❌ |
| A2 可安全单队施工（或已建半守护握手协议） | ✅/❌ |
| A3 git 工作区 clean（或用户授权 preflight） | ✅/❌ |
| A4 四必填已确认（时长 / 范围 / commit-push / 阻塞策略） | ✅/❌ |
| A5 持续运行契约已填写 | ✅/❌ |
| A6 base 同步 push 成功或已获准降级（如启用） | ✅/❌ |
| W1 在途阻塞/待审卡（警告项） | ✅/⚠️ 跳过继续 |
| W2 防睡眠可用（警告项） | ✅/⚠️ 记录风险 |

### 四必填答复

| 项 | 用户答复 |
|---|---|
| Q1 时长 | X 小时（起 HH:MM → 止 HH:MM） |
| Q2 范围 | 清空在途卡 / 完成某模块 / 自由发挥 / 参考 ______ 推进 |
| Q3 commit/push | A 同步 base 一次 + push auto 分支（默认）/ B 只 commit 不 push / C 都不做 |
| Q4 阻塞 | A 跳过继续（默认）/ B 保守继续 / C 停下等回 |

### 持续运行契约

| 契约项 | 值 |
|---|---|
| 时间窗 / 总时长 | HH:MM → HH:MM · X 小时 |
| 队列下限 | ≥ 3 张（4h 以上 ≥ 5 张） |
| 业务参考源 | 无 / 仓库 / 文档：______ |
| base 分支 / base 同步结果 | xxx · 成功/失败/未启用 |
| auto 分支 | `auto/YYYY-MM-DD-HHMM` |
| 防睡眠命令 / PID | `caffeinate -i -t <秒> &` · pid |
| 技术债账本 | `docs/技术债账本.md` |
| 合法退出原因（仅限） | 时间到 / 用户回来 / 连续 3 阻塞 / 护栏违规 / 不可恢复异常 |

| # | 任务标题 | 半径 | 子项目 | 文件域 | 状态 | 卡路径 | Commit | Push | 备注 |
|---|---|---|---|---|---|---|---|---|---|
| 1 | xxx | M | backend | `backend/src/xxx/` | 待做 | — | — | — | — |

### 补给记录

| 时间 | 触发原因 | 补给来源（模块/下一阶段/路线图/参考源/蓝图下放/QA） | 新增卡 |
|---|---|---|---|

### 蓝图自动下放记录

| 时间 | 来源 | 下放结果（路线图节点 → 焦点 → 卡） | 待复核点 |
|---|---|---|---|

### 空跑检查

| 时间 | 有实物产出？ | 证据 | 处置（继续/补小卡/违规停止） |
|---|---|---|---|

### 执行进度

完工 N · 跳过 M · 阻塞 K · 待做 L · 当前 #N · 已用 X:XX / 总 Y:00 · 连续阻塞 0 · 连续空跑 0 · 最近 push 状态

### 退出原因（最终填写 · 队列清空不是合法选项）

时间到 / 用户回来 / 连续 3 阻塞 / 护栏违规 / 不可恢复异常 / 连续空跑 —— 退出时刻 HH:MM

## 产物 3 · 心跳（`自动施工心跳.md`，每 30 分钟追加一行）

```
[HH:MM] 已用 X:XX / 总 Y:00 · 完工 N · 跳过 M · 阻塞 K · 当前 #NNN-任务名 · 状态 🔨 · 分支 auto/... · push ok
[HH:MM] ⚠️ 异常 · <一行描述>          ← 异常事件穿插追加
```

多队半守护时行尾追加 `· 握手 X队：无冲突 / 已避让`。退出时写尾部块：退出时间 / 原因 / 最终数据 / 总报告路径。

---

## 产物 4 · 总报告（`自动施工总报告-HHMM.md`）

1. **元信息**：起止 / 给定时长 / 实际时长 / **是否跑满** / 退出原因 / 关联三文件
2. **分支与推送**：base 同步结果 · auto 分支 push 成功/失败次数
3. **数据汇总**：初始候选 / 补给新增 / 完工 / 跳过 / 阻塞 / commit 数 / 心跳数 / 技术债数 / 下放次数
4. **完工清单**：卡 + commit hash + push + 证据简摘
5. **跳过 / 阻塞清单**：原因 + 已尝试动作 + 建议
6. **补给与下放记录**（从队列文件汇总）
7. **技术债摘要**：ID / 类型 / 推荐方案 / 已采取动作 / 回滚方式
8. **待用户复核清单**（按优先级）
9. **Commit 序列**：`git log --oneline <base>..auto/...`
10. **退出自证**：是否跑满？未跑满的硬阻塞？补给尝试了哪些来源？遇决策是否停下（必须"否，已记债继续"）？
11. **下一步建议**：审 diff → squash merge / cherry-pick / 废弃（附命令）

### 用户回来的标准回复

```
你回来了。无人值守跑了 X 小时（HH:MM ~ HH:MM），给定 Y 小时。
数据：完工 N · 跳过 M · 阻塞 K；分支 auto/...；推送 <状态>
技术债新增 Z 条；蓝图下放 N 次（均待复核）
总报告：docs/施工日志/<今天>/自动施工总报告-HHMM.md
最需要决策的事：1. xxx  2. xxx
建议：先审 origin/auto/... 的 diff，再决定 squash merge / cherry-pick / 废弃。
```
