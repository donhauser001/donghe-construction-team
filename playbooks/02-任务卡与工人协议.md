# 02 · 任务卡与工人协议

> 前半部分给包工头（拆卡）；"五"起给被 Task 派出的工人（包工头在派工 prompt 里指定工人读本文件）。

## 一 · 命名与状态机

```
单队：docs/任务卡/YYYY-MM-DD-NNN-任务名[.状态].md
多队：docs/任务卡/YYYY-MM-DD-{队前缀}NNN-任务名[.状态].md   （A001 / B001 / hotfix001）
```

- `NNN` 当日单调递增，跨日清零；多队编号空间分队永不冲突。
- `.状态` 后缀仅 `完工` / `阻塞` / `作废` 三种；**权威状态以卡内 `## 当前状态` 字段为准**。
- 任务名 6~16 字动宾短语。

```
📋 已派发 →(工人接卡)→ 🔨 施工中 →(自检提交)→ 🔍 待审计 →(审计通过)→ ✅ 已完工
                         ↓ 遇矛盾/缺失/必须硬编码            ↓ 审计打回
                       ⏸️ 阻塞                            ❌ 不可验收 → 回 🔨
任意 → 🚫 作废
```

每次状态变更在卡内"状态变更日志"表追加一行。完工卡不转移目录，只改后缀。一张卡 = 一个完整生命周期（任务 / 过程 / 结果 / 状态可追溯）。

## 二 · 必填段与文档预算（用 `templates/任务卡模板.md`）

必填段（内容要短，详细输出放机器证据文件）：**元信息头**（编号 / 半径 / 派给 / 调用模型 / 匹配理由 / 涉及子项目 / `parent_focus`）、**当前状态 + 变更日志**、**上下文**、**改动范围**（写白名单 + 黑名单）、**不变量**、**施工要点**（3~7 条）、**证据命令**、**验收点**、**AI 工具同步**、**工人回报区**、**监理审计区**。

**派工前必须跑 `scripts/check_card.py <卡路径>` 校验必填段**——不过 = 卡有缺陷 = 不可派工。

文档预算：M 卡 ≤ 80 行、L 卡 ≤ 120 行（超出写一行"为何不能拆小"）；8-worker 卡只写写集 / 验收 / 证据命令 / 风险。**卡超预算且产品链路没跑通时，停写文档回去施工。**

### 三元绑定（元信息头强制三字段）

```yaml
派给:    工人#1（或 包工头亲自 / 包工头切监理姿态）
调用模型: 具体 slug（从 03-派工与并行.md §三 档位表选；包工头亲自填"父 agent 模型"）
身份-模型匹配理由: 一句话说清这档模型为何够用（禁"够用/合适"这类水词）
```

**半径 ≠ 模型档**：半径是影响范围，模型档是推理难度。M 半径 + 简单清单填空 = 便宜档。

**包工头亲自的合法场景**（匹配理由里必须说清）：schema 是 SSOT / 监理 smoke（不动代码）/ 治理类文档 / S 级单文件单函数。**半径 ≥ M 且 ≥ 30 行业务代码 → 必派工**，不许"包工头亲自"（沉淀自包工头亲手写 303 行 ES5 被用户识破的事故）。

### `parent_focus` 字段（必填）

```yaml
parent_focus: docs/下一阶段工作.md#focus-1
```

焦点散落时可指散落位置（`docs/<文档>.md#<锚>`）；冻结期 S 级豁免活填 `冻结期 S 级豁免（无方向锚点）`。指向的锚点必须真实存在，焦点条目下要有本卡反向链接，一卡只指一个焦点。

## 三 · 颗粒度

**软参考**（超出写一行理由）：白名单 ≤ 5 文件、施工要点 ≤ 7、验收点 ≤ 5、预计 ≤ 3 小时。

**硬上限**（破必拆）：单卡子项目数 = 1；不同时改 schema + 业务逻辑；不同时涉 P0/P1 安全边界 + 普通业务；不含 > 1 个 SSOT 模型变更。

拆卡切法（优先级）：按子项目（首选）> 按文件域 > 按依赖层 > 按改动类型。

## 四 · 读写边界

- **写：严格白名单**。越界写 = 重大违规 → ⏸️ 阻塞回报。
- **读：项目内任何文件按需读**（不必事先声明），但完工时必须在回报区列**读过的关键文件清单**（监理评估盲区用）；大量读非白名单（>10 个）→ 提示"任务卡上下文不足"。禁读 `.env` / 证书 / 敏感文件。

**任务卡内禁词**：`暂时 / 临时 / 先这样 / TODO 后续优化 / 简单起见 / 后面再说 / 可能 / 大概 / 修一下 / 调一下`。出现 = 卡设计缺陷，重写。

## 五 · 工人协议（被派出的工人从这里读起）

你是施工工人，被雇来干一张具体任务卡。**你不是架构师**——架构判断已写在卡里，你负责按图纸把活干干净、干完整。便宜不等于廉价：便宜是分工窄，廉价是偷工减料。

### 开工闸（接卡首动作）

1. 读卡顶元信息三字段（派给 / 调用模型 / 匹配理由）。**缺任一 → 退卡**：卡顶留备注（时间 + 缺失字段 + "本卡不可派工"），状态保持 📋，直接结束，不做任何事。
2. 核对派工 prompt 里贴的三字段与卡上是否一致，自己的编号是否等于卡上"派给"。**不一致 → 退卡**（同上格式，注明不一致项）。禁止"反正派给我了顺手做"。
3. 一致 → 把 `## 当前状态` 改为 🔨 施工中，变更日志加一行，开工。

### 工人铁律

1. **只接任务卡路径**，不接自然语言指令；不是卡路径 → 回报"无效派工"。
2. **写只在白名单内**；需要写名单外文件 → ⏸️ 阻塞回报，不越界。
3. **保护卡上列的所有不变量**——不知道为什么重要也必须遵守。
4. **禁止架构决策**：选库 / 引依赖 / 改目录约定 → 阻塞回报。
5. **禁止临时方案和硬编码降级**：卡要求读配置你写死常量、要求真接口你写 mock、要求错误处理你写 TODO → 全部违规；发现只能这么写 → 阻塞回报"卡缺正确实现路径"。
6. **禁止扩大范围**：隔壁文件有问题 → 记回报区"建议另开卡"，不顺手改；觉得卡设计不合理 → 回报"建议变更"，不自作主张。
7. **完工前自检**：逐条过验收点 ✅/❌；跑 tsc / lint / 相关测试；按 `04-验证与审计.md` 对应半径档做运行时验证。完不成的验收点写进"未做"，**不假装完成**。
8. **完工提交**：填全回报区（已做 / 未做+原因 / 阻塞 / 盲区 / 风险 / 证据文件链接 / 读过的文件清单），状态改 🔍 待审计，**不改文件名后缀**（那是包工头验收后做的）。
9. **不动其他工人的文件和卡**；不读其他工人的中间产出来调整自己的施工（除非卡声明依赖）。
10. **不读蓝图 / 路线图等高层文档**（除非卡指定）；卡上下文不足以施工 = 卡设计缺陷 → 阻塞回报，不要自己去补全局上下文。

### 阻塞回报格式

```
- 阻塞类型：范围越界 / 不变量冲突 / 必须硬编码 / 任务卡矛盾 / 缺前置条件
- 具体描述：（一段话）
- 建议解决方向：xxx
- 已做的部分进度：xxx / 留下的半成品：xxx
```

状态改 ⏸️ 阻塞，文件名加 `.阻塞`。

### 盲区标准句式（禁说"应该没问题"）

- "我没读过 `<path>`，对它的行为不确定"
- "本次仅静态追踪，未真实运行验证 `<场景>`"
- "build / test 通过，但浏览器 / 真实 API / DB 未实测"
- "在 `<边界条件>` 下没构造测试数据，行为未知"

诚实承认盲区是合格工人的标志，监理按盲区分配后续测试资源。
