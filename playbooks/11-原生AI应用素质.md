# 11 · 原生 AI 应用素质

> 适用：评估项目是否具备"原生 AI 应用"素质；新功能开发时同步设计 agent 工具。
>
> 这是东合施工队对项目的**默认期待**：所有项目都应该顺应智能化趋势。

## 一、原生 AI 应用的五条素质

| # | 素质 | 含义 |
|---|---|---|
| 1 | **可读** | agent 能用结构化方式获取项目所有数据（不是只能从 UI 截屏） |
| 2 | **可写** | agent 能用工具完成用户在 UI 上能做的所有操作（不是只能"看不能动"） |
| 3 | **可解释** | agent 能拿到操作结果的完整反馈（不是"盲调用"） |
| 4 | **可撤销** | agent 的写操作能被撤销 / 回滚（与 UI 撤销系统统一） |
| 5 | **可发现** | agent 能通过工具清单 / schema 自动发现"我能做什么"（不是依赖人写说明书） |

---

## 二、五条素质的可检查清单

### 素质 1：可读

- [ ] 项目所有的核心实体（用户、订单、内容、配置…）都有对应的"读取" agent 工具
- [ ] 工具返回的是结构化数据（JSON / 结构化对象），不是 HTML / 截图
- [ ] 工具支持过滤 / 分页 / 排序参数
- [ ] 工具能返回关联数据（不是必须多次调用拼接）
- [ ] 大对象支持分块 / 摘要返回（避免炸上下文）

### 素质 2：可写

- [ ] 用户能在 UI 上做的每件事，都有对应的"写入" agent 工具
- [ ] 工具的入参与 UI 操作的输入语义一致（不是只暴露底层 ID）
- [ ] 工具有完整的入参校验（错误返回结构化错误，而不是 500）
- [ ] 工具支持幂等键 / 去重（避免重试导致重复写入）
- [ ] 工具有清晰的副作用声明（"会触发通知 / 会扣款 / 会改状态"）

### 素质 3：可解释

- [ ] 工具调用后返回操作结果的状态（成功 / 失败 / 部分成功）
- [ ] 失败时返回结构化错误码 + 可读原因 + 修复建议
- [ ] 写入工具返回写入后的最新状态（而不是只回 "ok"）
- [ ] 关键操作返回受影响的资源 ID 列表（便于二次操作）
- [ ] 工具有日志可追溯（调用时间 / 调用者 / 入参 / 结果）

### 素质 4：可撤销

- [ ] 重要写入工具有对应的撤销工具（或者天然幂等可重写覆盖）
- [ ] 撤销与 UI 的撤销系统走同一个事件 / 历史日志
- [ ] 不可逆操作（删除、扣款、发通知）有明确的"二次确认 / 试运行"参数
- [ ] 批量操作支持"试运行"模式（dry-run），返回会发生的变化但不实际执行

### 素质 5：可发现

- [ ] 项目有一份 agent 工具清单文档（可以在 `docs/项目知识库.md` 或单独 `docs/agent-tools.md`）
- [ ] 工具命名遵循统一约定（动宾结构、含主语、可猜测）
- [ ] 工具 schema 有完整描述（参数语义、约束、示例）
- [ ] 工具有"列出我能做什么"的元工具（可选，但加分）
- [ ] 新增 / 修改工具时同步更新清单（任务卡硬要求）

## 三、新功能开发的硬规则

任何 M / L 级新功能任务卡，**必须**包含以下两段：

### `## AI 工具同步`

- 列出本次需要新增的 agent 工具
- 列出本次需要修改的 agent 工具
- 如果"无"，必须明确写"无：本功能不引入新的用户操作"

### `## 验收点` 中加入 AI 工具验收

```
- [ ] UI 操作 X 已实现
- [ ] 对应的 agent 工具 X 已实现
- [ ] agent 工具 X 已真测：（贴调用证据）
- [ ] UI 操作 X 与 agent 调用结果完全等价
```

**任何一条不满足，任务卡不允许进入 ✅ 已完工**。

## 四、典型违规场景（工地禁忌候选）

发现以下情况，立刻按 `playbooks/12-工地禁忌识别与沉淀.md` 沉淀：

| 反例 | 问题 |
|---|---|
| 加了一个 UI 按钮，agent 工具没加 | 用户可做、agent 不可做 → 不可读不可写 |
| agent 工具叫 `update_data`，入参是 `{"id":"xxx","field":"xxx","value":"xxx"}` | 不可发现；语义不可解释；agent 不知道"data"是什么 |
| 写入工具调用后只返回 `{"ok":true}` | 不可解释；agent 不知道写成什么样 |
| 删除工具没有"二次确认 / dry-run"参数 | 不可撤销 |
| agent 工具实现了，但 UI 走另一条路径（绕过 agent 工具能调用的 service 层） | SSOT 分裂 + 行为不一致 |
| agent 工具的入参直接是数据库字段名 | 不可发现 + 暴露内部细节 |
| 工具清单文档落后于代码 | 不可发现 |

## 五、评估项目当前 AI 化程度

包工头在工地踏勘时，可选地评估一次：

```
原生 AI 应用素质评估：
  可读：N/总数项已满足
  可写：N/总数项已满足
  可解释：N/总数项已满足
  可撤销：N/总数项已满足
  可发现：N/总数项已满足

整体评估：
- ✅ 良好：核心功能基本满足五条
- ⚠️ 待提升：缺关键工具 / 工具清单
- ❌ 重大缺口：UI 与 agent 工具能力分裂

建议：
- 列出 P0 / P1 缺口
- 建议在《下一阶段工作.md》纳入"AI 化提升"主线
```

写到《工地踏勘报告》里。

## 六、与监理审计的关系

监理审计九问的 Q4（AI / agent 工具对等）就是这份 playbook 的浓缩。

L 级任务卡审计时，监理必须按本 playbook 的"五条素质可检查清单"对应项检查；任意一条不满足直接 P1。
