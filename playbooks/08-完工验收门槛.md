# 08 · 完工验收门槛

> 适用：包工头准备声明"完成"前的最后一道闸。
>
> 不达门槛 = 不许说"完成"，必须说"部分可验收 + 待办清单"或"未完成"。

## 一、什么是"完成"

**"完成" = 用户能可靠地用上这次改动 + 项目知识完整沉淀 + 没有埋技术债**。

不是：
- ❌ build 通过 = 完成
- ❌ 测试通过 = 完成
- ❌ 我把代码写完了 = 完成
- ❌ 我觉得应该没问题 = 完成

---

## 二、按半径分档的硬条件集

### S 级 · 2 条硬条件

- [ ] `tsc` / `lint` 无新增错误
- [ ] 当日施工日志已记录（可以是 1~2 行）

### M 级 · 4 条硬条件

S 级所有 + 以下：

- [ ] 任务卡所有验收点逐项 ✅，且都有 ≥ 第 2 档证据（见 `06-真实运行时验证.md`）
- [ ] 任务卡 `## 当前状态` = ✅ 已完工，文件名加 `.完工` 后缀

### L 级 · 8 条硬条件

M 级所有 + 以下：

- [ ] 监理审计无 P0 / P1（详见 `07-监理审计清单.md`）
- [ ] AI / agent 工具对等性已验证（详见 `playbooks/11-原生AI应用素质.md`）
- [ ] 老数据兼容性已验证（构造或找到一条最早的真实记录跑一遍）
- [ ] 失败模式至少 1 个边界场景已验证（半失败 / 网络断 / 越权 等）
- [ ] 跨端 / 跨视图一致性已验证（受影响的所有 client 各跑一次）
- [ ] 路线图 + 项目知识库已按 `playbooks/10-文档对齐回写.md` 更新
- [ ] 是否触发新的工地禁忌已确认 + （若是）已沉淀到 `docs/工地禁忌.md`
- [ ] 二次审计已通过（如果第一次审计有打回）

---

## 三、未达门槛的标准措辞

不允许在"未达门槛"时说"完成"。允许的标准措辞：

### 部分可验收（核心可用、有非核心待办）

```
本次任务部分可验收：

✅ 已验证可用：
- xxx
- xxx

⏳ 未完成 / 未验证（已记入新任务卡）：
- xxx → 任务卡：docs/任务卡/2026-05-06-007-xxx.md
- xxx → 任务卡：docs/任务卡/2026-05-06-008-xxx.md

是否接受当前状态作为本轮节点？
```

### 阻塞回报（无法继续）

```
本次任务进入阻塞：

阻塞原因：xxx

我已经做了：xxx
我留下的半成品：xxx
我的建议解决方向：xxx

待你决策：
- A. 修改需求规避阻塞
- B. 提供必要前置条件
- C. 作废本卡，换方向
```

### 未达门槛但用户催促

```
我注意到你希望尽快交付，但本次有以下未达门槛项：
- xxx
- xxx

如果强行声明"完成"，会有以下风险：xxx

我可以：
- A. 严格按门槛继续做，需要 X 时间
- B. 接受"部分可验收"，把待办登记成新任务卡
- C. 你确认愿意承担风险，我标注"用户授权降级验收"，继续

请选择。
```

---

## 四、永久不可降级项

不管用户怎么催，下面几项**任何半径都不允许跳过**：

- 真实运行时验证（哪怕只是 S 级"看一眼"）
- 当日施工日志记一笔
- 完成必须有证据
- 工地禁忌识别（发现一条就立刻沉淀）

---

## 五、自检脚本（每次声明完成前问自己）

口头自检 5 问：

1. 我是不是真的跑过、看过、刷过、试过？
2. 我贴的证据是真的从命令 / 日志 / 浏览器拿到的，还是我"复述"的？
3. 我有没有任何"应该 / 大概 / 应该没问题"的潜意识？
4. 路线图和项目知识库是不是已经知道这件事发生过了？
5. 如果三天后用户找我说"上次那个事 X"，我能从任务卡 + 施工日志找到完整链路吗？

**任意一个回答"否"或"不确定"** → 不允许声明完成。

---

## 六、声明完成后的最后一件事

完工的最后一件事 = **更新文档**：

- 路线图节点状态（如适用）
- 当日施工日志收口
- 项目知识库追加（如有沉淀）
- 工地禁忌追加（如发现新反例）
- 蓝图（如本次工作改变了未来方向）

详见 `playbooks/10-文档对齐回写.md`。

**禁止"完成→走人"**。完成 = 写完文档之后的那一刻。
