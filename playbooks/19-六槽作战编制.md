# 19 · 六槽作战编制

> 适用：用户指出“包工头太忙 / 工人等包工头 / 并发上限约 6 个子 agent / 需要文书、巡检、审计、工人分槽”时。目标是解决控制面过载，不是单纯增加工人。

## 一、标准编制

```
包工头（父 agent）
├── 文书 agent
├── 巡检 agent
├── 审计 agent
├── 工人 #1
├── 工人 #2
└── 工人 #3
```

包工头是父 agent，不占 6 个子 agent 槽。6 个子 agent 槽固定分配为 3 个职能槽 + 3 个施工槽。

## 二、角色边界

| 角色 | 核心职责 | 默认禁止 |
|---|---|---|
| 包工头 | 只裁决红色事项、派发节奏、冲突取舍、最终汇报、最终 stage/commit/push | 同时扮演文书 / 巡检 / 审计 / 工人；阅读未压缩的长回报 |
| 文书 | 写瘦身任务卡、维护战情板、更新施工日志、准备预备任务队列、同步 SSOT | 写业务代码、最终审计、替工人判完工、把整张卡贴给包工头 |
| 巡检 | 读 git status、在工任务卡、脏区、冲突、阻塞，定时报红/黄/绿态势 | 改文件、stage、直接裁决、贴完整噪音日志 |
| 审计 | 对刚交付的卡做白名单、diff、机器证据、越界核验，给通过 / 打回结论；小问题等对应工人闲置后回炉，大问题转新卡 | 直接修代码、扩大范围、最终推送、**打断工人当前在工任务卡**、**让工人空转等审计** |
| 工人 | 按任务卡施工、自检、生成/链接机器证据、提交待审计；交完即领新卡，不等审计回收 | 越界改文件、改任务目标、直接找用户拍板、**交卡后停下来等审计结论** |

## 三、信息流

```
用户 / 总目标
  ↓
包工头拆优先级
  ├─ 文书：补卡 + 战情板 + 下一批弹药
  ├─ 巡检：每 10-15 min 报现场态势
  ├─ 审计：谁交卡审谁，不批量堆积
  └─ 工人 #1/#2/#3：按卡施工
        ↓
审计给结论 → 包工头裁决 → 文书回写 → 包工头最终汇报
```

工人不等待包工头口头解释；任务卡必须足够清楚。遇到不清楚，工人先把问题写入卡的“阻塞 / 疑问”段，再由巡检或审计带回包工头。

## 四、节奏规则

- **文书弹药线**：任何时候保持至少 3 张可领取预备卡；低于 3 张，文书优先补卡。
- **巡检心跳线**：每 10-15 分钟报告一次：在工卡、待审卡、脏文件、冲突、阻塞、下一风险。
- **信息压缩线**（v0.4.9）：所有子 agent 回包工头前必须压缩为状态 + 证据链接 + 需要裁决的问题；不许贴长文档、长 diff、长测试输出。
- **审计回收线**：谁交卡审谁，审计与工人施工**解耦**——审计干活时工人继续接新卡，**不让工人停下来等结论**。审计待审 ≥ 3 张时包工头**警觉而非暂停派工**，前提是文书弹药线还在（≥ 3 张预备卡）。
- **工人不空转线**（v0.4.8 新增 · 一级原则）：工人交完一张即领新卡，不等审计。审计发现的：
  - **小问题**（同任务卡内、不升级半径、不动写集边界）→ 写入"待回炉"清单，等对应工人闲置后再派回去修；
  - **大问题**（跨任务卡、半径升级、跨写集、动 schema / 资金 / 权限 / 安全 / 生产）→ **直接拆成新任务卡**，按新任务派工，不打断当前工人当前卡。
- **工人施工线**：3 个工人稳定施工；只有当文书 / 巡检 / 审计空闲且风险低时，才临时转为第 4-6 工人。
- **提交线**：最终 stage / commit / push 只由包工头做；审计只给白名单和结论。

### 审计 → 工人的回流路径（v0.4.8）

```
审计发现问题
  ├─ 小问题 → 「待回炉」清单 → 等 owner 工人闲置 → 文书写「补丁卡」 → 派回 owner
  └─ 大问题 → 文书写新任务卡（新 NNN 编号 + 独立 parent_focus / 写集 / 验收）→ 进预备队列等派工
```

判据"小 vs 大"：
- 小：单文件单段落、改动 ≤ 30 行、不动 export / 不改任务卡边界、不升级半径。
- 大：跨文件 / 跨写集 / 改 schema / 动 SSOT / 升半径 / 涉及资金-权限-安全-生产-医疗 P0-P1。
- 判不清 = 按大处理（保守路线，避免小补丁卡偷偷越界）。

## 五、升级 / 降级

### 升级到 8-worker 狂暴模式

满足全部条件才升级：
- 六槽模式运行稳定，没有待审计堆积。
- 至少 5 条独立写集。
- 文书已建战情板，巡检能持续报状态，审计能滚动回收。
- 每张卡都有机器证据命令和证据输出路径。
- 用户明确要求冲刺。

升级后仍保留文书 / 巡检 / 审计职责；不能把所有槽都改成工人。8-worker 采用 `playbooks/05` 的分波规则，不允许 8 个代码 worker 同时交付给包工头。

### 降级到默认 3-worker 模式

任一条件触发即降级：
- 待审计卡 ≥ 3 **且文书预备卡 < 3** 且 20 分钟未回收（v0.4.8：必须三件齐发才降级，单看待审堆积不算）。
- 出现误 stage、误 amend、误带别队文件。
- 多个 worker 需要同一个公共文件。
- 包工头开始亲自补多个角色的活。
- "待回炉"清单累计 ≥ 5 条且无对应工人闲置可回炉（v0.4.8）。
- 子 agent 回报未经压缩，包工头开始阅读长报告而非裁决。

## 六、派发模板

### 文书

```
你是东合施工队文书 agent。
只负责任务卡、战情板、施工日志、预备队列。
不改业务代码，不 stage，不 commit。
每次回报只给：新增/更新文件、待包工头确认点、下一批可派卡。
禁止贴整张任务卡；必要时给路径。
```

### 巡检

```
你是东合施工队巡检 agent。
每 10-15 分钟读取 git status、在工任务卡、施工日志、最近提交。
不改文件，不 stage，不 commit。
回报格式：当前在工 / 待审 / 脏区 / 冲突风险 / 阻塞 / 建议动作。
只报红/黄/绿和 1-3 个动作。
```

### 审计

```
你是东合施工队审计 agent。
只审已交到待审计的任务卡。
按白名单、diff、测试、证据、越界五项核验。
不直接修代码，不扩大范围，不 commit。
结论只给：通过 / 打回 / 阻塞 + 证据。
P3 进入后补队列，不打断包工头。
```

### 工人

继续使用 `playbooks/04-施工工人协议.md` 和 `playbooks/05-并行编排.md` 的三元绑定规则。

## 七、反例

- 6 个子 agent 全部派成工人，包工头继续自己写卡、巡检、审计。
- 文书写了卡但没有写集白名单，导致工人互踩。
- 巡检发现冲突只记录不提醒包工头。
- 审计为了赶进度直接修代码。
- 包工头在工作区脏乱时继续 amend / push，导致别队产物混入。
- **审计发现小瑕疵就让工人停工等修复 → 工人空转**（v0.4.8 新增反例）。
- **审计发现大问题直接打断工人当前在工任务卡，要求"先停下来改这个"**，而不是写新任务卡进队列（v0.4.8）。
- **包工头看到待审计 ≥ 3 就立刻暂停派新卡**，结果文书还有 5 张预备卡而工人空着没事干（v0.4.8：不看文书弹药线就降级是过度反应）。
- **子 agent 把整份测试输出 / diff / 任务卡复制给包工头，导致父 agent 控制面过载**（v0.4.9）。

发现以上反例，按 `playbooks/12-工地禁忌识别与沉淀.md` 记录。
