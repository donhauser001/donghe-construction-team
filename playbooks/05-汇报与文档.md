# 05 · 汇报与文档

> 适用：给用户的每次汇报；完工后的文档回写；用户确认的分级。核心：**产品先于文档，文档只做边界、证据、交接和复盘**。

## 一 · 诚实汇报

按半径定结构，**S 不强写 M 的结构，L 不准用 S 的结构敷衍**：

- **S**（1~3 行）：改了什么 + 一行证据 + "日志已记一笔"。
- **M**（10 行内）：结论（可验收/部分/阻塞）· 改动清单 · 已验证（带证据等级 + 1~2 行运行时片段）· 未验证 + 原因 · 卡链接。
- **L**（完整）：M 全部 + 已知盲区 + 剩余风险 + 文档同步情况 + 元信息（几个工人 / 耗时 / 推进了哪个节点）。

**推荐句式**："已验证：浏览器实操 + DB 查询一致" / "未验证：本机无 staging 数据，待你跑一次" / "本次仅静态追踪" / "部分可验收：核心可用，xxx 已开新卡"。

**禁说**（无证据用语）：搞定 / 没问题 / 应该可以 / 大概率 OK / "我跑了一下没事" / "通过了所有测试"（不说哪些测试什么命令）。"完成"必须伴随证据。

**出错时第一句直接认错**："上轮我说 X 已验证，实际没验证 Y，出错了"——不说"可能有点小问题 / 情况有些变化"。

**进度同步**：M 级派工时一句话告知；L 级每完成一张卡 1~2 句、过半时小汇报；阻塞时立即汇报不等节点。给建议时不把判断权全推给用户："我倾向 A，原因 X，等你拍板"。

## 二 · 确认分级（何时问用户）

**A · 必须问**：方向变化（蓝图 / 路线图 / 阶段焦点）；schema / migration / SSOT 破坏性变更；资金 / 支付 / 订单核心状态写操作；权限 / 安全 / 隐私 / 医疗 P0-P1；生产部署 / SSH / 凭证 / 域名 / 环境变量；删文件 / 重写历史 / force push / 不可逆数据操作。

**B · 简问**（给推荐 + 备选，一句能答）：阶段切换候选；多队冲突处置；文档规范化方案；踏勘后骨架创建；无守护启动 4 项。

**C · 默认做后汇报**（不问）：施工日志、卡状态机更新、机器证据采集、知识库轻量追加、P2/P3 禁忌候选、链接 / 错别字 / 索引修复。

**一次最多问 3 个问题**；超过就先施工可确定部分，其余进"待用户决策清单"。

这也是总工 / 监理的姿态基准：先服务施工（给白名单 / 黑名单 / 验收命令 / 已知坑，不让队伍等长文档），并行补位（SSOT / 路线图 / 风险登记），不抢施工，不用审计替代实测，不因低风险问题冻结主线。

## 三 · 文档预算

| 半径 | 允许动作 | 预算 |
|---|---|---|
| S | 施工日志 1 行 | ≤ 5 分钟 |
| M | 瘦卡 + 机器证据 + 必要知识库 | 文档时间 ≤ 施工时间 20% |
| L | 多卡 + 战情板 + 证据索引 + 必要路线图同步 | ≤ 30%（阶段收口除外） |
| 无守护 / 多队 | 用模板 + 机器证据 | 心跳只写状态差异 |

**超预算且代码未跑通 → 立刻停写文档回去施工**；要记但不急的进"后补文档队列"；发现自己在写长段叙事 → 改成表格 / 链接 / 证据块。

**防文档分叉**：任何文档声称必须带同步锚点（commit hash / 卡链接 / 证据文件 / 测试输出 / file:line 至少一种）。审计发现"文档说已做但代码证据没有" = P1：文档先降级为"待验证"，不许错误文档继续当事实入口。文档与代码冲突时以代码和运行时为准。

## 四 · 回写哪些文档

| 文档 | 何时写 | 要点 |
|---|---|---|
| `docs/施工日志/YYYY-MM-DD.md` | **每天结束前必写**（S 级也记一行） | 当日卡状态变化 + 新发现；多队各写各的（`YYYY-MM-DD/<队>.md`）；不写空泛流水账，不复制长输出 |
| `docs/项目知识库.md` | 有**项目特有**沉淀才追加 | 命名约定 / 术语 / 踩坑 / "此文件是核心别乱动"；每条含场景 / 内容 / 来源卡 / 时间；通用编程知识不写 |
| `docs/工地禁忌.md` | 发现反例**立刻**沉淀（见"五"） | P0/P1 必写；P2/P3 可后补 |
| `docs/路线图.md` | 节点级触发 | 完成标 ✅、新增入未来节点、作废标 ❌ 留痕不删；不写细节 |
| `docs/下一阶段工作.md` | 阶段切换（用户拍板后） | 主线 / 次线 / 不做 三段 |
| `docs/蓝图.md` | 用户谈及未来方向时 | 未确定语气 + 探讨日期 + 待澄清问题；升级时标注转换 |
| `AGENTS.md` | 结构 / 约定重大变化 | 接入标识 + 等价物映射 + 子项目清单 + 方向链指针 |

**完成 = 写完文档之后的那一刻**，禁止"完成→走人"。

## 五 · 工地禁忌的识别与沉淀

**skill 教方法，项目记案例**——反例写进当前项目的 `docs/工地禁忌.md`，不写进 skill（A 项目的硬编码在 B 项目可能完全合理）。

**触发沉淀**（任一）：差点犯的错被拦下（near miss）；已犯的错被打回或用户点名；项目特有反模式；审计九问命中。

**自检信号**（心里出现这些话就停下自查）："工期紧先这样" / "之前好像遇到过" / "用户应该不会发现" / "build 通过就行" / "先这样后面再优化"。

**沉淀格式**（`templates/工地禁忌条目模板.md`）：标题 / 时间 / 来源卡 / 类型 / 严重度 / 场景 / 反例 / 正例 / 代价 / **检查方式**（未来怎么发现——九问哪问、grep 什么）。编号全项目单调递增。

**复盘**：每个里程碑翻一遍——久未出现的归档、重复的合并、失效的标 ⚠️ 过期保留、出现 ≥ 3 次的升级为知识库硬规则。禁忌要被任务卡"不变量"段和审计报告**引用**才有价值。

**不要做**：把通用编程反模式（"别用 var"）写进禁忌（那是 lint 的事）；写情绪（"我太蠢了"）；含糊沉淀（"以后注意"——必须有反例 + 正例 + 检查方式）。
