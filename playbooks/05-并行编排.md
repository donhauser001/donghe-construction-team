# 05 · 并行编排

> 适用：包工头拆完任务卡、准备派工时。**仅包工头读**。

## 一、并行铁则

### 数量上限

默认模式：**同时在工的工人数量 ≤ 3**。理由：
- 大多数项目里超过 3 个会让包工头来不及审收。
- 超过 3 个容易出现"等其他工人才能完成"的隐性依赖。
- ≤ 3 个时包工头可以一手把全局握住。

用户授权的狂暴施工模式：**单队总工人上限 ≤ 8，但 v0.4.9 起默认分波派发**。启用口令包括用户明确说“8 队 / 8 worker / 狂暴施工 / 一个队同时派 8 个 worker”。这不是默认值，只在用户拍板、任务足够独立、写集可切干净、机器证据路径明确时启用。

8-worker 模式的核心判断：
- 能切出 5-8 条独立任务线 → 开狂暴模式，但第一波最多 4 个代码 worker + 1 个证据/测试 worker。
- 只有 3-4 条独立任务线 → 派 3-4 个，不硬凑 8。
- 少于 3 条任务线 / 写集交叉 / 需要连续架构判断 → 回默认模式或串行。

### 永不阻塞自己

**派完工立刻继续做下一批规划**，不要空等所有人完工才动。包工头的产能 = 同时手握"已派出的工人 + 正在拆的下一批卡 + 正在审收的回报"。

## 二、分线规则（防踩脚）

并行派出的多张任务卡之间必须满足以下**任一**分线条件：

| 分线方式 | 含义 | 例 |
|---|---|---|
| **按子项目（首选）** | 每张卡的范围严格落在一个子项目根内（monorepo 友好） | 卡 A 改 `backend/`，卡 B 改 `admin-web/`，卡 C 改 `miniapp-companion/` |
| **按文件域** | 每张卡的"白名单文件"互不相交 | 卡 A 改 `src/orders/`，卡 B 改 `src/payment/`，卡 C 改 `src/notify/` |
| **按依赖层** | 改不同层（前端 / 后端 / DB），且接口契约本次不动 | 卡 A 改 UI，卡 B 改服务层逻辑，卡 C 改 DB 查询 |
| **按子模块** | 改同一项目的不同模块 | 卡 A 改订单模块，卡 B 改支付模块，卡 C 改通知模块 |
| **按改动类型** | 同一文件的不同关注点（罕见，需谨慎） | 卡 A 加迁移，卡 B 加测试，卡 C 加文档 |

**monorepo 强建议**：优先按子项目分线，每张卡明确写"涉及子项目"字段（详见 `playbooks/03-任务卡规范.md` 元信息头）。
**多队场景**：分线时还要避开"友邻施工队"的占用范围，详见 `playbooks/14-多施工队协同.md`。

**禁止**两个工人同时改同一个文件。哪怕只是同一个文件不同函数也禁止 —— git 合并冲突 + 你没法干净地审收谁先回来谁覆盖谁。

### 8-worker 分片规则（狂暴模式必填）

启用 8-worker 前，包工头必须先写一张战情板或在任务总卡中填表：

| 工人 | 任务卡 | 写集白名单 | 黑名单 | 依赖 | 验收命令 | 回收顺序 |
|---|---|---|---|---|---|---|
| #1 | Axxx | `path/a` | `path/b` | 无 | `npm run ...` | 1 |
| #2 | Axxx | `path/c` | `path/a` | 无 | `node --check ...` | 2 |

硬规则：
- **一工人一写集**：白名单不交叉，目录交叉时继续细化到文件。
- **一共享契约一工人**：API 契约、schema、路由注册、公共类型、公共 UI 组件只能由一个 worker 拥有；其他 worker 只消费，不改。
- **一生产资源一工人**：SSH、Docker、migration、支付回调、凭证注入、线上配置同一时段只给一个 worker。
- **一子项目最多 2 个代码 worker**，除非都是纯页面样式 / 纯文档 / 纯测试且文件完全互斥。
- **文档 / 测试 / 复测可补满并发**：代码线不够 8 时，用复测、截图、grep、报告、禁词检查补位，而不是把代码拆碎。
- **证据先行**（v0.4.9）：每张卡必须写明机器证据命令和输出路径；没有证据路径，不进 8-worker。

## 三、依赖处理（串行排队）

并行的卡之间必须**互相独立**。有依赖的卡走串行：

```

狂暴模式下额外要求：
- 任何 worker 需要等待另一个 worker 产出，说明任务拆错；等待方不派，等依赖完成后作为下一波。
- 需要统一接口契约时，先派“契约 worker”或包工头定契约，再派前后端 worker。
- schema / migration 卡完成并验收前，所有依赖 schema 的 worker 只允许读现状或写文档，不许猜字段。
情况 A：卡 A 必须先于卡 B（B 依赖 A 的产出）
       → 派出 A，等 A 进入 ✅ 已完工，再派出 B
       → 派 A 的同时可以并行派 C / D（C/D 与 A 无依赖）

情况 B：卡 A、B 之间互相独立，但都修改同一个 schema
       → schema 拆出第三张卡 S 先做，S 完工后再并行派 A、B
```

## 四、Cursor / Codex Task tool 调用模板

### Cursor 侧（subagent_type 选择）

包工头默认在 Cursor 中可以使用 `Task` 工具派 `generalPurpose` / `shell` 子 agent。**派工 prompt 用"路径派工"模式**（短而清晰，工人首动作就是读卡）：

```
Task(
  subagent_type="generalPurpose",  # 或 "shell" / "best-of-n-runner"
  description="工人#1 · <任务名称>",
  model="composer-1.5",  # ⚠️ 必填 · 便宜档示例 slug · 以 §四·5·2 档位表为准 · 不写 = 默认继承父 agent (常为最贵)
  prompt="""
你是东合施工队的施工工人 #1。
铁律：阅读并严格遵守
  ~/.cursor/skills-cursor/donghe-construction-team/playbooks/04-施工工人协议.md

你的任务卡（首动作：读它）：
  <项目根绝对路径>/docs/任务卡/YYYY-MM-DD-NNN-任务名.md

期望产出：
- 完工 → 卡内当前状态置为 🔍 待审计 + 工人回报区填全
- 阻塞 → 卡内当前状态置为 ⏸️ 阻塞 + 文件名加 .阻塞 后缀

最终回复只一句：
"卡已提交待审计：<卡路径>" 或 "卡已阻塞：<卡路径> · <一行原因>"
不要重复卡内内容，不要写额外解释。
""",
  run_in_background=True,  # 多工人并行时必须 background
)
```

**为什么用路径派工而非内联任务卡内容**：
- 任务卡是落盘活文档，内联会导致工人改的是"参数副本"而非真卡 → 包工头无法审收
- 路径派工 + 工人改真卡 = 状态机统一 + 多工人产出可审 + 历史可回溯
- 内联派工**仅用于不写任务卡的 S 级**（罕见）

### Codex 侧

Codex 的子 agent 调用形式由当前 host 决定（`codex-as-tool` 或类似机制）。**关键参数等价**：

- 子 agent 的 system prompt 必须引用 `~/.codex/skills/donghe-construction-team/playbooks/04-施工工人协议.md`
- 输入是任务卡路径
- **模型必须显式指定便宜档**（详见 §四·5 · 不指定 = 默认继承 = 反成本优化）
- 长任务后台运行
- 多工人并行时**禁止串行等待**

## 四·5 模型选择硬规则（v0.4.2 · 2026-05-06 必读）

### 4·5.1 为什么必须显式指定 model

**Cursor / Codex Task tool 不指定 `model` 参数 = 默认继承父 agent 模型**。包工头通常是 IDE 里给的强模型（如 Claude Opus 4.7 / GPT-5.5）· 工人沿用 = 工人也用最贵模型 = 单次派工成本可能放大 3-10 倍 · **这是反成本优化的默认行为**。

**关键认知**：默认行为不是"自动选最便宜"，而是"自动继承父 agent 最贵"。**必须主动指定。**

### 4·5.2 模型档位表 + 选择决策树（⚠️ 全套文档唯一维护模型 slug 的地方）

> **单一事实来源**：模型 slug 只在本节维护。SKILL.md / 其它 playbook / 模板一律只说"便宜档 / 中档 / 强档"并引用本节。模型代际更替时，只改这里。

**档位定义**（当前推荐 slug · 以所在 IDE 实际可用模型列表为准）：

| 档位 | 当前推荐 slug | 用途 |
|---|---|---|
| **便宜档** | `composer-1.5`（备选 `composer-2-fast`）| 工人默认档 · 按清单填空 / 结构化编辑 |
| **中档** | `claude-4.6-sonnet-medium-thinking`（备选 `gpt-5.5-medium`）| 中等代码实现 / 边界判断 |
| **强档** | `claude-4.6-opus-high-thinking`（备选 `claude-opus-4-7-thinking-xhigh`）| 包工头 / 监理 / 架构决策 |

**核心原则**：模型选择按**任务难度** · 不按**任务重要度**。详细的清单 + 高质量 prompt + 工人级任务 = 便宜档完全够用。

| 任务性质 | 推荐档位 | 备选 | 不要用 |
|---|---|---|---|
| 结构化文档编辑（按详细清单填空 / 追加 / 标准化）| 便宜档 | 中档 | 强档 |
| 中等代码实现（单模块改 · 接口已设计 · 工人按规范填）| 中档 | 便宜档 | 强档 |
| 跨模块代码 / 中等推理 / 边界判断 | 中档 | 中档备选 | 强档（除非真涉及深度推理）|
| 架构决策 / 深度推理 / 多文件交叉 / 设计 SSOT | 强档 | 强档备选 | 便宜档（推理深度不够）|
| Best-of-N 并行尝试（创意 / 设计探索 / 多方案对比）| 多档混合（每个 attempt 用不同档）| — | 全用同一档（失去对比） |

> 用户没明示偏好时 · 默认走"够用即可"原则 · 倾向便宜档为工人通用默认。

### 4·5.3 派工前校验清单（每次必过 · v0.4.3 升级为"三元绑定双闸门"）

**派工闸门**（包工头 Task 派出前自检 · 缺一即派工违规）：

- [ ] **三元字段**：任务卡元信息头里"派给 / 调用模型 / 身份-模型匹配理由"三字段全填（不是占位 / TODO / 留空）？
- [ ] **字段一致性**：Task 调用里的 `model="..."` 与卡上"调用模型"字段**完全相同**？
- [ ] **决策树**：卡上"调用模型"是否符合 §4·5.2 决策树推荐？
- [ ] **匹配理由质量**：卡上"身份-模型匹配理由"是否能站得住（不是"够用"这种水词）？
- [ ] **告知用户**：在 todo / chat 里**告知用户** "派 X 个 worker · 工人#N · 用 Y 模型 · 理由 Z"？
- [ ] **同档**：多并行 worker 是否选了**同一档**模型？（混档容易产出风格不一致）
- [ ] **半径 vs 难度**：检查任务半径（S/M/L）vs 复杂度（低/中/高）是否分清（详见 `playbooks/03-任务卡规范.md` §二·2·2）
- [ ] **prompt 贴卡**：Task prompt 里是否**贴出**卡上三字段让工人核对（开工闸前置）？

**派工 prompt 推荐结构**（含三元绑定提示）：

```
你是东合施工队的施工工人。
铁律：阅读并严格遵守
  ~/.cursor/skills-cursor/donghe-construction-team/playbooks/04-施工工人协议.md

你的任务卡（首动作：读它）：
  <项目根绝对路径>/docs/任务卡/YYYY-MM-DD-NNN-任务名.md

⚠️ **首动作必须先做"身份-模型自检"**（playbook 04 §一·1）：
- 卡上写了"派给：工人#N"和"调用模型：<slug>"
- 你必须核对自己实际是工人#N + 自己的实际模型 == 卡上 <slug>
- 不一致 → 立即罢工 + 卡顶留备注 + 不进入 🔨 状态

期望产出：
- 完工 → 卡内当前状态置为 🔍 待审计 + 工人回报区填全
- 阻塞 → 卡内当前状态置为 ⏸️ 阻塞 + 文件名加 .阻塞 后缀
- 身份-模型不一致 → 工人退卡（保持 📋 · 卡顶加备注）

最终回复只一句：
"卡已提交待审计：<卡路径>" 或 "卡已阻塞：<卡路径> · <一行原因>" 或 "工人退卡：<卡路径> · 身份-模型不匹配"
```

### 4·5.4 红线（违反 = 重大违规 · 必沉淀对应工地禁忌）

**v0.4.2 红线**（model 参数显式 · 已沉淀 OPS-001）：

- ❌ 派 Task 不写 `model` 参数 → 默认继承父 agent → 工人莫名贵
- ❌ 用 Opus high-thinking 跑"按清单填空"型任务 → 杀鸡用牛刀
- ❌ 不告知用户就并行多个贵 worker（用户没机会干预成本）
- ❌ 把"任务重要"误判为"需要好模型" — 任务**重要 vs 难度**是两件事
- ❌ 用 `model="..."` 但选的模型不在用户 IDE 可用清单 → 静默失败 / 回退默认

**v0.4.3 新增红线**（三元绑定双闸门 · 已沉淀 OPS-003）：

- ❌ 任务卡缺"派给"或"调用模型"或"身份-模型匹配理由"任一字段 → 派工闸违规
- ❌ Task `model="X"` 但卡上"调用模型：Y" → 字段不一致 = 派工闸违规
- ❌ "派给：包工头亲自" 但任务半径 ≥ M + 工作量 ≥ 30 行业务代码 → 包工头下场反模式（详见 `playbooks/03-任务卡规范.md` §二·2·4）
- ❌ 工人接卡跳过"身份-模型自检"直接施工 → 开工闸违规
- ❌ 工人发现身份-模型不一致仍"先做一点"或"反正派给我了我顺手做" → 严禁的"顺手"反模式

### 4·5.5 失败模式回顾（本节沉淀来源）

**事故 1（v0.4.2 沉淀）**：2026-05-06 18:48 ~ 18:51 · kekeling2.0 项目 · 包工头（Claude Opus 4.7 high-thinking）在做 A012 v0.2 现场情报修正时派 3 个 `generalPurpose` subagent 并行处理 3 处 docs 修正（每个 task 都是按详细清单做 search-replace + 段落追加 · 典型工人级工作）· **3 个 Task 调用都没指定 model 参数** → 全部沿用 Opus 4.7 high-thinking · 跑了约 161 秒被用户中断 + 提出成本质疑 · 沉淀 §4·5 全节硬规则 + 项目级 OPS-001。

**事故 2（v0.4.3 沉淀）**：2026-05-06 22:00 ~ 22:08 · kekeling2.0 项目 · 包工头（Claude Opus 4.7 high-thinking）在做 A010 第二阶段 miniapp Sentry SDK 时**未派工 / 未拆任务卡 / 自己亲手写**：
- `miniapp/utils/sentry.js` 303 行 ES5（M 级）
- `miniapp-companion/utils/sentry.js` cp + sed 复制 + 改 SDK_NAME
- 两端 `app.js` 顶部 init 改造
- 自跑 smoke test
- 全程没有任务卡 · 没有派工 · 没有"派给"+"调用模型"字段
→ 用户当面识破："我发现你很多活还是自己在干 · 你不知道你的职责和分工吗 · 需不需要你再巩固一下"
→ 沉淀 v0.4.3 三元绑定双闸门 + 项目级 OPS-003。

**教训**：
- "任务清单详细 + 难度低" → 工人就该用 `composer-1.5`，不是 Opus high-thinking（事故 1）
- "我没指定 model" ≠ "agent 自动选最便宜" · 而是 = "自动继承父 agent 最贵"（事故 1）
- "我自己懂 + 觉得快" ≠ "可以自己干" → M 级 ≥ 30 行业务代码必派工（事故 2）
- "派给"+"调用模型"+"匹配理由" 必须显式落卡 · 用合规仪式硬性切断"顺手干"反模式（事故 2）

### 4·5.6 紧急修复 SOP（违反红线后立即做）

1. 立刻 `git status` 看当前已花费的 subagent 是否产出有效内容
2. 给用户 3 个选项：
   - A · 接受现有部分产出 + 用便宜模型补完缺口
   - B · `git checkout` 回滚 + 用便宜模型重派
   - C · 包工头亲自补完（如果剩余工作量小）
3. 沉淀进当时项目的 `docs/工地禁忌.md`（项目级实战记录）+ 检查本 skill 是否需要 v 升级
4. 当日施工日志记一笔（含失败模式 + 教训 + 触发的 skill 升级）

## 五、回收节奏

```
工人 #N 提交（卡 → 🔍 待审计） → 包工头立刻审收（不批量等所有人）
                                ↓
                        监理姿态判定（07-监理审计清单.md）
                                ↓
                        ✅ 已完工：加 .完工 后缀，记当日施工日志
                        ❌ 不可验收：写明 P0/P1 发现，重派给同一工人或拆卡重派
                                ↓
                        无论结果，立刻继续派下一批
```

默认模式下，**禁止**等到三个工人全部完工才统一审。理由：
- 后回来的工人会撞上前面工人的修改 → git 状态混乱
- 包工头记忆衰减 → 审收质量下降
- 单个失败会拖累整批

### 8-worker 回收节奏

狂暴模式下按"分波滚动回收"而不是"八个一起等"：

1. 每个 worker 回来后立即看三件事：是否越界、是否有证据、是否污染别人的写集。
2. 第一波最多 4 个代码 worker + 1 个证据/测试 worker；第二波只有在第一波至少 2 张证据齐全后才派。
3. 同时待审计卡达到 3 张时，**暂停的是新派工，不是工人**——若文书预备卡 ≥ 3 且工人槽空，工人继续接已经写好的下一批卡。审计与工人施工解耦（v0.4.8 · 详见 `playbooks/19-六槽作战编制.md` § 四 "审计回收线"+"工人不空转线"）。
4. 有冲突文件出现，立刻冻结相关 worker 的后续提交，包工头决定保留哪条线，另一条线改为重派 / 手工合并 / 作废。
5. 每 20-30 分钟更新一次战情板；战情板只放任务卡、写集、状态、证据文件、commit hash、阻塞，不写长描述。
6. 每个 worker 必须精确 stage 自己白名单；严禁 `git add .`。
7. **审计发现的问题不打断当前 worker 当前在工卡**（v0.4.8）：小问题进 owner 工人的"待回炉"清单等闲置时回炉；大问题（半径升级 / 跨写集 / 动 schema 资金 权限 安全）由文书拆新任务卡按新任务派工。
8. 包工头手上待裁决 P0/P1/P2 超过 3 项时，立刻降级到默认 3-worker，先清裁决队列。

---

## 六、何时不并行

下列场景**只派 1 个工人**（哪怕你忍不住想加速）：

- 改 schema / 数据库迁移类工作
- 涉及钱、权限、安全的工作
- 第一次接触的领域，你对踩脚边界不确定
- 任务卡总数 < 3 张
- 你（包工头）当前已经手握 ≥ 3 个 🔍 待审计的卡（说明你的审收能力已饱和）

下列场景**禁止开 8-worker 狂暴模式**：

- 共享 schema / migration 尚未稳定。
- 多条任务都要改同一个 API 契约、同一个公共工具、同一个 store、同一个全局配置。
- 生产部署 / SSH / 凭证 / 支付 / 资金 / 权限 / 医疗安全需要多点同时操作。
- 没有战情板或没人负责滚动审收。
- 没有机器证据命令 / 证据输出路径。
- 包工头已经在手工整合长文档、长审计、长回报。
- 只是为了“看起来很忙”而拆任务，实际没有 5 条以上独立价值线。

## 七、并行的反例（工地禁忌）

以下都是会被记入工地禁忌的反例：

- 同一个文件被两个工人同时改
- 派出工人后包工头空闲等所有人完工
- 一张大卡硬塞给一个工人，应拆未拆
- 工人 A 等工人 B 的产出（隐性依赖未声明）
- 派工后包工头不再读任务卡，全权交给工人自由发挥
- **开 8-worker 但没有战情板 / 写集表 / 回收顺序**
- **为凑满 8 人把同一模块硬拆到多个 worker，导致公共文件互踩**
- **8 个 worker 一起提交后才统一审，包工头变成冲突清理工**
- **8-worker 没有分波，所有人同一时间交付，父 agent 被迫靠记忆整合**
- **战情板写成长篇日报，没有机器证据路径和 commit hash**
- **派 Task 不写 `model` 参数 → 默认继承父 agent 最贵模型**（v0.4.2 新增 · 详见 §四·5）
- **用 Opus / GPT-5.5 跑工人级"按清单填空"任务**（v0.4.2 新增 · 任务难度判错）
- **不告知用户就并行多个贵 worker**（v0.4.2 新增 · 用户没机会干预成本）

发现以上情况，立刻按 `playbooks/12-工地禁忌识别与沉淀.md` 沉淀。
