# 12 · 工地禁忌识别与沉淀

> 本 playbook 不写具体反例（反例属于项目）。本 playbook 只教**如何识别 + 如何沉淀**。
>
> 具体反例由包工头 / 监理在施工过程中**动态记录到当前项目的** `docs/工地禁忌.md`。

## 一、为什么禁忌库属于项目而不是 skill

- skill 通用，反例**项目特定**：A 项目的"硬编码省份列表"在 B 项目可能完全合理
- 反例需要**随项目演进**：旧反例可能因为架构升级失效，新反例不断出现
- 反例需要**和当前任务卡 / 施工日志互相引用**：放在项目里才能引
- 写在 skill 里的反例会变成"放之四海而皆不准"的死规则

所以：**skill 教方法，项目记案例**。

---

## 二、什么样的事值得记入工地禁忌

满足以下**任一**条件，立刻沉淀：

### 类型 A · 差点犯下的错（near miss）
- 工人差点引入硬编码 / 临时方案 / 跳过验证，被自己 / 监理拦下
- 包工头差点漏读项目文档导致方向偏
- 任务卡差点忘了 AI 工具同步

### 类型 B · 已经犯下的错
- 已经写出去的代码被监理打回（P0 / P1）
- 用户指出"上次那个 X 别再做了"
- 同样的失误第 2 次出现

### 类型 C · 项目特有的反模式
- 这个项目历史上踩过的坑，回顾时识别出来
- 这个项目特定的命名 / 边界 / 安全约束被违反

### 类型 D · 监理九问命中的反模式
- SSOT 分裂、AI 工具假实现、缓存陈旧、撤销不全等命中后

---

## 三、识别提示词（用于自检）

施工 / 审计过程中如果出现以下信号，**自检一次**是不是该沉淀：

- "我心里觉得这样写有点不对，但工期紧"
- "之前好像也遇到过类似的问题"
- "用户应该不会发现"
- "监理可能不会查这里"
- "build 通过就行了"
- "改完跑一下就好，应该没问题"
- "这个先这样后面再优化"

每一句都对应一类典型反模式。出现就停下来，按本 playbook 沉淀。

---

## 四、沉淀格式（追加到 `docs/工地禁忌.md`）

每条用 `templates/工地禁忌条目模板.md` 的格式：

```markdown
## 禁忌-N：<一句话标题，能让未来的工人快速识别>

- **发现时间**：YYYY-MM-DD
- **发现于任务卡**：YYYY-MM-DD-NNN-... （或施工日志日期）
- **类型**：A 差点 / B 已犯 / C 项目特有 / D 监理九问
- **严重度**：P0 / P1 / P2 / P3
- **场景**：（一段话说清楚发生在什么上下文）
- **反例**：
  ```代码 / 配置 / 流程
  错误的做法
  ```
- **正例**：
  ```代码 / 配置 / 流程
  正确的做法
  ```
- **代价**：（如果不改会发生什么）
- **检查方式**：（怎么在未来发现这种问题——审计九问哪一问？grep 什么关键字？）
- **关联项目知识库条目**（若有）：
```

编号 N 全项目单调递增，不重置。

---

## 五、何时沉淀

- 施工过程中识别 → **立刻**写入（不要等任务完工）
- 审计过程中识别 → 写在监理报告 + 同步追加到工地禁忌
- 用户主动指出 → **当场**追加，并在汇报里告知"已记入工地禁忌"
- 写当日施工日志时回顾 → 补记今天遗漏的

---

## 六、复盘机制

每个里程碑 / 阶段切换时，包工头主动做一次：

- 翻 `docs/工地禁忌.md`，看哪些是过去 N 个月再没出现的（候选归档）
- 看有没有相互重复的（合并）
- 看有没有过期的（架构升级后失效，标 ⚠️ 已过期 但保留）
- 把高频禁忌（出现 ≥ 3 次的）升级为"项目知识库"里的"硬规则"

---

## 七、引用机制

- 任务卡的 `## 不变量` 段可以引用工地禁忌 ID（如 "禁忌-7"）
- 监理报告的发现段可以引用工地禁忌 ID
- AGENTS.md 的"必读速查"可以列高频禁忌的标题

让禁忌真的能**反向影响后续工作**，而不是写完就躺在文件里。

---

## 八、永远不要做的事

- ❌ 把通用编程反模式（"不要用 var"、"不要 console.log"）写进项目工地禁忌 → 那是 lint 该做的事
- ❌ 把"对自己的指责"写进工地禁忌（"我太蠢了"）→ 写**事实**和**机制**，不写情绪
- ❌ 复制 skill 里的句子粘到项目工地禁忌 → 项目禁忌只写**这个项目特有的**
- ❌ 沉淀时含糊（"以后注意 xxx"）→ 必须有反例 + 正例 + 检查方式
