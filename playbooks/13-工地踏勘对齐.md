# 13 · 工地踏勘对齐

> 适用：项目首次接入后的所有"重新踏勘"场景。
>
> 踏勘报告 = **项目当前实际状态的客观快照**，是活文档，必须定期对齐。
>
> v0.4.9：踏勘输出以方向和风险为主，文档骨架创建走简问，日志/索引/轻量修补默认做后汇报；不要把踏勘变成文档施工。

## 一、踏勘报告的定位

```
蓝图          → 项目要去哪里（未来）
路线图        → 按时序怎么走（节奏）
下一阶段工作  → 当前聚焦什么（焦点）
项目知识库    → 已沉淀的约定（手册）
工地禁忌      → 已踩过的坑（避雷）
施工日志      → 每天发生了什么（流水）
工地踏勘报告  ← 项目当前实际站在哪里（全景航拍）
```

踏勘报告是其他文档的**索引和入口**，先看它能极大节省后续每次接活的对齐成本。

## 二、文档落盘位置

```
docs/
├── 工地踏勘报告.md         # 当前最新版（始终覆盖）
└── 踏勘历史/               # 历史版本归档
    ├── 2026-05-06-初次踏勘.md
    ├── 2026-06-15-月度对齐.md
    └── 2026-07-20-M2节点对齐.md
```

历史版本归档命名：`YYYY-MM-DD-<触发原因>.md`（如 `初次踏勘` / `月度对齐` / `M2节点对齐` / `结构变更对齐`）。

## 三、何时需要重新踏勘（触发器）

按强度分两类：

### 强触发（必须跑）

- **里程碑触发**：路线图节点完成时
- **结构变更触发**：技术栈改 / 子项目增减 / 数据 schema 重大变 / 大规模重构
- **方向变更触发**：蓝图升级到路线图 / 下一阶段切换 / 主线调整
- **方向链耗尽触发**（v0.4 新增 · 任一层耗尽都触发，详见 `playbooks/17-方向链反向递进.md`）：
  - 任务卡层耗尽（在工 + 计划数 = 0）→ 触发反向探查 + 重新踏勘
  - 下一阶段工作层耗尽（所有焦点条目完成）→ 触发阶段切换
  - 路线图层耗尽（无可启动节点）→ 触发蓝图取材对话
  - 蓝图层耗尽（无 💭 条目）→ 触发冻结 + 蓝图扩展对话
- **用户主动触发**：用户说"重新踏勘"
- **检测到漂移**：包工头在世界观对齐时发现实际情况与上次踏勘明显不符

### 软触发（仅提醒）

- **定期**：距上次踏勘 ≥ 4 周
- **多队加入**（呼应 14）：另一队首次接入项目时

软触发**只提醒不强制**：包工头在接活契约阶段顺手说一句"距上次踏勘已 X 周，建议尽快重新踏勘"，是否真做由用户决定。

## 四、新鲜度自检（每次接活前）

包工头在每次接活契约前**顺手做**：

```
1. 读 docs/工地踏勘报告.md 的"踏勘时间"
2. 与今天对比，算距上次踏勘多久
3. 检查路线图：自上次踏勘后是否有里程碑标 ✅
4. 检查项目结构：是否新增了子项目 / 文档 / 技术栈
5. 检查项目知识库 / 工地禁忌：是否新增条目 > 上次踏勘时
6. 形成"新鲜度评估"：✅ 新鲜 / ⚠️ 偏旧 / ❌ 严重过期
```

评估结果写入接活契约的"风险提示"段。

## 五、重新踏勘的流程（与首次踏勘不同）

不是从零跑，而是 **diff 模式**。

> **时序铁律**：Step 1~3 **仅读不写**；Step 4 按"落盘分级"执行（L1 默认落，L2 / L3 等用户确认）。

### Step 1 · 读上一份踏勘报告（旧快照）

完整读 `docs/工地踏勘报告.md`，记住其中的：
- 项目快照
- 已识别的世界观
- 等价物映射
- 当时的阶段判断

### Step 2 · 重新扫描 + 通读（新快照）

- 重新跑 `playbooks/00-接活契约.md` 的 Step 1 + Step 2
- 重点扫"自上次踏勘以来变化的部分"（git log / docs/ mtime / 子项目结构）

### Step 3 · 输出《踏勘对齐报告》

用 `templates/踏勘对齐报告模板.md`，必填：

- **本次触发原因**（强 / 软触发分类 + 具体场景）
- **变化清单**：
  - 🆕 新增：新增的子项目 / 文档 / 模块 / 路线图节点
  - ✅ 已完成：上次踏勘到现在完成了哪些里程碑 / 任务卡
  - ❌ 已作废：被作废 / 删除的方向 / 文档
  - 📈 演进：当前阶段从 M? → M? / 技术栈版本变更
  - 🌪 漂移：实际状态与"下一阶段工作"是否对齐？是否需要调整？
- **新发现**：自上次踏勘后识别的新约定 / 新风险 / 新等价物
- **方向链快照**（v0.4 · 详见 `playbooks/17-方向链反向递进.md`）：4 层探查结果 + 情形 A/B/C/D/E/F 之一，命中 A/B/C/D 必入 Q 列表
- **建议动作**：是否需要更新蓝图 / 路线图 / 下一阶段工作 / 项目知识库
- **本次评估**：项目健康度（按需）

### Step 4 · 落盘（按落盘分级 L1/L2/L3 执行）

> 与 `playbooks/00` Step 4 落盘分级一致。

**L1 · 默认直接落**（无需用户确认）：

```
1. 把当前 docs/工地踏勘报告.md 拷贝到 docs/踏勘历史/YYYY-MM-DD-<触发原因>.md（归档）
2. 用新内容覆盖 docs/工地踏勘报告.md
3. 在聊天界面同步复述本次变化清单 + Q 列表
```

**L2 · 用户确认后落**：

```
- 在 AGENTS.md 更新"上次踏勘时间"字段（这是入口手册改动）
- 按 Q 列表里的拍板新建 / 更新 docs/下一阶段工作.md（焦点切换）
- 按 Q 列表里的拍板归档已完成阶段段落到 docs/路线图.md 历史段
```

**L3 · 用户确认后落**：

```
- git add / commit 本次踏勘对齐结果
- git switch -c 新分支（如用户拍板"新阶段起新分支"）
```

**回滚机制**：用户对覆盖结果不满意 → 包工头从 `docs/踏勘历史/YYYY-MM-DD-X.md` 复制回 `docs/工地踏勘报告.md` 即可。所以 L1 默认落是安全的。

## 六、踏勘报告的内容要求

每份踏勘报告（首次 / 对齐均同）必须有：

- **元信息头**：踏勘时间 / 触发原因 / 上一版归档路径
- **项目快照**：项目名 / 定位 / 技术栈 / 子项目 / git 状态
- **已存在文档清单 + 等价物映射**
- **项目世界观**：不变量 / SSOT / 命名约定 / 安全约束
- **当前阶段**：路线图位置 / 在工任务 / 阻塞
- **原生 AI 应用素质评估**（可选 · L 级踏勘必填）
- **风险与建议**

## 七、与"已接入项目快速对齐"的关系

每次接活的**轻量对齐** vs **重新踏勘**：

| 维度 | 轻量对齐（每次接活） | 重新踏勘（按触发器） |
|---|---|---|
| 频率 | 每次接活 | 周期性 / 按触发 |
| 范围 | 必读清单 10 项 | 全量重新扫描 + diff |
| 时间 | 5~10 分钟 | 30~60 分钟 |
| 产物 | 接活契约六栏 | 踏勘对齐报告 → 落盘覆盖 |
| 是否覆盖文档 | 否 | 是（用户确认后） |

包工头自己分清楚两者，不要把每次接活都做成重新踏勘。

## 八、踏勘对齐反例（工地禁忌候选）

- 几个月不重新踏勘，世界观严重过期
- 重新踏勘后**没归档旧版本就直接覆盖**（绕过 L1 归档机制）
- 未经用户确认就**改 `AGENTS.md` 入口手册或 `docs/下一阶段工作.md`**（L2 越界）
- 未经用户确认就**跑 git commit / push**（L3 越界）
- 检测到明显漂移但不告知用户，继续按旧报告工作
- 重新踏勘时只复制粘贴上一版，没真正重新扫描
- 踏勘对齐报告里只写"无变化"但其实有大量变化（敷衍）
- 方向链快照命中 A/B 却不告警继续接 M/L 级活
- 方向链耗尽时包工头自行从蓝图拍板新阶段方向（绕过用户确认 · 违反 v0.4 八大宪法第 8 条）
- 跨层取材后**不在源层级标"已展开"**（导致下次重复展开）

## 九、与多队场景的协同

多队场景下：

- **只有主队（首队）**有权重新踏勘 + 覆盖踏勘报告
- 别队发现需要重新踏勘 → 在自己的施工日志记一笔"建议主队重新踏勘"，告知用户
- 别队加入项目时**信任**当前踏勘报告，仅做轻量对齐（详见 14）

避免多队同时踏勘导致版本冲突。
