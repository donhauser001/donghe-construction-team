# 20 · 产品施工优先与文档瘦身

> 适用：默认适用于所有施工。目标是防止"文档施工"压过"产品施工"，降低包工头控制面负担，减少用户确认疲劳，并用机器证据降低文档与代码分叉风险。

## 一、一级原则

1. **产品先于文档**：用户链路、可运行代码、真实验证优先；文档只服务边界、证据、交接和复盘。
2. **代码 / 运行时是 SSOT**：文档不得声明代码没有实现的事实；证据必须来自命令、测试、浏览器、DB、日志或 commit。
3. **机器生成优先**：验证命令、commit hash、git status、测试输出、构建输出优先由脚本生成；手写只补上下文和判断。
4. **确认分级**：只把高风险决策交给用户；低风险文档和日志默认做后汇报。
5. **包工头降载**：包工头只裁决红色事项和最终整合；文书/巡检/审计必须先过滤、压缩、给建议，不把原始噪音丢回父 agent。

## 二、文档预算

| 半径 | 允许文档动作 | 文档预算 |
|---|---|---|
| S | 不建任务卡；施工日志 1 行；必要时最终汇报 1 行证据 | ≤ 5 分钟 |
| M | 瘦身任务卡 + 机器证据块 + 关键风险；知识库/禁忌仅有项目级沉淀才写 | 文档时间 ≤ 施工时间 20% |
| L | 多卡拆分 + 战情板 + 机器证据索引 + 必要路线图同步 | 文档时间 ≤ 施工时间 30%，阶段收口除外 |
| 无守护 / 多队 / 8-worker | 允许更重，但必须使用模板和机器证据，禁止手写长报告替代运行验证 | 每轮心跳只写状态差异 |

**超预算处理**：
- 文档已经超过预算但代码未跑通 → 立刻停写文档，回到最小可运行闭环。
- 需要记录但不影响当前施工的内容 → 放入"后补文档队列"，不阻塞工人。
- 包工头发现自己在写长段叙事 → 改成表格 / 链接 / 机器证据块。

## 三、机器证据

优先使用 `scripts/collect_evidence.py` 生成证据块：

```bash
python scripts/collect_evidence.py \
  --cmd "typecheck::npm run typecheck" \
  --cmd "test::npm test -- --runInBand" \
  --out docs/任务卡/_evidence/YYYY-MM-DD-NNN.md
```

证据块必须包含：
- 当前 commit hash：`git rev-parse HEAD`
- 工作区状态：`git status --short`
- 相关 diff 摘要：`git diff --stat`
- 每条验证命令、退出码、开始/结束时间、stdout/stderr 摘要

任务卡里只贴证据文件链接 + 关键结论；不要大段手抄命令输出。最终汇报只摘最关键 3 条证据。

## 四、确认分级

### A · 必须问（不得默认）

- 方向变化：蓝图扩展、路线图切换、阶段焦点变化。
- schema / migration / SSOT 破坏性变更。
- 资金、支付、退款、订单核心状态写操作。
- 权限、安全、隐私、医疗 P0/P1。
- 生产部署、SSH、凭证、证书、域名、环境变量。
- 删除文件、重写历史、force push、不可逆数据操作。

### B · 简问（给默认建议，用户不回则不升级）

- 阶段切换候选：给 1 个推荐 + 1 个备选。
- 多队冲突：给冲突文件、建议占用/退让方案。
- 文档规范化：给 A/B 方案，默认薄壳映射，不移动原文。
- 首次踏勘后的骨架创建：只问缺失骨架和 AGENTS 写入，不逐字确认模板内容。
- 无守护启动：只问时间窗、范围、自动 commit/push 策略、阻塞策略；其他用默认安全值，并按 playbook 16 执行时间契约、队列补给和临时分支推送。

### C · 默认做后汇报

- 当日施工日志。
- 任务卡状态机更新。
- 机器证据采集。
- 项目知识库轻量追加。
- P2/P3 工地禁忌候选。
- 文档链接、索引、错别字、标题层级、引用修复。

**原则**：确认问题一次最多 3 个。超过 3 个时先施工可确定部分，把其余放入"待用户决策清单"。

## 五、包工头降载

六槽模式下，各槽必须先压缩信息：

| 角色 | 回报给包工头的格式 |
|---|---|
| 文书 | 可派卡数量、缺口、唯一需要裁决的问题；不得贴整张卡 |
| 巡检 | 红/黄/绿态势 + 需要立即处理的 1-3 项；不得贴完整 `git status` 噪音 |
| 审计 | 通过 / 打回 / 阻塞 + P0/P1/P2 摘要 + 证据链接；P3 不打断 |
| 工人 | 卡路径 + 证据链接 + 是否越界 + 是否阻塞 |

包工头每轮只做四类裁决：
- 是否继续派工。
- 是否触发 A 类必须问。
- 是否接受机器证据为可验收。
- 是否需要降级并发。

## 六、8-worker 降爆规则

8-worker 模式默认改为**分波狂暴**，不是 8 个回报同时砸给包工头：

1. 第一波最多 4 个代码 worker + 1 个证据/测试 worker。
2. 第二波只有在第一波至少 2 张进入可验收或待审证据齐全后才派。
3. 任意时刻包工头手上待裁决 P0/P1/P2 不超过 3 项；超过即暂停新派工。
4. 战情板必须只记录机器可核对字段：任务卡、写集、状态、证据文件、commit hash、阻塞。
5. 8-worker 不用来补文档数量；文档补位必须服务验收或冲突控制。

## 七、防文档分叉

任何文档更新都必须带一种同步锚点：
- commit hash
- 任务卡链接
- 证据文件链接
- 测试命令输出
- 代码 file:line

审计发现"文档说已做，但代码/证据没有"时，按 P1 处理：文档先降级为"计划 / 待验证"，不得让错误文档继续作为事实入口。

## 八、反例

- 写了 300 行任务卡，代码还没跑起来。
- 任务卡里手写"测试通过"，但没有命令、退出码、commit hash。
- 用户被一次性问 9 个确认点，其中 6 个只是文档整理。
- 8 个 worker 同时交付，包工头开始靠记忆合并结论。
- 文档更新了路线图完成状态，但对应代码没有 commit 或证据。
