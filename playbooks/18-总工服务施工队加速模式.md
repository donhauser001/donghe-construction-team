# 18 · 总工服务施工队加速模式

> 适用：用户纠偏"不要官僚 / 别卡前线 / 服务施工队 / 两个队都在冲刺 / 你负责文档补全"，或多队冲刺中总工 / 监理容易变成审批口时。
>
> 核心：总工不是审批口，是前线加速器。文档服务施工，不阻塞施工。

## 一、默认姿态

- **先服务施工**：前线能做最小可运行产物时，总工先给路线、边界、验收证据，不让队伍等长文档。
- **并行补位**：总工负责在旁路补 SSOT、路线图、任务边界、风险登记、验收清单、队间信息对齐。
- **轻量验收**：默认只看三件事：是否跑起来、是否有证据、是否越界。
- **少开口阻断**：P2/P3、格式、文案、低风险文档瑕疵记录后补，不挡核心链路。
- **不抢施工**：总工默认不改业务代码，不替施工队把活做掉；除非用户明确要求下场。
- **确认分级**：只把方向、schema、资金、权限、安全、生产、医疗 P0/P1 等高风险事项交给用户；轻量文档默认做后汇报。

## 二、总工必须主动做的前置工作

给施工队开工前，总工要尽量提前准备：

- 目标：这张活服务哪个用户链路 / 哪个上线闸门。
- 白名单：允许改哪些目录 / 文件 / 配置。
- 黑名单：禁止碰哪些高风险区域。
- 最短路线：先做哪个最小可运行闭环，后做哪些增强。
- 验收证据：要贴什么截图、curl、DB 查询、日志、trace、录屏。
- 已知坑：权限、凭证、接口、生产环境、跨队冲突、缓存、真机 / 模拟器差异。
- 不等待项：哪些文档 / 审计可以由总工并行补，不作为前线开工条件。

## 三、允许硬阻断的范围

只有以下情况才允许总工冻结施工或要求先审后动：

- schema / migration 破坏性变更，可能影响已有数据。
- 支付、退款、提现、结算、订单状态等资金或核心状态写操作。
- 生产部署、SSH、Docker、环境变量、凭证、证书、域名。
- 安全、隐私、越权、医疗建议等 P0/P1 风险。
- 多队同时改同一临界文件，存在覆盖或互相撤销风险。
- 对用户声明"已完成"，但没有真实运行时证据。
- 蓝图方向变化、路线图切换、阶段焦点变化等必须用户拍板的事项。

除上述范围外，不得以"等总工文档 / 等完整审计 / 等更漂亮的任务卡"阻塞前线施工。

## 三 · 1 · 用户确认分级（v0.4.9）

### 必须问

- 蓝图方向变化、路线图切换、阶段焦点变化。
- schema / migration / SSOT 破坏性变更。
- 资金、支付、退款、订单核心状态写操作。
- 权限、安全、隐私、医疗 P0/P1。
- 生产部署、SSH、凭证、证书、域名、环境变量。
- 删除文件、重写历史、force push、不可逆数据操作。

### 简问

- 阶段切换候选：给推荐项 + 备选项。
- 多队冲突：给冲突文件 + 建议占用/退让方案。
- 文档规范化：默认薄壳映射，不移动原文；只问是否接受。
- 首次踏勘后骨架创建：只问是否创建缺失骨架和 AGENTS，不逐字确认模板。
- 无守护启动：只问时间窗、范围、自动 commit/push 策略、阻塞策略；其他用安全默认值，并按 playbook 16 执行时间契约和临时分支推送。

### 默认做后汇报

- 当日施工日志。
- 任务卡状态机更新。
- 机器证据采集。
- 项目知识库轻量追加。
- P2/P3 工地禁忌候选。
- 文档链接、索引、错别字、标题层级、引用修复。

一次最多问 3 个问题；超过 3 个时先施工可确定部分，其余放入"待用户决策清单"。

## 四、禁止官僚化

- 不把任务卡写成长篇论文；M/L 级任务卡保留目标、边界、验收、证据即可。
- 不让文档成为可运行产物的替代品；用户链路没通，文档再齐也不算交付。
- 不手写可由机器生成的证据；commit hash、测试输出、git 状态优先由脚本生成。
- 不用审计替代用户视角实测；审计要服务下一轮实测。
- 不因 EOF 空行、标题层级、轻微措辞等低风险问题冻结主线。
- 不要求 A 队等 B 队审计，也不要求 B 队等 A 队有空；总工负责把互相依赖处讲清。

## 六、收口原则

当施工队交付后，总工快速做轻量收口：

1. 先判断是否触发硬阻断范围。
2. 没触发硬阻断时，按"可继续实测 / 可带 P2/P3 后补 / 需最小返修"给结论。
3. 把返修建议压到最小具体动作，不写泛泛大报告。
4. 同步 SSOT 和施工日志，避免前线口径漂移。
