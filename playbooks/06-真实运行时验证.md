# 06 · 真实运行时验证

> 适用：每张任务卡完工自检时；监理审计前；声明完成前。**包工头与工人都读**。v0.4.9 起优先机器生成证据，减少手写文档。

## 一、宪法句

> **"build 通过 / tsc 通过 / unit test 通过" 不是"功能完成"。**
>
> 真实运行时的可观察行为才是。

---

## 二、按半径分档的验证清单

### S 级（最低底线）

- [ ] `tsc` / `lint` 无新错误
- [ ] 改动文件相关的 unit test 跑通
- [ ] **如果是 UI 改动**：人工或截图至少一次"看一眼"

### M 级（标准）

S 级所有 + 以下：

- [ ] **API 真调**：用 `curl` / Postman / `playwright` 真打一次接口，**贴出请求 + 响应原文**
- [ ] **关键 DB 字段真查**：用 `psql` / `prisma studio` / `sqlite3` / 项目自带 CLI 看一眼数据落得对不对
- [ ] **happy path 真走一遍**：浏览器或对应 client 实操一次
- [ ] **至少 1 个边界场景**：空值 / 超长 / 越权 / 重复提交，任选一个真测

### L 级（完整）

M 级所有 + 以下：

- [ ] **跨层联调真测**：前端→后端→DB→响应→渲染 完整跑一遍
- [ ] **跨端联调真测**：本次涉及的所有客户端（Web / Mobile / Admin / CLI）各跑一次
- [ ] **持久化真验**：操作后**刷新页面 / 重启服务**再看一次
- [ ] **实时通道真验**（如有 SSE / WebSocket）：建立连接 → 触发 → 收到消息 → **断开 → 重连** → 再触发 → 仍能收到
- [ ] **AI agent 工具同步真验**：UI 能做的，agent 工具调一次也能做（详见 `playbooks/11-原生AI应用素质.md`）
- [ ] **撤销 / 历史回滚真验**（如适用）：做 → 撤销 → 重做 → 状态一致
- [ ] **老数据兼容真验**：拿一条创建时间最早的真实记录跑一遍
- [ ] **3 个边界场景**：空值 / 超长 / 越权 / 并发 / 离线 / 多租户 中至少 3 个

---

## 三、证据格式

每条验证项必须有可贴出的证据。优先使用 `scripts/collect_evidence.py` 生成：

```bash
python scripts/collect_evidence.py \
  --cmd "typecheck::npm run typecheck" \
  --cmd "test::npm test -- --runInBand" \
  --out docs/任务卡/_evidence/YYYY-MM-DD-NNN.md
```

机器证据文件应作为任务卡和最终汇报的主证据；手写说明只补充 UI 观察、DB 语义和阻塞原因。

**证据可选形式**：

| 证据类型 | 形式 | 示例 |
|---|---|---|
| **命令输出** | 贴 stdout 片段 | `curl -X POST .../api/orders → 201 {"id":"abc"}` |
| **DB 查询** | 贴查询语句 + 结果 | `SELECT status FROM orders WHERE id='abc' → status='paid'` |
| **截图描述** | 文字描述 + 关键 UI 文案 | "订单详情页可见 '已支付 ¥39.00'，按钮变灰" |
| **日志片段** | 贴 server log 关键行 | `[INFO] order.created id=abc amount=3900` |
| **测试输出** | 贴 test runner 结果 | `PASS  src/orders.test.ts (12 tests)` |
| **机器证据文件** | 链接 + 关键结论 | `docs/任务卡/_evidence/2026-05-08-001.md · typecheck exit=0` |

**禁说"我跑过了，没问题"**。**禁说"应该 OK"**。**禁不贴证据就声明完成**。

---

## 四、写入位置

### 工人完工时
在任务卡 `## 工人回报区` → "真实运行时证据" 段，贴证据文件链接 + 关键结论。不要复制整份长输出。

### 包工头汇报时
在最终给用户的汇报（`playbooks/09-诚实汇报.md`）里，只摘最关键 3 条证据 + 证据文件路径。

### 监理审计时
按 `playbooks/07-监理审计清单.md` 的"证据等级"判定：贴出来的属于哪一档？是不是真的对应了某个具体验证项？

---

## 五、AI 工具能力同步验证（L 级强制）

任何会被用户用 UI 操作触发的能力，都必须**额外验证 agent 工具能做同一件事**：

- [ ] 找到对应的 agent 工具（如果没有 → P1 缺失，立刻记入工地禁忌）
- [ ] 用 agent 工具调用同样的功能
- [ ] 比对 UI 操作和 agent 调用的结果是否一致（数据状态、副作用、可见反馈）
- [ ] agent 调用的输入是否易于另一个 agent 自动构造（参数有意义、可发现）

详见 `playbooks/11-原生AI应用素质.md`。

---

## 六、当你**没法**真测时怎么办

诚实承认 + 在汇报里明确标注**未验证**：

- 没法在本环境跑：标 "环境受限，未验证；建议用户在 X 环境验证"
- 没法构造测试数据：标 "缺测试数据，未验证；建议下次准备 fixture"
- 改动量超出本卡范围导致连带验证：标 "本次未覆盖 X，原因：超出本卡范围"

**未验证 ≠ 失败**；**未验证伪装成已验证 = 重大违规**。

---

## 七、验证清单不要"假打勾"

每个 checkbox 后面**必须**跟"证据"或"未验证 + 原因"。**禁止**只打勾不附证据。

```
✅ API 真调
   curl ... → 200 {"orderId":"abc"}

❌ 持久化真验
   未验证。原因：本机 DB 与生产 schema 不同，需要在 staging 跑一次。
```

## 八、防文档分叉

文档不得单独成为"已完成"依据。监理发现以下任一情况，按 P1 处理：

- 任务卡写"已验证"，但证据文件没有对应命令或真实操作。
- 路线图 / 施工日志写"已完成"，但当前 commit / diff / 测试输出无法支撑。
- 文档中的接口、字段、状态与代码实现不一致。

处理方式：先把文档结论降级为"待验证 / 计划中"，再补机器证据或修代码。
