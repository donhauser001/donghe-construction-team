# 16 · 无守护自动施工模式

> 适用：用户主动告知要离开 / 出门 / 睡觉，希望施工队在用户给定时间内**自己持续施工**，并在用户回来后可审计、可回滚、可接管。
>
> **核心原则 v0.4.11**：用户给了多长时间，就按多长时间建立施工契约；宁可做小而连续的安全工作，也不允许空跑、早退、单卡做完就下班。
>
> **关键变化 v0.4.11**：遇到决策问题不停车，先按推荐方案继续推进，把重大决策、阻塞和取舍写入 `docs/技术债账本.md`；用户目标完成但时间未到时，自动阅读蓝图并下放路线图、下一阶段工作、任务卡，直到跑满用户给定时长。

## 一、触发协议

### 触发关键词

听到以下任一类表达，包工头必须立刻进入本 playbook 的 Step 0：

- "我要走了 / 出门了 / 不在 / 离开 / 离场 / 下班了"
- "我要睡了 / 睡觉了 / 上床了 / 我去睡了"
- "你自己干 / 自动跑 / 无人值守 / 无守护 / 自动施工 / 自己看着办"
- "我去 X，X 小时后回 / 你先做着 / 你接着干"
- "继续加班 / 持续跑 / 能跑多久跑多久 / 跑到天亮 / 跑到熔断"
- "没有总工 / 定期和 X 队握手 / 你们自己对齐信息"

### Step 0 · 立刻确认 4 件必填事

包工头第一反应必须是确认施工契约。不要把所有问题都抛给用户，先确认 4 个必填项，其余使用本 playbook 的安全默认。

1. **持续多久**：1h / 2h / 4h / 6h / 8h / 自定义。必须是小时级或明确截止时间。
2. **施工范围**：清空在途任务卡 / 完成下一阶段某模块 / 自由发挥 / 参考指定项目或文档推进。
3. **自动 commit / push 策略**：
   - A · 自动 commit + 启动同步推送 base 分支一次 + 后续 push 到 `auto/YYYY-MM-DD-HHMM`（默认推荐）
   - B · 自动 commit 到 `auto/YYYY-MM-DD-HHMM`，不 push
   - C · 不 commit / 不 push，仅保留 working tree 改动（用户明确要求才选）
4. **阻塞策略**：
   - A · 跳过阻塞卡继续下一张（默认，最符合无人值守）
   - B · 选最保守方案继续
   - C · 停下等回（仅用户明确希望保守等待时）

安全默认：

- **持续语义**：跑到用户给定时间；当前卡完成、当前阶段完成、队列清空都不是默认退出原因。
- **队列下限**：默认至少 3 张候选卡；4 小时及以上默认至少 5 张候选卡。
- **队列补给**：队列空且时间未到时，必须按补给器继续找安全卡。
- **业务参考源**：用户说"参考 X"时，X 是决策来源；不得因为业务细节不确定就停下。
- **决策不中断**：遇到需要用户拍板的问题，默认按包工头推荐方案继续；重大决策和阻塞写入技术债账本。
- **目标完成后续跑**：用户下达的目标已完成但时间未到时，自动从蓝图下放下一步方向继续施工。
- **交付物形态**：commit 序列、任务卡清单、心跳、总报告、推送状态、退出自证全开。

如果用户已经明确给出时间、范围和无人值守意图，且没有禁止自动 push，默认选择 **A · 自动 commit + 临时分支 push**。如果用户明确禁止 push，选择 B。

### Step 0.5 · 持续运行契约

命中"持续跑 / 能跑多久跑多久 / 继续加班 / 跑到天亮 / 定期握手"时，包工头必须在启动确认里明确以下契约：

| 契约项 | 硬规则 |
|---|---|
| 时间窗 | 必须有开始、截止、最长运行时长；没有时间窗就追问，不得自行设"单卡结束即止" |
| 队列 | 除非用户明确说"只做这一张"，否则启动队列至少 3 张候选卡 |
| 续跑策略 | 当前卡完成且时间未到时，必须取下一张；队列空时必须补给，不得把队列空当自然结束 |
| 业务参考 | 用户给了参考项目 / 文档 / 仓库时，必须优先参考它继续决策 |
| 决策处理 | 遇到决策问题不等用户；按推荐方案继续，重大取舍记入 `docs/技术债账本.md` |
| 目标完成 | 用户目标完成但时间未到时，读蓝图并自动下放路线图、下一阶段工作、任务卡 |
| 心跳 | 每 30 分钟追加心跳，写清已用 / 剩余 / 当前卡 / 完工数 / push 状态 |
| 空跑识别 | 连续思考、等待、整理报告，但没有代码 / 测试 / 文档 / commit / QA 证据产出，视为空跑 |
| 合法退出 | 只允许：时间到、用户回来、连续阻塞熔断、护栏违规、系统或 Git 不可恢复异常、启动同步推送失败且用户未允许降级 |
| 非法退出 | 当前单卡完成、当前段完成、队列清空、包工头觉得差不多了、业务细节需要参考但懒得参考，均不是合法退出原因 |
| 退出自证 | 总报告必须包含"是否跑满用户时间"；未跑满必须列出硬阻塞证据和已尝试的补给路径 |

**事故反例**：用户给 4 小时无人值守，并明确说"业务能力参考 donhauser-design"，包工头只跑 20 分钟后以"队列耗尽 / 等用户决策"退出。这是重大流程缺陷。

## 二、自动模式硬性护栏

### 施工半径

- 默认只做 S / M 卡。
- 用户明确授权"无人值守持续推进"，且给了业务参考源 / 明确范围时，可以做 **L-lite**：可回滚、可迁移、无破坏性的模型补齐 / 管理页搭建 / 文档映射 / 测试补齐。
- 以下永远不自动做：破坏性 schema 迁移、生产数据操作、支付 / 安全 / 鉴权核心逻辑、密钥配置、不可逆删除、依赖体系大改。
- 触发真正 L 级决策时，不停工、不等待用户；先写入 `docs/技术债账本.md`，再按推荐的可回滚路径继续。若推荐路径仍触碰禁区，则记录债务并切到下一张安全卡。

### 决策不中断与技术债账本

无人值守模式下，"需要用户决策"不是停车理由。

处理顺序：

1. 包工头先给出推荐方案，并按推荐方案继续推进。
2. 如果推荐方案会产生长期影响，写入 `docs/技术债账本.md`。
3. 如果遇到阻塞，先记录阻塞、已尝试方案、推荐后续，再切换到下一张安全卡。
4. 如果所有卡都遇到决策问题，继续执行蓝图下放器或 QA / 测试 / 文档债卡，不得空等。

技术债账本字段：

- 时间、来源任务卡、类型、严重度、推荐方案、已采取动作、未来需要用户确认的问题、回滚方式、关联 commit。

---

### 操作禁用清单

| 类别 | 禁用项 | 理由 |
|---|---|---|
| 数据 | 不做破坏性 schema / migration / SSOT 重写 | 数据迁移失败不可逆 |
| 文件系统 | 不删除任何既有文件，除非用户明确点名 | 删错无法恢复 |
| Git | 不 `reset --hard` / 不 `push --force` / 不 `rebase` / 不删 commit | 历史不可恢复 |
| 依赖 | 不引入新依赖，除非已有项目规范明确要求且风险可控 | 引入依赖是架构决策 |
| 安全 | 不动安全 / 钱 / 权限 / 鉴权核心代码 | 错了出大事 |
| 配置 | 不动 `.env` / 密钥 / 证书 / 生产部署配置 | 隐性生产风险 |
| 文档规范化 | 不触发 `playbooks/15-项目文档规范化.md` | 移动文件需要用户在场确认 |

### Git 与推送策略

无守护模式允许自动 push，但只能按下面顺序：

1. 识别 base 分支：`git rev-parse --abbrev-ref HEAD`。
2. `git fetch --all --prune`，记录远程状态。
3. 如果工作区不 clean：除非用户已明确允许 preflight commit / stash，否则拒绝启动。
4. 如果当前在 `main / master / production / release/* / prod/*`：
   - 只允许做启动同步推送：`git push origin <base>:<base>`。
   - 同步完成后必须立刻新建 `auto/YYYY-MM-DD-HHMM` 分支。
   - 后续任何 commit / push 都不得落到 base 分支。
5. 如果当前不在保护分支，也仍然记录 base，并按用户策略决定是否先 `git push origin <base>:<base>`。
6. 新建临时分支：`git switch -c auto/YYYY-MM-DD-HHMM`。
7. 每张完工卡单独 commit。
8. 若启用 push：首次 `git push -u origin auto/YYYY-MM-DD-HHMM`，后续 `git push`。
9. push 失败时重试 3 次；临时分支 push 仍失败则继续本地 commit 并标记 push degraded；base 启动同步 push 失败则停止，除非用户明确允许降级为不 push。

禁止项：

- 禁止 `git push origin main` 用于施工成果。
- 禁止 force push。
- 禁止 rebase 远程历史。
- 禁止为了让 push 成功而丢弃用户改动。

### 多队

- 用户没有明确要求多队协同时，检测到多队必须拒绝标准无守护模式。
- 用户明确要求"定期和 X 队握手 / 多队继续跑"时，只能进入半守护协同模式：
  - 必须填写握手对象、间隔、边界；
  - 必须写 `docs/施工队-临界占用.md`；
  - 默认只做一个文件域 / 一个焦点；
  - 当前焦点结束且时间未到时，仍然必须补给下一张安全卡。

### 模型档位

- 包工头 / 监理：可继续用强模型。
- 工人：强制便宜模型，除非用户明确要求质量优先。
- 心跳 / 文档对齐：便宜模型即可。

### 循环熔断

| 触发 | 行为 |
|---|---|
| 连续 3 次阻塞 | 退出主循环，输出当前总报告 |
| 单卡审计连续 2 次不通过 | 标该卡跳过，进下一张，记入禁忌候选 |
| 单 M 卡施工 > 30 分钟 | 切下一张，标阻塞 |
| 时间预算耗尽 | 优雅退出 |
| 用户发任何消息 | 立刻退出，输出当前状态 |
| 检测到自身违反护栏 | 立刻退出，保留证据，输出违规报告 |
| 队列清空且时间未到 | 必须执行队列补给器，不允许退出 |
| 连续 2 个空跑检查周期无实物产出 | 视为流程违规，输出报告并停止 |

### 可回滚保证

- 自动模式启动时必须新开 `auto/YYYY-MM-DD-HHMM` 分支。
- 每张完工卡单独 commit。
- commit message 强制带前缀：

```text
[auto] [card YYYY-MM-DD-NNN] <一行描述>

任务卡：docs/任务卡/YYYY-MM-DD-NNN-xxx.完工.md
工人：自动模式 worker
证据：tsc pass · lint pass · <真实运行时简述>
```

## 三、防睡眠

### macOS

主循环开始前：

```bash
caffeinate -i -t <用户给的秒数> &
echo $! > /tmp/donghe-caffeinate.pid
```

主循环退出：

```bash
kill $(cat /tmp/donghe-caffeinate.pid) 2>/dev/null
rm -f /tmp/donghe-caffeinate.pid
```

### Linux

```bash
systemd-inhibit --what=idle:sleep --who=donghe-construction-team --why="auto-build" sleep <秒数> &
```

### fallback

如果防睡眠工具不可用，必须告知用户电脑可能睡眠。用户已明确无人值守且要求继续时，可以继续，但总报告必须记录风险。

## 四、自动施工任务队列

文件位置：

```text
docs/任务卡/_自动施工队列-YYYY-MM-DD-HHMM.md
```

主循环开始前必须落盘队列：

- 列出候选任务，建议 5 到 20 张。
- 每张含：优先级、任务标题、半径预判、涉及子项目、涉及文件域、验证方式。
- 优先级排序：
  1. 先做不依赖外部决策的卡。
  2. 先做修复类，再做新增类。
  3. 先做小卡，再做大卡。
  4. 用户给了参考项目时，优先做可从参考项目映射出的业务卡。

### 队列补给器

队列空且时间未到时，必须按顺序补给：

1. 当前模块的下一张自然任务卡。
2. "下一阶段工作"中的同模块任务。
3. 项目路线图 / README / AGENTS.md 中的未完成项。
4. 用户指定参考项目 / 文档中的可迁移业务能力。
5. 用户下达目标已完成时，进入蓝图下放器，自动从蓝图生成路线图节点、下一阶段工作和任务卡。
6. 测试、QA、样式一致性、文档补齐、类型修复、重复代码收敛。
7. 若仍找不到安全卡，写缺口清单和技术债账本，再做验证 / 审计 / 文档对齐类卡，直到时间到或触发熔断。

### 蓝图下放器（无人值守例外）

当用户下达的任务目标已经完成、但时间还没到时，包工头必须自动读取方向链并继续下放：

1. 读取 `docs/蓝图.md` 或 AGENTS.md 中映射的等价蓝图文档。
2. 选择最高价值、依赖最少、可回滚的 💭 蓝图条目。
3. 将它下放为 `docs/路线图.md` 的计划中节点，标记来源为"无守护自动下放"。
4. 将路线图节点下放为 `docs/下一阶段工作.md` 的阶段焦点，标记为"自动下放，待用户复核"。
5. 从阶段焦点拆 1 到 N 张任务卡，写入 `parent_focus`。
6. 继续施工，直到时间到。

限制：

- 不创建全新的蓝图愿景；只能从已有蓝图 / 用户指定参考源 / 已有路线图中取材。
- 下放内容必须标记"待用户复核"，并进入总报告。
- 下放过程中的重大取舍写入 `docs/技术债账本.md`，但不得因此停止。
- 如果蓝图不存在或为空，退回参考项目、路线图、QA、测试、文档债补给，不得空跑。

## 五、主循环

```text
Step A · 启动前自检

A1. 项目是否已接入东合？
    否 -> 拒绝启动。
A2. 当前是否多队模式？
    是且用户未明确要求多队协同 -> 拒绝启动。
A3. git 工作区是否 clean？
    否且用户未授权 preflight commit / stash -> 拒绝启动。
A4. 用户是否完成 4 个必填确认？
    否 -> 回到 Step 0。
A5. 是否已记录持续运行契约？
    否 -> 拒绝启动。
A6. fetch 远程并记录 base 分支。
A7. 若启用 push，先 push base 分支一次。
A8. 新建 auto/YYYY-MM-DD-HHMM 分支。
A9. 落盘启动确认、队列文件、心跳文件。
A10. 启动防睡眠。

Step B · 主循环

start_time = now()
deadline = start_time + 用户给的时长
连续阻塞次数 = 0
空跑次数 = 0

while True:
    if now() >= deadline:
        break "时间预算耗尽"
    if 检测到用户消息:
        break "用户中途回"
    if 检测到护栏违规:
        break "护栏违规"
    if 连续阻塞次数 >= 3:
        break "连续 3 次阻塞"

    if 队列中无待做:
        执行队列补给器
        if 仍无代码卡:
            生成验证 / QA / 文档对齐卡
        continue

    if 用户下达目标已完成 and now() < deadline:
        执行蓝图下放器
        将自动下放结果写入队列
        continue

    候选 = 队列.取优先级最高待做
    候选.状态 = 在做
    任务卡 = 按 03-任务卡规范拆卡并落盘

    if 任务卡触碰禁区:
        候选.状态 = 跳过
        写技术债账本
        执行队列补给器
        continue

    派 1 个工人施工
    运行真实验证
    监理审计

    if 监理通过:
        git add 白名单
        git commit -m "[auto] [card xxx] xxx"
        if push 策略开启:
            git push 临时分支，失败最多重试 3 次
        候选.状态 = 完工 + commit hash + push 状态
        连续阻塞次数 = 0
        空跑次数 = 0
    elif 可重做且重做次数 < 2:
        重做
    else:
        候选.状态 = 跳过或阻塞
        连续阻塞次数 += 1

    每 10 到 15 分钟执行空跑检查：
        如果本周期没有代码 / 测试 / 文档 / commit / QA 证据:
            空跑次数 += 1
            立即补给下一张更小的安全卡
        如果 空跑次数 >= 2:
            break "连续空跑，流程违规"

    每 30 分钟写心跳：
        已用 / 剩余 / 当前卡 / 完工数 / 阻塞数 / 分支 / 最近 push 状态

Step C · 优雅退出

C1. kill caffeinate。
C2. 若启用 push，最终 push 临时分支一次。
C3. 写自动施工总报告。
C4. 在 AGENTS.md 自动施工历史追加一行。
C5. 不切回原分支，保留 auto 分支给用户审。
C6. 输出起止时间、是否跑满、退出原因、commit 序列、push 状态、待决策清单。
```

## 六、心跳

每 30 分钟追加一行到 `docs/施工日志/<今天>/自动施工心跳.md`：

```text
[12:30] 已用 0:30 / 总 4:00 / 剩余 3:30 · 完工 1 · 跳过 0 · 阻塞 0 · 当前 #003-接入客户登录 · 分支 auto/2026-07-09-1230 · push ok
```

心跳必须服务用户回来接管：

- 看到是否真的在跑。
- 看到是否接近空跑。
- 看到最近一次提交和推送状态。
- 看到下一个计划动作。

## 七、用户中途回 / 自然结束

### 用户中途发任何消息

- 包工头识别为打断，立刻退出主循环。
- 先执行 Step C，再回应用户。
- 输出当前状态、总报告路径、分支、push 状态、建议下一步。

### 自然结束

- 只有时间到、熔断、护栏违规、用户回来、不可恢复异常可以自然结束。
- 队列清空不是自然结束；必须补给。

---

## 八、自动施工总报告

报告必须包含：

1. 元信息：起止时间、用户给定时长、实际运行时长、是否跑满、退出原因。
2. 分支信息：base 分支、base 启动同步 push 状态、auto 分支、auto push 状态。
3. 数据汇总：完工 / 跳过 / 阻塞 / 心跳 / push 成功失败次数。
4. 完工清单：每张卡、commit hash、push 状态、任务卡链接、真实验证证据。
5. 跳过清单：跳过原因、是否进入技术债账本 / 待用户复核。
6. 阻塞清单：阻塞原因、已尝试动作、下一步建议。
7. 队列补给记录：什么时候补给、补给来源、补给结果。
8. 蓝图自动下放记录：从哪个蓝图条目下放到了哪些路线图 / 下一阶段 / 任务卡。
9. 技术债账本摘要：重大决策、阻塞、推荐方案、已采取动作。
10. 空跑检查记录：每次检查是否有实物产出。
11. 待用户复核清单。
12. 风险提示。
13. 下一步建议：审临时分支、squash merge、cherry-pick、废弃。
14. 退出自证：如果未跑满，必须证明为什么不能继续。

## 十、自动模式反例

- 没确认时间窗就开始自动跑。
- 用户给 4 小时，只跑 20 分钟就说"队列耗尽"。
- 用户给了参考项目，但不去参考，直接停下等用户。
- 遇到决策点就停下等用户，而不是按推荐方案继续并记入技术债账本。
- 用户目标完成但时间未到，没有读蓝图继续下放下一步方向。
- 自动模式下做了不可逆 L 级架构决策。
- 自动模式下动了破坏性 schema / 删文件 / force push。
- 启动后在 main 上 commit 施工成果。
- 队列空了但没有执行补给器。
- 心跳没写，用户回来不知道发生了什么。
- 用户中途发消息后还继续施工。
- push 失败不重试、不记录、不降级说明。
- 空跑两个周期但仍假装在推进。

## 十一、老 auto/ 分支处置建议

- 用户处置后在 `AGENTS.md` 的自动施工历史表标记：merged / cherry-picked / discarded。
- 累计未处置 >= 3 个时，包工头在接活契约里主动提醒清理。
- 已 merged 到主线后可删除本地和远程 auto 分支。
- 已废弃时，建议沉淀失败原因到 `docs/工地禁忌.md`。

包工头不主动删除用户未确认的 auto 分支。

## 十二、不支持的能力

- 不支持 force push。
- 不支持在保护分支承载施工成果。
- 不支持用户回来后还继续无守护施工。
- 不支持绕过启动确认直接长跑。
- 不支持为了跑满时间而做无意义改动；必须有实物产出。
