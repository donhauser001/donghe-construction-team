# 15 · 项目文档规范化

> 适用：项目首次接入东合施工队、踏勘报告发现项目文档布局不规范时；或用户主动要求"做一次工程入场规范化"。
>
> **铁律：只动文档，不改代码。**
>
> v0.4.9：默认采用"薄壳映射 / 索引修复 / 不移动原文"的轻量方案。移动、重命名、删除、git 操作才需要用户显式确认。

## 零、绝对边界（违反即视为重大违规）

### ❌ 永远禁止做的事

- **不改任何代码文件**：`.ts .tsx .js .jsx .py .go .rs .java .kt .swift .c .cpp .h .hpp .cs .rb .php` 等
- **不改任何配置文件**：`package.json / tsconfig.json / Cargo.toml / pyproject.toml / go.mod / Gemfile` 等
- **不改任何 SQL / migration / schema 文件**
- **不改任何运行时文件**：`.env / Dockerfile / docker-compose.yml / .gitignore / Makefile`
- **不改任何业务数据 JSON / YAML / TOML**
- **不动 git 历史**（不 reset / 不 rebase / 不 force push / 不删 commit）
- **不删除任何文件**（哪怕是用户明显不要的——只移动 / 重命名，删除留给用户）

### ✅ 允许做的事

- **移动**纯文档文件：`.md / .markdown / .rst / .txt / .adoc`
- **重命名**纯文档文件
- **创建新目录**（建立东合标准骨架）
- **补全文档元信息**（追加 frontmatter / 追加目录索引）
- **创建索引文档**：`AGENTS.md` / 各级 `README.md` 的目录段
- **修改 `AGENTS.md`** 的等价物映射表 / 子项目清单 / 已接入施工队列表
- **建软链接**（如果用户允许且 OS 支持）

### 📝 灰色地带（需用户特批）

- 修改 README.md 中"过时的目录树描述"——**不允许**自动改，只能提醒用户
- 修改文档内的文件路径引用（grep 找出会失效的链接）——**不允许**自动改，只列清单让用户改
- 移动 `.gitignore` `.editorconfig` 等"非纯文档但是配置文档"——**不允许**

## 一、何时触发本流程

按强度分两类：

### 强触发（用户明确同意）

- 用户在踏勘报告中勾选"是，做一次项目文档规范化"
- 用户主动说："给我做一次入场规范化 / 文档整理"

### 软触发（仅建议，不擅自执行）

包工头在踏勘 / 接活时发现：
- 文档散落在项目根目录而非 `docs/`
- 文档命名混乱（无版本号 / 无日期 / 多种风格混用）
- 没有 `docs/` 目录但有大量 .md
- 缺失 AGENTS.md 但项目有重要约定文档

→ 在踏勘报告 / 接活契约的"建议"段提一句"建议做一次项目文档规范化"，**等用户拍板**。

## 二、规范化的目标布局

```
项目根/
├── AGENTS.md                              # 项目级 agent 指南（含等价物映射）
├── README.md                              # 项目入口（保留用户原版，不改内容）
└── docs/
    ├── 工地踏勘报告.md                     # 当前最新踏勘
    ├── 蓝图.md                             # 或薄壳指向用户原文档
    ├── 路线图.md                           # 或薄壳指向用户原文档
    ├── 下一阶段工作.md
    ├── 项目知识库.md
    ├── 工地禁忌.md
    ├── 任务卡/
    ├── 施工日志/
    ├── 踏勘历史/
    ├── 施工队-临界占用.md                  # 多队场景启用
    └── 参考资料/                           # 用户原始文档归此（如不改名）
        ├── 科科灵2.0-后端与网页端建设蓝图-v1.md
        ├── 医疗陪诊平台-开发计划-v1.md
        └── ...
```

**注意**：`参考资料/` 目录是**可选方案**，详见"四、三种处置策略"。

## 三、规范化流程

### Step 1 · 评估

包工头扫描项目，输出《项目文档规范化方案》（用 `templates/项目文档规范化方案模板.md`）。方案必含：

| 段 | 内容 |
|---|---|
| **不需要改的** | 已规范的文档（清单） |
| **建议移动的** | `当前路径 → 目标路径` 清单 + 原因 |
| **建议重命名的** | `旧名 → 新名` 清单 + 原因 |
| **等价物映射建议** | 用户已有 X 文档 → 东合 Y 角色（详见"四"） |
| **建议新建的骨架** | 缺失的标准文件清单 |
| **受影响的引用** | grep 找出引用了将变路径的文件清单（不自动改） |
| **执行后的目标布局** | 树形示意图 |

### Step 2 · 用户分项确认

用户按段确认 / 取消 / 修改。**包工头不能"打包默认全部接受"**。

### Step 3 · 执行（按用户确认的清单）

#### 3.1 移动文件

**优先用 `git mv`** 而非 `mv`，保留 git 历史：

```bash
git mv 旧路径 新路径
```

每次移动**只动一个文件 / 一组明显相关的文件**，便于回溯。

#### 3.2 重命名文件

同样用 `git mv`：

```bash
git mv 旧名.md 新名.md
```

#### 3.3 新建骨架

按 `templates/` 下的模板创建空骨架：
- `docs/任务卡/`（目录，加 `.gitkeep`）
- `docs/施工日志/`（目录，加 `.gitkeep`）
- `docs/踏勘历史/`（目录，加 `.gitkeep`）
- 各 .md 文件按对应模板创建

#### 3.4 写入等价物映射

在 `AGENTS.md` 写"东合标准角色 → 项目实际文件"映射表（详见"四"）。

#### 3.5 不改引用

grep 找出"引用了将变路径"的文件 → **列清单给用户**，**不自动改**。
理由：自动改引用风险高（容易破坏链接 / 改错 / 误改代码内的字符串）。

### Step 4 · 输出《规范化完成报告》

包含：
- 已动过的文件清单（git diff 可证）
- 受影响的引用清单（待用户决定是否更新）
- 建议的 git commit 信息（让用户决定是否提交）
- 后续建议（如"建议把'参考资料/'下的旧版本文档纳入归档计划"）

**包工头不主动 git commit**——是否入库由用户决定。

## 四、三种处置策略（关键决策）

用户已有文档（如 `科科灵2.0-后端与网页端建设蓝图-v1.md`）和东合标准角色（蓝图）的处置：

### 策略 A · 薄壳指向（推荐 · 最安全）

- 在 `docs/蓝图.md` 创建薄壳文件：1~2 屏摘要 + 一行链接指向用户原文档
- 用户原文档**不动**（位置 / 命名 / 内容都保留）
- AGENTS.md 写映射表："蓝图 = `docs/蓝图.md`（薄壳，详见 `参考资料/科科灵2.0-后端与网页端建设蓝图-v1.md`）"

**优点**：零破坏 + 项目布局规范 + 包工头能找到入口
**缺点**：薄壳和原文档要同步更新（漂移风险）

### 策略 B · 移动归档（中等）

- 把用户原文档**移动**到 `docs/参考资料/` 下（保留原名）
- 在 `docs/蓝图.md` 创建薄壳，指向 `docs/参考资料/<原名>`
- 用 `git mv` 保留历史

**优点**：项目根 / docs/ 一级更整洁
**缺点**：可能破坏外部引用（用户文档里互相引用、代码里引用）

### 策略 C · 重命名替代（破坏性大，不推荐）

- 把用户原文档**重命名**为 `docs/蓝图.md`
- 删除原名

**只有用户明确同意时才能用**。强烈建议改为 A 或 B。

### 策略 D · 不动用户文件，仅 AGENTS.md 映射

- 不创建任何薄壳，不移动任何文件
- AGENTS.md 写："蓝图 = `docs/科科灵2.0-后端与网页端建设蓝图-v1.md`"
- 包工头读"蓝图"时按映射跳转

**优点**：完全零破坏
**缺点**：项目布局不规范，新人 / 新 agent 接入要先读 AGENTS.md 映射

**默认推荐 A**，用户拒绝时降级到 D。

## 六、规范化反例（工地禁忌候选）

- ❌ 用 `mv` 移动文件（git 历史断裂）
- ❌ 一次性"打包接受全部建议"，没逐项确认
- ❌ 自动改文档引用导致代码内字符串被误改
- ❌ 删除用户文件（哪怕是"明显重复"的）
- ❌ 改了代码 / 配置 / schema（违反铁律）
- ❌ 自动 git commit
- ❌ 用策略 C 没经用户明确同意
- ❌ 把"参考资料/"下的文档当成可以随便改的二等公民

## 七、规范化与首次踏勘的衔接

```
首次踏勘 → 输出踏勘报告 → 包含"是否建议规范化"的判断
                          ↓
            用户拍板：是 → 进入本 playbook
                       否 → 仅做 AGENTS.md 映射（策略 D）
                       延后 → 记入项目知识库待办
                          ↓
            规范化完成 → 更新踏勘报告（标"已规范化 v?"）→ 进入正常接活
```

## 八、规范化与多队场景

多队场景下：

- **只能由主队**发起规范化（其他队提议后转交主队）
- 规范化期间**临界占用大量全局文档** → 必须在 `docs/施工队-临界占用.md` 写"规范化进行中，预估 X 时长"
- 别队**暂停所有涉及全局文档的工作**，直到规范化完成
- 规范化完成后所有队需要**重新做一次轻量对齐**（读新版 AGENTS.md 的映射表）
