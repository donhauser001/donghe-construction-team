# 02 · 改动半径分级

> 适用：每次接活契约写完后、动手前判断本次工作的"重量级"。

## 一、三档定义

### S 级 · trivial（轻量）
**典型场景**
- 单文件单函数内部修改
- 修一个 typo / 注释 / 文案
- 改一个明显的样式参数（颜色 / 间距）
- 调一个明显的 lint 报错
- 改一行 import / rename 一个内部变量

**判定要点**
- 不动公共契约（API、数据库 schema、共享组件 props）
- 不影响其他文件的使用方式
- 不引入新依赖
- 改完不需要其他人重新理解

### M 级 · standard（标准）
**典型场景**
- 单功能模块内的功能开发 / 修复
- 加一个表单字段（前后端联动但不动 schema）
- 重构一个函数 / 拆分一个组件
- 调整一个内部 helper 的签名 + 调用点
- 修复一个跨文件但不跨层的 bug

**判定要点**
- 改动 ≤ 5~8 个文件
- 不动数据库 schema、不动 SSOT 模型
- 不动跨端契约（前后端 API、agent 工具签名）
- 一个工人能 1~3 小时内独立完成且能自检

### L 级 · critical（关键）
**典型场景**
- 跨层修改（前端 + 后端 + DB 任意两个以上）
- 跨端修改（Web + Mobile + Admin + CLI 任意两个以上）
- 改数据库 schema / 加迁移
- 动核心 SSOT / 领域模型 / 业务状态机
- 加一个新的 agent 工具 / 改 agent 工具签名
- 一个明确的"节点交付"（路线图上的一个里程碑）
- 涉及钱、权限、医疗、安全的任何修改

**判定要点**
- 改动 > 8 个文件，或跨多个子项目
- 改完需要跨端 / 跨服务联调
- 改完老数据可能行为变化
- 改完用户能直接感知

---

## 二、判定流程

```
1. 第一感觉是哪一档？
2. 检查"升档触发器"（见下表）—— 命中任意一条就升档
3. 写到接活契约的"改动半径预判"栏
4. 进入对应流程
```

### 升档触发器（出现一条就 +1 档）

- 涉及配置 / 价格 / 阈值 / 限流 / 安全 / 权限边界
- 涉及钱（订单、退款、佣金、对账、发票）
- 涉及医疗 / 健康数据 / 个人隐私
- 改动文件位于"项目核心目录"（如 `prisma/`、`src/auth/`、`src/finance/`、`src/orders/` 等"听名字就知道是核心"的地方）
- 项目知识库里已经标注"此处历史敏感"
- 工地禁忌里有相关条目
- 用户措辞包含"重要 / 关键 / 紧急 / 全量 / 上线 / 发布"

**升档原则**：拿不准时**升一档**；可向上升不可向下降。

## 三、各档对应的流程清单

| 流程节点 | S | M | L |
|---|---|---|---|
| 世界观对齐（`01`） | 跳过 4-7 自检（但仍需快速看下相关文档） | 必做 | 必做 |
| 接活契约（`00`） | 简版（一句话即可） | 必做 | 必做 |
| 任务卡落盘（`03`） | **跳过** | 1 张 | 1~N 张 |
| 派工（`05`） | 不派，自己干 | 1 工人 | 默认 ≤3 工人并行；用户授权且写集互斥时分波 ≤8 |
| 施工（`04`） | 自己施工 | 工人按卡 | 工人按卡 |
| 自检 + 试机（`06`） | 代码核查 + 类型/lint | + 关键 API 真测 + 机器证据 | 真实运行时全测 + 机器证据 |
| 监理审计（`07`） | 跳过 | 抽查 | **必做完整审计** |
| 完工门槛（`08`） | 2 条硬条件 | 4 条 | 8 条 |
| 诚实汇报（`09`） | 简版 | 标准 | 完整带证据 |
| 文档对齐（`10`） | 施工日志 1 行 | 文档预算内 + 项目知识库（如有沉淀） | 文档预算内路线图 + 施工日志 + 项目知识库 + 蓝图（按需） |

## 四、永久豁免与永远不豁免

**永久可豁免（即使是 L 也可以简化）**：
- 紧急生产事故止血（先止血再补流程）—— 但事后必须补一张回溯任务卡

**永远不可豁免**（不管半径大小都必须做）：
- 真实运行时验证（哪怕只是手动跑一遍）
- 当日施工日志（即使是 S 级也要记一笔）
- 完成必须有证据（禁说"应该可以 / 大概率 OK"）
- 工地禁忌识别（发现一条就立刻沉淀）
- **方向锚点**（v0.4 新增 · 八大宪法第 8 条）：任何活必须能反向追溯到一条 `parent_focus`（指向下一阶段工作里的某条焦点条目）。详见下一节"S 级方向锚点豁免规则"

## 四 · 1 · S 级方向锚点豁免规则（v0.4）

S 级活的"方向锚点"要求**比 M/L 级宽松**，但**不能完全没有**：

| 场景 | 是否豁免 `parent_focus` 字段 | 替代方案 |
|---|---|---|
| 修 typo / 注释错字 | ✅ 豁免（仅施工日志记一笔即可） | `日志: [HH:MM] S 级修复 typo @ <文件>:<行号>` |
| 改一个明显的样式参数 | ✅ 豁免 | 同上 |
| 调一个明显的 lint 报错 | ✅ 豁免 | 同上 |
| 改一行 import / rename 内部变量 | ✅ 豁免 | 同上 |
| 任何**与业务功能相关**的 S 级活（哪怕只改一行业务逻辑） | ❌ **不豁免**，必须填 `parent_focus` | 至少在施工日志写"对齐 docs/下一阶段工作.md#<focus_id>" |
| **方向链冻结状态下的 S 级活** | ✅ 全部豁免（冻结时只能接 S 级） | 但必须在日志写"⛔ 冻结期 S 级修复" |

**判断口径**：S 级活如果**改动后产品行为有任何用户可见变化**（即使只改一行），就要 `parent_focus`。零用户可见变化的纯技术修复才豁免。

**反例**（违反 v0.4 第 8 条宪法）：
- ❌ "我改了一行价格计算逻辑，是 S 级所以不用 parent_focus" → 业务相关，必须有锚点（且半径升 M）
- ❌ "我加了一个新按钮跳转，是 S 级所以不用 parent_focus" → 业务相关，必须有锚点
- ✅ "我改了变量名 `tmp1` → `userInput`，无业务影响，仅日志记一笔"

## 四 · 2 · S 级对话隔离原则（v0.4.1 · 防上下文污染）

**核心**：包工头收到 S 级活时，**先判断它是否在当前对话的"任务链条"上**。不在 → **强建议用户新开 agent 对话**做这个 S 级活，避免污染当前对话的上下文。

### 判定矩阵

| 场景 | 处置 | 理由 |
|---|---|---|
| S 级活在**当前任务卡**的施工要点 / 验收点 / 改动范围内 | ✅ **当前对话做**（顺手） | 是任务链条的有机部分 |
| S 级活在**当前任务卡的 `parent_focus`** 焦点条目下的相关旁支 | ✅ **当前对话做**（顺手） | 同一焦点上下文复用 |
| 同一项目**同一阶段但不同焦点** | ⚠️ **默认建议新开**，用户可坚持当前对话 | 焦点切换 = 上下文切换 |
| **完全无关**（不同子项目 / 不同焦点 / 不同阶段 / 不同方向链分支） | ❌ **强建议新开** | 污染最严重，新对话成本几乎为 0 |
| 当前对话**正在 M/L 级深度施工 / 监理审计 / 踏勘对齐**等需要稳定上下文的关键节点 | ❌ **强建议新开**（即使是相关 S 级活） | 关键节点期不应被打断 |

### 建议新开对话的标准话术

```
这个 S 级活看起来不在当前 [<当前任务卡名 或 当前焦点名>] 的任务链条上：
- 当前 parent_focus：<当前焦点条目，如 docs/下一阶段工作.md#focus-1>
- 本次活推断 parent_focus：<推断出的位置，如 docs/下一阶段工作.md#focus-3 / 无方向锚点>

**建议你新开一个 agent 对话**做这个 S 级活，避免污染当前对话的上下文。

新对话只需告诉它：
```
修 <文件路径>:<行号> 的 <typo / lint / 样式参数 / 注释错字>
```

当前对话继续专注 [<当前任务/焦点>]。
```

### 用户坚持在当前对话做时

包工头**尊重用户意愿**，但**必须先在汇报中明确**：

```
✅ 收到，按你的要求在当前对话做。

⚠️ 上下文影响告知：
- 本次 S 级活 ≠ 当前任务链条
- 当前焦点 [X] 的上下文会被部分挤出
- 完成后我会主动复述当前焦点状态，帮你拉回主线
```

### 例外：以下情形包工头**不必建议新开**

- 当前对话刚开始（< 5 轮 tool 使用），上下文还很轻
- 当前对话在等待用户回复 Q 列表（已经处于"等待"态）
- S 级活就是当前对话用户最近一条指令的直接延伸（如刚说"顺便也修一下 X"）

### 反例

- ❌ 用户在 M 级支付重构对话中说"顺便把 about 页文案改一下" → 包工头默默改了（污染了支付重构的上下文）
- ❌ 用户在踏勘对齐对话里说"顺便修个 typo" → 包工头默默改了（污染了踏勘的扫描状态）
- ❌ 包工头机械执行"必须新开"，连"修当前任务卡里的一个 typo"也建议新开（过度防御，违反例外条款）

### 正例

- ✅ 当前在 M 级支付重构，用户说"顺便修 about 页文案" → 包工头："建议新开对话做 about 页文案，当前对话继续支付重构。新对话告诉它：'修 admin-web/src/pages/About.tsx 的副标题文案'"
- ✅ 当前在做任务卡 A，用户说"修一下任务卡 A 改的那个文件里的 typo" → 包工头：当前对话顺手做（在任务链条内）
- ✅ 用户说"我刚发现 docs/工地踏勘报告.md 第 3 行有个错字" → 包工头根据当前对话状态判断，若深度工作中就建议新开

## 五、写入位置

每张任务卡 / 每次接活契约里都要明确标注：

```
半径：M（升档触发器：涉及订单金额计算 → 由 S 升至 M）
```

让自己和用户都能复盘这个判定是否合理。
