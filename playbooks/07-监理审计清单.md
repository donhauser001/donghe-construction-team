# 07 · 监理审计清单

> 适用：包工头切换为监理姿态、或派独立监理子 agent 时。**仅监理读**。
>
> 默认 M 级抽查、L 级必做完整审计、S 级跳过。

## 零、监理不是审批口

监理 / 总工的目标是让施工队更快更稳地把用户链路跑通，不是把自己变成等待项。

- 先判断是否真的需要阻断。只有 P0/P1 或 v0.4.5 高风险范围才允许冻结。
- P2/P3、格式、文案、低风险文档瑕疵记录后补，不挡核心链路。
- 审计输出必须给最小返修路径或可继续施工路径。
- 审计不能替代用户视角实测；审计要服务下一轮实测。

## 一、监理姿态须知

切换为监理时：

- **默认不动代码**。只读、只问、只判定。
- 不替施工工人善后；不补丁；不顺手改。
- 输出**结论标签 + 发现清单 + 修复方向**（不替工人写实现）。
- 不因低风险格式问题、文档缺口、非核心 P2/P3 阻塞前线；此类问题登记后补。
- 对不可验收项必须写清最小返修动作，避免泛泛打回。

唯一例外：用户明确说"你直接修"，再切回包工头姿态进入实施流程。

---

## 二、审计九问

每张待审任务卡，按下面 9 个问题逐一过：

### Q1 · SSOT
- 这次改动有没有把**同一份业务状态**写到两处？
- 所有视图 / client / agent 工具 / 导出路径，是不是都从同一个权威模型读？
- 有没有引入"UI-only" / "缓存-only" / "影子状态"？

### Q2 · 持久化
- 改动结果**刷新页面 / 重启服务 / 重新登录**后还在吗？
- 写库的字段类型 / nullable / 默认值是否合理？
- 是否有未提交的事务、未刷的缓冲？

### Q3 · 撤销 / 重做（如适用）
- 这个操作是否进入统一的历史 / 撤销系统？
- 撤销后回滚是否完整（不是只撤销 UI 状态而留下 DB 写入）？
- 重做是否能恢复到一致状态？

### Q4 · AI / agent 工具对等
- 用户在 UI 上能做的这件事，agent 工具能做吗？
- agent 工具的入参是否有意义、可被其他 agent 自动构造？
- agent 工具是否能读出操作结果（不是只能"盲写"）？
- 详见 `playbooks/11-原生AI应用素质.md`

### Q5 · 失效（staleness）
- 缓存 / hash / 索引 / 预计算 / 派生视图，在源头变化时是否被失效？
- 异步报告 / 校验结果，是否有"过期"机制？
- 用户停留在旧数据上时是否能感知或自动刷新？

### Q6 · 老数据迁移
- 这个改动会让**已存在的老记录**产生什么变化？
- 缺字段 / 旧 enum 值 / 历史 schema 是否有 migration 或 fallback？
- 有没有"只对新数据生效"导致两套并存？

### Q7 · 多视图渲染一致
- 同一份数据在多个视图（编辑 / 列表 / 预览 / 导出 / 详情 / 移动端）是否一致？
- 各视图是否走同一份渲染逻辑（或至少同一份数据源）？

### Q8 · 多端一致
- Web / 移动端 / 后台 / CLI / agent 同时在线时，看到的真相一致吗？
- 一端变化是否能（或者不需要）反映到另一端？

### Q9 · 失败模式
- 多步操作半途失败时，会留下不一致状态吗？
- 批量操作部分失败如何告知用户、如何回滚？
- 实时通道断开时如何降级？
- 网络慢 / 离线 / 并发冲突的行为是什么？

## 三、证据等级（判定证据强度）

工人贴的"证据"属于哪一档？只有第 1~3 档算"已验证"：

| # | 证据形式 | 等级 |
|---|---|---|
| 1 | 真实运行时复现（浏览器 / app / CLI 真操作）+ 持久化数据 | ★★★★★ 最高 |
| 2 | API 真调 + DB 查询 + 实时通道观察 | ★★★★ |
| 3 | 针对性 unit / integration / e2e 测试证明不变量 | ★★★ |
| 4 | 静态代码追踪（带 file:line） | ★★ 不算"已验证" |
| 5 | 仅声称的实施说明 | ★ 不算"已验证" |

**任何 user-visible 行为，必须 ≥ 第 2 档才允许声明"完成"**。

---

## 四、严重度锚点

| 严重度 | 触发场景（任一） |
|---|---|
| **P0** | 数据丢失 / 数据写错位置 / 持久化失败 / 已提交编辑无法撤销 / 安全 / 隐私 / 越权绕过 / 核心流程不可用 |
| **P1** | SSOT 分裂 / UI 与缓存陈旧 / agent 工具假实现 / 报告与已应用状态不一致 / 老数据破坏 / 跨视图或跨端不一致 |
| **P2** | UX 集成不完整 / 关键链路缺测试覆盖 / 校验弱 / 错误信息差 / 可观测性缺口 / 维护性问题 |
| **P3** | 命名 / 文案 / 注释 / 微小润色 |

---

## 五、判定结论（三选一）

### ✅ 可验收

**所有以下条件全部满足**才允许：

- 任务卡所有验收点都打 ✅ 且有 ≥ 第 2 档证据
- 审计九问无 P0 / P1 发现
- 不变量未被破坏
- AI 工具对等性满足（L 级）
- 老数据兼容性已验证（L 级）

### ❌ 不可验收

**任一**条件命中：

- 存在 P0 / P1 发现
- 任务卡有验收点未打勾且无说明
- 工人回报区"真实运行时证据"为空或低于第 2 档
- 不变量被破坏

→ 把卡的当前状态改为 ❌ 不可验收，写明每条 P0/P1 发现 + 最小修复方向，重派给同一工人或拆卡重派。

**不得**因为 P2/P3、EOF 空行、标题层级、轻微措辞、非核心文档缺项，把本可继续实测的核心链路冻结。此类问题走"部分可验收 / 后补清单"。

### ⚠️ 部分可验收

**核心链路**已验证 + **非核心切片**仍有问题。允许接受，但必须：

- 把"未完成的非核心切片"明确列出，开新任务卡跟踪
- 在汇报里标注"部分可验收：xxx 待办"

---

## 六、二次审计（修复后强制）

工人修复后再次提交（卡 → 🔍 待审计）→ 监理**必须**再走一遍 9 问，**不能复用第一次审计结论**。理由：修复可能引入新问题。

二次审计结果同样写在卡内"监理审计区"的"二次审计结果"段。

---

## 七、监理报告写在哪

**只写在任务卡内**（`## 监理审计区`），**不另开文件**。理由：避免文件爆炸，也方便从任务卡完整溯源"任务 / 过程 / 结果 / 状态"。

---

## 八、监理姿态切换提示

切换前对自己说一遍：

> "现在我不再是写代码的那个人。我是来挑刺的。
> 我的目标是发现 P0/P1，不是为'已经做完的事'背书。
> 如果我找不到任何问题，我应该怀疑自己是不是审得不够细，
> 而不是赞美这次工作很完美。"

---

## 九、监理与"工地禁忌"

每次审计如果发现新的反例（哪怕这次没有阻断验收），按 `playbooks/12-工地禁忌识别与沉淀.md` 沉淀到 `docs/工地禁忌.md`。
