# 03 · 派工与并行

> 适用：包工头拆完卡准备派工时。仅包工头读。

## 一 · 并发档位

| 档位 | 上限 | 启用条件 |
|---|---|---|
| **默认** | 3 工人 | 常态 |
| **六槽编制** | 3 职能槽（文书/巡检/审计）+ 3 工人 | 控制面过载 / 用户要求分槽 |
| **8-worker 狂暴** | 8 工人（分波） | **仅用户明确授权**（"8 worker / 狂暴施工"）且 ≥ 5 条独立任务线、写集互斥、证据路径齐全 |

**永不阻塞自己**：派完立刻规划下一批 / 审收回报，不空等所有人完工。

**只派 1 个工人的场景**：改 schema / migration；涉钱权安；首次接触的领域；卡总数 < 3；手握 ≥ 3 张待审计卡（审收已饱和）。

## 二 · 分线规则（防踩脚）

并行卡之间必须满足任一分线：**按子项目（首选）** / 按文件域（白名单互不相交）/ 按依赖层（契约本次不动）/ 按子模块。

- **禁止两个工人同时改同一个文件**（同文件不同函数也禁止）。
- 有依赖的卡串行：A 完工验收后再派 B；共享 schema 拆独立卡 S 先做。
- 需要统一接口契约时，先定契约再派两端；schema 卡验收前，依赖它的工人不许猜字段。

## 三 · 模型档位表（⚠️ 全套文档唯一维护模型 slug 的地方）

> 单一事实来源：slug 只在本节维护，其它文档一律说"便宜档 / 中档 / 强档"。模型换代只改这里。

| 档位 | 当前推荐 slug | 用途 |
|---|---|---|
| **便宜档** | `composer-1.5`（备选 `composer-2-fast`）| 工人默认 · 按清单填空 / 结构化编辑 |
| **中档** | `claude-4.6-sonnet-medium-thinking`（备选 `gpt-5.5-medium`）| 中等代码实现 / 边界判断 |
| **强档** | `claude-4.6-opus-high-thinking`（备选 `claude-opus-4-7-thinking-xhigh`）| 包工头 / 监理 / 架构决策 |

**为什么必须显式指定 model**：Task 不带 `model` 参数 = **继承父 agent 模型**（通常最贵），成本放大 3~10 倍。默认行为不是"自动选最便宜"，是"自动继承最贵"。（沉淀自真实事故：3 个填空型 docs 修正 Task 全部继承 Opus 高思考，被用户中断质疑成本。）

**选择原则**：按**任务难度**选，不按重要度选。清单详细 + prompt 高质量的工人级任务 = 便宜档够用；跨模块推理 = 中档；架构 / SSOT 设计 = 强档（一般包工头亲自）。多并行 worker 选同一档（防风格不一致）。选的 slug 必须在用户 IDE 可用清单内。

## 四 · 派工闸（派 Task 前必过）

1. `scripts/check_card.py <卡路径>` 通过（三元字段 + parent_focus + 白名单 + 证据命令 + 禁词）。
2. Task 调用的 `model=` 与卡上"调用模型"完全一致。
3. 在 chat / todo 里告知用户："派 N 个 worker · 用 X 模型 · 理由 Y"（用户有权干预成本）。
4. prompt 里贴出卡上三字段（供工人开工闸核对）。

**派工 prompt 模板**（路径派工，不内联卡内容——内联会让工人改"参数副本"而非真卡）：

```
Task(
  subagent_type="generalPurpose",   # 或 "shell"
  description="工人#1 · <任务名>",
  model="<卡上的 slug>",            # ⚠️ 必填
  run_in_background=True,           # 多工人并行时必须
  prompt="""
你是东合施工队的施工工人 #1。
先读并遵守：<skill 安装路径>/playbooks/02-任务卡与工人协议.md 第五节起。
你的任务卡（首动作读它）：<项目根>/docs/任务卡/YYYY-MM-DD-NNN-任务名.md

卡上三元字段（开工闸核对用）：
  派给：工人#1 · 调用模型：<slug> · 匹配理由：<一句话>

期望产出：完工 → 状态置 🔍 待审计 + 回报区填全；阻塞 → 状态置 ⏸️ + 文件名加 .阻塞。
最终回复只一句："卡已提交待审计：<路径>" / "卡已阻塞：<路径> · <一行原因>" / "工人退卡：<路径> · <原因>"。
""")
```

**红线**（违反 = 重大违规 + 施工日志记一笔 + 沉淀项目禁忌）：不带 `model` 派工；强档跑填空任务；不告知用户就并行多个贵 worker；check_card 不过就派工；`model=` 与卡不一致。

**违规后急救**：`git status` 看已产出内容 → 给用户三选项（接受部分产出补完 / 回滚重派 / 包工头补完）→ 沉淀禁忌 + 日志记一笔。

## 五 · 回收节奏

工人提交（🔍 待审计）→ **立刻审收，不批量等**（后回来的会撞前面的修改；包工头记忆衰减）→ 通过加 `.完工` 记日志 / 打回写明 P0/P1 + 最小修复方向重派 → 无论结果立刻派下一批。

**工人不空转 / 审计异步**（一级原则）：

1. 工人交完一张即领新卡，不等审计结论。
2. 审计与施工解耦：审计干活时工人继续接新卡。
3. 审计发现**小问题**（单文件单段落、≤ 30 行、不动 export / 不升半径 / 不动写集边界）→ 进 owner 工人"待回炉"清单，等他闲置时派补丁卡回炉。
4. **大问题**（跨卡 / 跨写集 / 升半径 / 动 schema-资金-权限-安全-生产）→ 拆**新任务卡**按新任务派工，**不打断当前工人在工卡**。判不清 = 按大处理。

## 六 · 六槽编制（控制面过载时启用）

```
包工头（父 agent · 只裁决、派发、整合、最终 stage/commit、汇报）
├── 文书：写瘦卡 / 战情板 / 施工日志 / 预备队列          —— 不写业务代码
├── 巡检：git status / 在工卡 / 脏区 / 冲突，每 10-15 分钟报红黄绿  —— 不改文件
├── 审计：白名单 / diff / 证据 / 越界五项核验，给通过/打回  —— 不修代码、不打断工人在工卡
└── 工人 #1 / #2 / #3：按卡施工，交完即领新卡
```

- **文书弹药线**：任何时候保持 ≥ 3 张可领取预备卡；低于即优先补卡。
- **暂停派工的唯一条件是文书弹药线 < 3**；待审 ≥ 3 只警觉不暂停（工人有卡就继续）。
- **信息压缩线**：所有子 agent 回包工头必须压缩为"状态 + 证据链接 + 需裁决的问题"，禁贴长文档 / 长 diff / 长测试输出。
- 工人有疑问先写卡内"阻塞/疑问"段，由巡检/审计带回，不排队等包工头口头解释。
- 文书 / 巡检 / 审计的派工 prompt：一句角色 + 一句职责边界 + 一句回报格式（参照上面的角色表写）。

**降级回默认 3-worker**（任一触发）：待审 ≥ 3 且预备卡 < 3 且 20 分钟未回收；误 stage / 误带别队文件；多 worker 需要同一公共文件；包工头开始亲自补多角色的活；待回炉 ≥ 5 条无人闲置；包工头在读未压缩长报告。

## 七 · 8-worker 狂暴模式

启用前提：六槽稳定 + ≥ 5 条独立任务线（少于 5 按实际派，不硬凑）+ 用户明确授权。

- **分波**：第一波最多 4 代码 worker + 1 证据/测试 worker；第二波要等第一波 ≥ 2 张证据齐全。
- **战情板必填**：工人 / 任务卡 / 写集 / 状态 / 证据文件 / commit hash / 阻塞 / 回收顺序——只放机器可核对字段，不写长描述。
- **一共享契约一工人**：API 契约 / schema / 路由注册 / 公共类型 / 公共组件只归一个 worker，其他只消费。
- **一生产资源一工人**：SSH / Docker / migration / 支付回调 / 凭证同一时段只给一个。
- 一子项目最多 2 个代码 worker；代码线不够 8 时用测试 / 复测 / 文档补位，不拆碎代码。
- 每个 worker 精确 stage 自己白名单，**严禁 `git add .`**；冲突文件出现立刻冻结相关 worker 提交，包工头裁决。
- 包工头手上待裁决 P0/P1/P2 > 3 项 → 立刻降回 3-worker 先清队列。

**禁开场景**：schema 未稳定；多卡要改同一契约 / 公共工具 / store；生产资源需多点操作；无战情板 / 无滚动审收 / 无证据路径；为"看起来忙"硬拆任务。

## 八 · 反例（工地禁忌候选）

- 同文件两工人同时改；派完工空等；隐性依赖未声明（A 等 B 产出）。
- 一张大卡硬塞一个工人应拆未拆；派工后不再读卡全权放飞。
- 8-worker 无战情板 / 无分波 / 8 个同时交付逼包工头靠记忆整合。
- 6 槽全派成工人，包工头自己写卡巡检审计。
- 审计发现小瑕疵让工人停工等修复（工人空转）；看到待审 ≥ 3 就暂停派工却不看文书弹药线。
- 子 agent 把整份测试输出 / diff 复制给包工头（控制面过载）。
