# 06 · 项目接入与踏勘

> 适用：首次进入项目（强制踏勘）；重新踏勘；项目文档规范化。

## 一 · 首次进入（强制，踏勘完成前禁止接活）

**铁律：改用户文件之前先告知；Step 4 之前不写任何项目文件（纯读除外）。**

### Step 1 · 扫描

根目录 + 子项目结构；Glob 所有 `*.md`；`package.json` 等清单识别技术栈；`git log --oneline -n 30`；`git status`。含中文路径必须用 `git -c core.quotepath=false status --short`（否则路径变 `\xxx` 转义看不清）。顺手扫项目根的非代码资产（图片 / 压缩包 / 设计稿）。

**工作区清洁度**：未追踪 + 已修改文件数 N ≤ 5 🟢 顺带一提；6~29 🟡 风险段列出；≥ 30 🔴 **强制入 Q 列表让用户定 commit 策略，本轮禁止擅自 `git add`**。整个子项目 untracked = 🔴 极高风险。

### Step 2 · 通读已有文档（只读）

所有 `docs/*.md`（含"蓝图 / 规划 / 路线图 / 契约 / 规范"关键词的优先）+ 各级 README。大文件先读前 200 行看大纲。收集：术语、约定、SSOT、安全约束、agent 工具能力。

### Step 3 · 等价物识别

按文件名 / 内容启发式映射：含"蓝图 / 总体设计" → 蓝图；"路线图 / 开发计划 / 里程碑" → 路线图；"任务 / TODO" → 候选任务卡；"代码规范 / 约定" → 知识库注入。输出映射建议，**最终由用户确认**。

### Step 4 · 落盘踏勘报告 + 聊天复述

用 `templates/工地踏勘报告模板.md` 写，至少含：项目快照（技术栈 / 子项目 / git 状态 / 清洁度 / 根资产）、文档清单 + 映射建议、缺失骨架清单、世界观（不变量 / SSOT / 约定 / 安全）、方向链快照（4 层探查，按 `01-接活流程.md` §六）、是否建议文档规范化、Q 列表（待拍板项）。

**落盘分级**（首次踏勘与重新踏勘通用）：

| 级 | 动作 | 确认 |
|---|---|---|
| **L1** | 踏勘报告本身落盘 / 旧版归档到 `docs/踏勘历史/` | 默认直接做（不落盘下次找不到上下文；不满意可覆盖重写，零成本） |
| **L2** | 建骨架 / AGENTS.md 写入 / 改文档结构 | **必须显式确认** |
| **L3** | 任何 git 写操作（add / commit / stash / mv / switch） | **必须显式确认** |

**授权口径**："落盘骨架 / 建 / go / Q5=A / 按建议来" = 授权；"嗯 / 好 / 看起来不错" = **不算**，须反问一次；用户沉默或转问别的 = 按兵不动。

### Step 5 · 用户回复 Q 列表后

覆盖式重写报告（拍板结果入正文）→ 按确认建骨架（`docs/任务卡/` `施工日志/` `踏勘历史/` `项目知识库.md` `工地禁忌.md` `下一阶段工作.md` 及蓝图/路线图薄壳）→ `AGENTS.md` 写入 `<!-- donghe-construction-team: initialized YYYY-MM-DD · vX.Y -->` + 映射表 + 子项目清单 + 方向链指针 → 按拍板执行 git 操作 → 进入正常接活。

## 二 · 重新踏勘（diff 模式，不从零跑）

**强触发**（必须跑）：路线图里程碑完成；技术栈 / 子项目 / schema 重大变更；方向链任一层耗尽；用户说"重新踏勘"；对齐时发现明显漂移。
**软触发**（只提醒不强制）：距上次 ≥ 4 周；新队加入。

**每次接活前顺手做新鲜度自检**：看报告"踏勘时间"、路线图是否有新 ✅、结构是否新增——评估 ✅ 新鲜 / ⚠️ 偏旧 / ❌ 严重过期，写进接活契约风险段。**提醒归提醒，做不做用户定。**

流程：读旧报告 → 重扫变化部分 → 输出《踏勘对齐报告》（`templates/踏勘对齐报告模板.md`：触发原因 / 变化清单 🆕✅❌📈🌪 / 新发现 / 方向链快照 / 建议动作）→ 按 L1/L2/L3 落盘（L1 = 旧版归档 + 覆盖报告；L2 = AGENTS 时间字段 / 阶段文档；L3 = git）。

回滚：用户不满意 → 从 `docs/踏勘历史/` 复制回来即可（所以 L1 默认落是安全的）。

**反例**：不归档旧版直接覆盖；重扫时复制粘贴上一版敷衍；漂移不告知继续按旧报告干；每次接活都做成重新踏勘（分不清轻量对齐和踏勘）。

## 三 · 项目文档规范化（用户拍板后才执行）

**铁律：只动文档，不改代码。** 默认走"薄壳映射 / 索引修复 / 不移动原文"的轻量方案。

**永远禁止**：改任何代码 / 配置 / SQL / migration / `.env` / Dockerfile / 业务数据文件；动 git 历史；**删除任何文件**（删除留给用户）。
**允许**：移动 / 重命名纯文档（`.md .txt .rst`）；建目录骨架；建索引；改 AGENTS.md 映射表。
**灰色地带**（列清单给用户，不自动改）：README 里过时的目录树；文档内将失效的路径引用。

**流程**：输出《规范化方案》（不动的 / 建议移动的 / 建议重命名的 / 映射建议 / 新建骨架 / 受影响引用 / 目标布局）→ **用户逐项确认（不许打包默认全收）** → 执行（移动一律 `git mv` 保历史）→ 输出完成报告（动过的清单 + 待用户处理的引用 + 建议的 commit 信息）→ **不主动 commit**。

**四种处置策略**（用户已有文档 vs 东合角色）：**A 薄壳指向**（推荐：`docs/蓝图.md` 做 1~2 屏摘要 + 链接原文档，原文档不动）；B 移动归档到 `docs/参考资料/` + 薄壳；C 重命名替代（破坏性，用户明确同意才用）；**D 仅 AGENTS.md 映射**（零破坏，用户拒绝 A 时的降级）。

目标布局：项目根 `AGENTS.md` + `docs/`（踏勘报告 / 蓝图 / 路线图 / 下一阶段工作 / 知识库 / 禁忌 / 任务卡/ / 施工日志/ / 踏勘历史/ / 参考资料/）。

**反例**：用 `mv` 不用 `git mv`；打包接受全部建议；自动改引用误伤代码字符串；删用户文件；自动 commit。

多队场景：只有主队可发起规范化和重新踏勘；规范化期间登记临界占用，别队暂停涉全局文档的工作。
