# 17 · 方向链反向递进

> 适用：每次接活、任务卡耗尽、焦点切换时的**方向探查机制**。
> 核心原则：**无方向不施工**。
>
> **无守护例外 v0.4.11**：用户明确启动无人值守并给定时长后，如果用户目标已完成但时间未到，包工头可以从既有蓝图 / 路线图自动下放下一步方向，标记"待用户复核"，并把重大取舍写入 `docs/技术债账本.md`。不得新造蓝图愿景，不得空跑，不得停下等用户。

## 一、方向链是什么

### 正向定义（自顶向下，定义项目方向）

```
蓝图（终局 / 概念 / 长期愿景）        ← 用未确定语气写
   ↓
路线图（按时序拆阶段，每阶段有目标和半径估算）
   ↓
下一阶段工作（本阶段焦点条目 · 主线/次线/不做）
   ↓
任务卡（具体可执行单位 · S/M/L）      ← 工人拿着干活
```

### 反向获取（自底向上消耗，包工头每次接活的探查方向）

```
任务卡耗尽         → 下一阶段工作取焦点条目 → 下放成任务卡          【自主】
下一阶段工作耗尽   → 路线图取新阶段方向 → 落盘到下一阶段工作          【常规：⚠️ 必须用户确认 / 无守护：自动下放待复核】
路线图耗尽         → 蓝图取新概念 → 规划落盘到路线图                 【常规：⚠️ 必须用户确认 / 无守护：自动下放待复核】
蓝图耗尽           → 冻结所有工作，进入用户对话                      【❄️ 冻结】
                     → 产出新蓝图条目 → 落盘 → 才能往下递进
```

## 二、确认门槛矩阵（**核心规则**）

| 跨层 | 是否需要用户确认 | 理由 | 落盘分级（呼应 `00` Step 4） |
|---|---|---|---|
| 任务卡 ← 下一阶段工作 | ❌ **自主** | 拆分已认可的焦点 = 战术决策 | L1 |
| 下一阶段工作 ← 路线图 | ✅ **必须用户确认** | 切换项目当前焦点 = 战略决策 | L2 |
| 路线图 ← 蓝图 | ✅ **必须用户确认** | 改变项目节奏 = 项目级决策 | L2 |
| 蓝图扩展 ← 用户对话 | ❄️ **必须用户对话产出** | 改变项目方向 = 方向级决策 | L2 |

**永久铁律**：跨层级取材时，**禁止包工头自行拍板上层方向**。包工头只能复述、建议、规划，不能替用户做"我猜你想要 X"。

**无守护覆盖条款**：在 `playbooks/16-无守护自动施工模式.md` 生效期间，且用户给定时间未到时，上表的"必须用户确认"改为"自动下放 + 待用户复核 + 技术债账本留痕"。包工头只能从已有蓝图/路线图/参考源取材，不得凭空创造新方向。

## 三、反向探查的两个时机

### 时机 A · 顺手扫（每次接活前 1 秒）

包工头在接活契约阶段顺手扫 4 个数：

| 层 | 顺手扫的指标 |
|---|---|
| 任务卡 | `docs/任务卡/` 下不带 `.完工` `.作废` 后缀的 .md 数（在工 + 计划） |
| 下一阶段工作 | `docs/下一阶段工作.md` 中未打勾的主线 / 次线条目数 |
| 路线图 | `docs/路线图.md` 中 🔄 进行中 + 🆕 计划中节点数 |
| 蓝图 | `docs/蓝图.md` 中 💭 仅探讨 状态条目数 |

**任一为 0** → 警觉，进入时机 B 正式探查。
**全 ≥ 1** → 走正常接活流程，但仍要在接活契约的"风险提示"段标注 `parent_focus = <下一阶段工作的哪条焦点>`。

### 时机 B · 正式探查（耗尽时 / 首次踏勘时 / 重新踏勘时）

进入完整 4 层探查流程，输出"📐 方向链快照"（用 `templates/工地踏勘报告模板.md` 六、段）。

## 四、4 层探查详细动作

### 第 1 层 · 任务卡

**读位置**：`docs/任务卡/`

**判定**：
- 数量 ≥ 1 且与本次接活方向匹配 → ✅ **可直接接活**
- 数量 = 0 或都不匹配 → 进入第 2 层

### 第 2 层 · 下一阶段工作

**读位置**：`docs/下一阶段工作.md`

**判定**：
- 主线 / 次线还有未完成项 → 找匹配焦点条目 → **下放成任务卡**（自主，无需用户确认）
- 全部完成 → 该层耗尽，进入第 3 层
- 文件不存在 → 进入第 3 层判断（看路线图）

**下放动作**：
1. 选一个焦点条目（多个匹配时由用户选）
2. 按 `playbooks/03-任务卡规范.md` 拆出 1~N 张任务卡
3. **每张任务卡的元信息必须有 `parent_focus` 字段**（v0.4 新增），格式：
   ```
   parent_focus: docs/下一阶段工作.md#主线-1-接入用户登录页JWT
   ```
4. 在下一阶段工作对应焦点条目下追加任务卡链接
5. 包工头继续干活

### 第 3 层 · 路线图

**读位置**：`docs/路线图.md`（或等价物，如本项目的"开发计划-v1.md"）

**判定**：
- 三、进行中节点 + 四、计划中节点至少一项可启动 → 选一个新阶段方向 → 常规模式 **必须用户确认** / 无守护模式自动下放待复核 → 落盘到下一阶段工作
- 全部 ✅ 已完成 / 没有可启动项 → 该层耗尽，进入第 4 层
- 文件不存在 → 进入第 4 层

**落盘动作**：
1. 给用户列**所有可启动候选**（不要替用户选），让用户拍板选哪个 / 拒绝全部 / 自定义新节点
2. 用户拍板后，把选中的路线图节点状态改为 `🔄 已展开 → docs/下一阶段工作.md#<阶段名>`（v0.4 新增已展开标）
3. 用 `templates/下一阶段工作模板.md` 覆盖式落盘 / 创建 `docs/下一阶段工作.md`
4. 给用户复述展开后的下一阶段工作（"主线 / 次线 / 不做" 三段是否合理）
5. 用户对下一阶段工作内容也确认后，才进入第 1/2 层下放任务卡

**无守护例外**：用户不在场且时间未到时，包工头选择"最高价值、依赖最少、可回滚"的路线图节点自动下放；在 `docs/下一阶段工作.md` 标记"无守护自动下放，待用户复核"，并把取舍写入 `docs/技术债账本.md`。

### 第 4 层 · 蓝图

**读位置**：`docs/蓝图.md`（或等价物）

**判定**：
- 二、近期蓝图条目中有 💭 仅探讨 状态的条目 → 选一个 → 常规模式 **必须用户确认** / 无守护模式自动下放待复核 → 规划落盘到路线图
- 全部 🆕 已升级 / ❌ 已放弃 → 蓝图耗尽 → 进入第 5 层冻结
- 文件不存在 → 进入"蓝图首次共建"流程（情形 A，见五）

**落盘动作**：
1. 给用户列所有 💭 仅探讨 状态的蓝图条目作为候选
2. 用户拍板后，把蓝图条目状态改为 `🆕 已升级 → docs/路线图.md#<节点名>`（沿用现有状态机）
3. 在 `docs/路线图.md` 第三 / 四节按位置插入新节点（用 `templates/路线图模板.md` 的节点格式）
4. 给用户复述展开后的路线图新节点
5. 用户确认后才进入第 3 层落盘下一阶段工作

**无守护例外**：用户不在场且时间未到时，包工头从已有 💭 蓝图条目中选择"最高价值、依赖最少、可回滚"的一项自动升级到路线图；在蓝图、路线图和技术债账本中标记"无守护自动下放，待用户复核"。如果蓝图为空或不存在，不进入冻结等待，而是退回参考项目、QA、测试、文档债补给。

### 第 5 层 · 冻结

**判定**：蓝图也耗尽（或不存在）

**冻结动作（4 处留痕）**：

1. **`AGENTS.md` 顶部插入醒目段**：
   ```markdown
   > ⛔ **项目方向冻结** · 自 YYYY-MM-DD · 等待蓝图层对话
   >
   > 当前蓝图、路线图、下一阶段工作、任务卡 4 层全部耗尽。
   > 包工头仅接 S 级修复活，M/L 级一律拒接。
   > 解除条件：与用户开"蓝图层对话"，产出新蓝图条目并落盘。
   ```

2. **当日施工日志记一笔**：
   ```
   [HH:MM] 方向链耗尽，包工头进入冻结状态。蓝图层无 💭 仅探讨 条目。
   ```

3. **包工头只能接 S 级零散修复活**（如打错字、补注释、明显的 lint 错），M/L 级一律拒接

4. **接到 M/L 级活时回复模板**：
   ```
   ⛔ 项目方向已冻结。
   当前 蓝图 / 路线图 / 下一阶段工作 / 任务卡 4 层均无可执行项。
   请先与我开"蓝图层对话"，把新概念落到 docs/蓝图.md。
   我们再往下递进。

   或者你确认这就是个 S 级零散活，那我可以接（请明确说"S 级"）。
   ```

**冻结状态的解除**：

1. 用户与包工头开"蓝图层对话"
2. 产出新蓝图条目（用未确定语气，按 `templates/蓝图条目模板.md` 的"二、近期蓝图条目"段格式）
3. 落盘到 `docs/蓝图.md`（这本身是 L2 操作，需用户确认落盘内容）
4. 把 `AGENTS.md` 顶部 ⛔ 冻结段移除（或归档到附录）
5. 然后正常进入第 4 层 → 第 3 层 → 第 2 层 → 第 1 层

**无守护例外**：无人值守期间不得因冻结状态空等。若蓝图层耗尽，包工头记录冻结事实到技术债账本，然后转向用户指定参考源、QA、测试、文档债、代码健康类安全卡，直到时间到。

## 五、4 种异常情况

### 情形 A · 项目从未有任何方向文档

**判定**：`docs/蓝图.md` / `docs/路线图.md` / `docs/下一阶段工作.md` 三个文件**全部不存在**

**性质**：比"耗尽"更严重 = 项目无定义

**动作**：
- 进入"**蓝图首次共建**"对话
- 让用户口述项目愿景 / 解决什么问题 / 给谁用 / 长期目标
- 用 `templates/蓝图条目模板.md` 落盘第一份蓝图（含"长期愿景"段 + 至少 1 个 💭 蓝图条目）
- 然后按正向链 蓝图 → 路线图 → 下一阶段工作 → 任务卡 一次性建起来（每跨层都需用户确认）
- 全程禁接任何 M/L 级活，仅可接 S 级文档骨架建立类工作

### 情形 B · 各层都有但散落分布

**判定**：四层中至少一层散落在多份文档（如 kekeling2.0 项目把"下一阶段"散落在 23 条业务链路 .9 小节）

**动作**：
- 在 `AGENTS.md` 加"当前进度索引"段统一索引（不强制让用户合并文件，尊重项目现实形态）
- 报告里标"⚠️ 焦点散落，已索引"
- 接活时 `parent_focus` 字段可以指向散落位置（如 `docs/科科灵2.0-核心业务链路梳理-v1.md#L05`）
- 可接 S/M 级活；L 级建议先等 AGENTS.md 索引完成

### 情形 C · 路线图存在但下一阶段不存在

**判定**：有 `docs/路线图.md` 但无 `docs/下一阶段工作.md`

**动作**：
- 直接进入第 3 层 → 第 2 层
- 包工头从路线图当前节点拆下一阶段工作骨架草稿
- 用户确认后落盘

### 情形 D · 任意层条目全部已完成

**判定**：某一层所有条目状态都是 `✅` 或类似"完成"标记

**动作**：
- 触发该层"已完结"告警
- 触发上一层探查
- 同时触发**重新踏勘**建议（详见 `playbooks/13-工地踏勘对齐.md`，强触发列表已加"阶段完结触发"）

## 七、设计原则

**为什么必须有这个机制**：
- AI 凭直觉接活 = 项目走样的最大风险源
- 没有"方向锚点"的活 = 无法回溯到任何业务价值
- 包工头是"对结果负责"，不是"对听话负责" —— 用户要做无方向的活，包工头有责任拒绝

**为什么常规模式跨层取材必须用户确认**：
- 任务卡拆分 = 战术决策（自主）
- 切换阶段焦点 = 战略决策（必须用户）
- 切换路线节奏 = 项目级决策（必须用户）
- 扩展蓝图概念 = 方向级决策（必须对话）

**为什么有"冻结"机制**：
- 防止包工头在无方向状态下"为了干活而干活"
- 让用户意识到"项目已经没有清晰下一步" 的事实
- 强迫用户做战略思考 / 项目复盘
- **冻结不是失败，是健康项目的自然节点**（如 MVP 完成后停下问"下一步往哪走"）

## 八、反例（工地禁忌候选）

- 任务卡耗尽时包工头自行从蓝图拍板新阶段方向（跨多层，绕过用户确认）
- 任务卡 `parent_focus` 字段空着（无方向锚点）
- 蓝图耗尽时继续接 M/L 级活
- 跨层取材后**不在源层级标"已展开"**（导致下次重复展开）
- 用户随口提了一个想法，包工头直接落到任务卡（绕过蓝图 → 路线图 → 下一阶段工作三层确认）
- 接活时不顺手扫 4 层数（时机 A 漏做）
- 下放任务卡时把多个 `parent_focus` 塞到一张卡（违反"一卡一焦点"）
- AGENTS.md 已写"⛔ 冻结"但仍接 M/L 级活（违反冻结纪律）
