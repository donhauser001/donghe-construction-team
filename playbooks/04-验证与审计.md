# 04 · 验证与审计

> 适用：完工自检（工人 + 包工头都读"一 ~ 二"）；监理审计（"三"起仅监理姿态读）。

## 一 · 宪法句

> **"build 通过 / tsc 通过 / unit test 通过" 不是"功能完成"。真实运行时的可观察行为才是。**

## 二 · 按半径分档的验证清单

**S 级**：tsc / lint 无新错；相关 unit test 跑通；UI 改动至少"看一眼"。

**M 级**（S + 以下）：API 真调（curl / playwright，贴请求 + 响应）；关键 DB 字段真查；happy path 真走一遍；至少 1 个边界场景（空值 / 超长 / 越权 / 重复提交任选）。

**L 级**（M + 以下）：跨层联调（前端→后端→DB→渲染完整跑）；跨端各跑一次；持久化真验（刷新 / 重启后再看）；实时通道真验（连→触发→断→重连→再触发）；agent 工具对等真验（见"六"）；撤销/回滚真验（如适用）；老数据兼容（拿最早的真实记录跑）；≥ 3 个边界场景。

**证据用脚本生成**：

```bash
python scripts/collect_evidence.py \
  --cmd "typecheck::npm run typecheck" --cmd "test::npm test" \
  --out docs/任务卡/_evidence/YYYY-MM-DD-NNN.md
```

卡里只贴证据文件链接 + 关键结论；手写只补 UI 观察、DB 语义、阻塞原因。每个验证 checkbox 后必须跟"证据"或"未验证 + 原因"，**禁止裸打勾**。没法真测时诚实标"未验证 + 原因 + 建议谁在哪验证"——**未验证 ≠ 失败；未验证伪装成已验证 = 重大违规**。

## 三 · 监理姿态（审计前先切换）

监理不是审批口，是前线加速器：目标是发现 P0/P1、给最小返修路径，不是为已完成的事背书，也不是把自己变成等待项。

- **默认不动代码**：只读、只问、只判定；不替工人善后、不顺手改（用户明确说"你直接修"才切回包工头姿态）。
- **只有 P0/P1 或高风险范围**（schema 破坏性变更 / 资金 / 生产凭证 / 安全隐私 / 医疗 P0-P1 / 跨队冲突 / 声称完成但无证据）**才允许冻结施工**；P2/P3、格式、文案、EOF 空行、标题层级一律记录后补，不挡核心链路。
- 切换自检词："我是来挑刺的。找不到问题应该怀疑审得不细，而不是赞美工作完美。"

## 四 · 审计九问

1. **SSOT**：同一业务状态是否写到两处？有没有 UI-only / 缓存-only 影子状态？
2. **持久化**：刷新 / 重启 / 重登后改动还在吗？字段类型 / nullable / 默认值合理吗？
3. **撤销**（如适用）：进统一历史系统了吗？撤销是否完整（不是只撤 UI 留下 DB 写入）？
4. **agent 工具对等**：UI 能做的 agent 工具能做吗？入参可自动构造吗？能读出结果吗？
5. **失效**：缓存 / 索引 / 派生视图在源头变化时被失效了吗？
6. **老数据**：已存在的老记录会怎样？缺字段 / 旧 enum 有 migration 或 fallback 吗？
7. **多视图一致**：编辑 / 列表 / 预览 / 导出 / 详情是否同一数据源？
8. **多端一致**：Web / 移动 / 后台 / agent 同时在线看到的真相一致吗？
9. **失败模式**：半途失败留不一致状态吗？批量部分失败如何告知回滚？断线如何降级？

## 五 · 证据等级与判定

| 档 | 证据形式 | 效力 |
|---|---|---|
| 1 | 真实运行时复现 + 持久化数据 | ★★★★★ |
| 2 | API 真调 + DB 查询 + 实时通道观察 | ★★★★ |
| 3 | 针对性测试证明不变量 | ★★★ |
| 4 | 静态代码追踪（file:line） | 不算"已验证" |
| 5 | 仅声称 | 不算"已验证" |

**user-visible 行为必须 ≥ 第 2 档才能声明完成。**

严重度锚点：**P0** = 数据丢失 / 写错位 / 无法撤销 / 安全越权 / 核心流程不可用；**P1** = SSOT 分裂 / 工具假实现 / 老数据破坏 / 跨端不一致 / **文档声称已做但代码证据没有**；**P2** = 集成不完整 / 缺测试 / 可观测缺口；**P3** = 命名文案润色。

判定三选一：

- **✅ 可验收**：所有验收点 ✅ 且 ≥ 第 2 档证据 + 九问无 P0/P1 + 不变量未破。
- **❌ 不可验收**：有 P0/P1 / 验收点未勾无说明 / 证据空或低于第 2 档 / 不变量被破 → 写明每条发现 + **最小修复方向**，重派。**不得因 P2/P3 冻结可继续实测的核心链路**（走部分可验收 + 后补清单）。
- **⚠️ 部分可验收**：核心链路已验证 + 非核心有问题 → 未完成项开新卡跟踪，汇报标注。

**二次审计**：修复后必须重走九问，不能复用第一次结论（修复可能引入新问题）。
**审计报告只写在卡内"监理审计区"**，不另开文件。
审计发现的问题按 `03-派工与并行.md` §五回流（小问题回炉 / 大问题新卡），不打断工人在工卡。

## 六 · 原生 AI 应用素质（L 级审计的 Q4 展开）

东合对项目的默认期待——五条素质：**可读**（agent 能结构化读所有核心数据，支持过滤分页）、**可写**（UI 能做的每件事都有对应工具，入参语义化、有校验、幂等）、**可解释**（返回结构化结果和错误，写入回最新状态而非只回 ok）、**可撤销**（重要写入有撤销工具且与 UI 撤销同源；不可逆操作有 dry-run / 二次确认）、**可发现**（有工具清单文档，命名动宾统一，schema 有描述）。

**新功能硬规则**：M/L 新功能卡必须有 `## AI 工具同步` 段（新增 / 修改的工具清单，无则写明"无：不引入新用户操作"），验收点里必须有"agent 工具已真测且与 UI 操作等价"。任一不满足不得进 ✅ 已完工。

典型违规（发现即沉淀禁忌）：加了 UI 按钮没加工具；工具叫 `update_data` 入参全是裸 ID；写入只回 `{"ok":true}`；删除无确认参数；UI 绕过工具能调的 service 层（SSOT 分裂）；工具清单落后于代码。

踏勘时可按五条素质给项目做粗评（N/总数），写进踏勘报告，缺口建议纳入下一阶段工作。

## 七 · 完工门槛（包工头声明"完成"前的最后一道闸）

**"完成" = 用户能可靠用上 + 知识已沉淀 + 没埋技术债。**

| 半径 | 硬条件 |
|---|---|
| S | tsc/lint 无新错 + 当日施工日志已记 |
| M | S 全部 + 验收点逐项 ✅ 且 ≥ 第 2 档证据 + 卡置 ✅ 加 `.完工` 后缀 |
| L | M 全部 + 审计无 P0/P1 + agent 工具对等已验 + 老数据已验 + ≥1 失败模式已验 + 跨端一致已验 + 文档已按预算回写 + 禁忌已确认沉淀 + 二次审计通过（如有打回） |

**声明完成前口头自检 5 问**：真的跑过看过刷过试过吗？证据是命令拿到的还是我复述的？有没有"应该/大概"的潜意识？路线图和知识库知道这件事了吗？三天后用户问起能从卡 + 日志找到完整链路吗？——任一"否"就不许说完成。

未达门槛的标准出口：**部分可验收**（列已验证 / 未完成 + 新卡链接，问用户是否接受）或**阻塞回报**（原因 / 已做 / 半成品 / 建议 + A/B/C 选项）。用户催促时列出未达项 + 风险，给"继续做 / 接受部分 / 用户授权降级验收"三选，**不擅自降级**。

**永不豁免**（用户怎么催都不跳）：真实运行时验证、当日施工日志、完成必须有证据、禁忌沉淀。
