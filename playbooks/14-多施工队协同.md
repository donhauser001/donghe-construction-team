# 14 · 多施工队协同

> 适用：用户在另一 IDE / 另一窗口又开了一队，需要和已在工的施工队共享项目根目录。
>
> **核心原则**：默认轻量（单队零开销），按需加重（多队启用规范）。
>
> v0.4.9：多队只在冲突、临界占用、方向/生产风险上打扰用户；日志、状态、证据链接默认各队自行写后汇报。

## 一、为什么需要这个 playbook

真实场景：
- 用户在 Cursor 主窗口跑主线，同时在 Codex 终端开第二队赶 hotfix
- 同一项目根目录，两队都在写文件
- 互相不知道对方在做什么

会发生的踩脚（按危险性）：

| # | 踩脚类型 | 后果 | 危险等级 |
|---|---|---|---|
| 1 | 同时改同一个代码文件 | git 冲突 / 互相覆盖 | 🔴 P0 |
| 2 | 同时改当日施工日志 | 文档覆盖 | 🔴 P0 |
| 3 | 任务卡编号冲突（都用 001） | 文件覆盖 | 🔴 P0 |
| 4 | 同时改路线图 / 蓝图 / 项目知识库 | 文档覆盖 | 🟡 P1 |
| 5 | 重复劳动 / 方向冲突 | 工作浪费 | 🟡 P1 |
| 6 | 完成度判断冲突 | 误判 | 🟡 P1 |

## 二、模式启用判定（自动检测 + 主动询问）

包工头每次开工时自动检测是否进入多队模式：

```
扫描：
- docs/施工队-临界占用.md 存在且非空 → 多队
- docs/施工日志/<今天>/ 是目录（不是文件） → 多队
- docs/任务卡/ 含别队前缀的卡（如 B001 / hotfix-001） → 多队
- AGENTS.md 中"已接入施工队"列表含本队以外 → 多队

满足任一 → 自动进入多队模式
未满足 → 进入"二次主动询问"
```

### 二次主动询问（防首次切换盲点 · per-session 一次）

**自动检测的盲点**：第一次切多队的瞬间（A 队还没建任何多队文件）→ B 队会误判为单队 → 撞 A 队。

**询问触发条件（同时满足才问，否则不问）**：

1. 自动检测结果 = 单队
2. **本 session 内首次接活**（同一 chat 里第二次起不再问，避免烦扰）
3. 项目接入时间 ≥ 24 小时（新接入的项目无须问）
4. 用户当前消息里没有任何明确单队 / 多队的语境信号（如"我现在唯一在跑这个" / "我另一处还有别的队"）

**询问话术**：

> 进入项目检测到当前为单队模式。本次首次接活，例行确认一下：
> **是否在其他 IDE / 窗口已开启了别的施工队？**
> - 否 / 只有这一队 → 我按单队模式开工（本 session 后续不再问）
> - 是，本队代号 X → 我按多队模式开工，详见 14
>
> 用户没回 → **等**，不要默认按单队往下走。

用户也可以**主动告知**："本次起进入多队模式，本队代号 X"。

**跳过场景**（任一即跳过，不必问）：
- 项目接入时间 < 24 小时（新项目，多队概率低）
- 本 session 已问过一次（per-session 缓存）
- 用户消息里已含明确单队信号（"现在只有我一个 chat 在跑" / "你是这个项目唯一的 agent"）

## 三、施工队代号

代号格式（用户给定为准，包工头不擅自命名）：

- 字母制：A 队 / B 队 / C 队
- 自由命名：主队 / 赶工队 / hotfix 队 / cursor-主 / codex-赶工 ...
- 两者皆可，但**同一项目内代号必须唯一**

代号写入：
- 当日施工日志开头
- 每张任务卡元信息头
- AGENTS.md 的"已接入施工队"列表
- `docs/施工队-临界占用.md` 占用记录

## 四、任务卡命名（多队后缀）

```
单队：docs/任务卡/2026-05-06-001-用户登录.md
多队：docs/任务卡/2026-05-06-A001-用户登录.md      # A 队的 001
       docs/任务卡/2026-05-06-B001-紧急修复.md      # B 队的 001
```

**编号空间分队**：
- A 队用 `A001 ~ A099`（队内单调递增）
- B 队用 `B001 ~ B099`
- C 队用 `C001 ~ C099`
- 永不冲突

进入多队模式后，已存在的"无前缀"卡视为**主队（默认 A 队）**的卡，新卡才用前缀。不要回头改老卡的命名。

## 五、施工日志按队分写

```
单队：docs/施工日志/2026-05-06.md（单文件）
多队：docs/施工日志/2026-05-06/
       ├── A队.md          # A 队的当日日志
       ├── B队.md          # B 队的当日日志
       └── 总览.md         # 各队完工情况汇总（包工头维护，可选）
```

每队**只写自己的日志**，避免覆盖。

**总览.md** 的维护由"主队"负责，简单列出"今天 A 队完成 X 张卡 / B 队完成 Y 张卡 / 阻塞 Z 张"。

**模式切换**：从单队切到多队的当日，把已有的 `2026-05-06.md` 移动为 `2026-05-06/A队.md`，并新建该目录。

## 六、临界占用清单（打招呼协议）

对**绝对不能并发改**的全局文件：
- `docs/路线图.md`
- `docs/蓝图.md`
- `docs/下一阶段工作.md`
- `docs/项目知识库.md`
- `docs/工地禁忌.md`
- `docs/工地踏勘报告.md`
- `AGENTS.md`
- `prisma/schema.prisma` 或同类核心 schema 文件
- 任何被工地禁忌标注为"高频变更冲突"的文件

包工头改之前必须**追加占用记录**到 `docs/施工队-临界占用.md`：

```markdown
| 占用时间 | 队 | 文件 | 预估时长 | 释放时间 |
|---|---|---|---|---|
| 2026-05-06 12:30 | A 队 | docs/路线图.md | ~5 分钟 | （改完后填） |
```

改完**立刻删除**这一行 + 在自己的当日施工日志记一笔。

**别队的处理**：
- 看到占用 → **等**（不要插队）
- 占用记录超过预估时长 2 倍仍未释放 → 在自己的施工日志记一笔"A 队占用超时，建议用户介入"，停下当前涉及该文件的工作

**`docs/施工队-临界占用.md` 文件首次创建**：当包工头第一次进入多队模式时，从 `templates/施工队-临界占用.md` 复制到 `docs/` 下作为初始骨架。

## 七、开工前的友邻扫描（多队模式必做）

包工头每次接活契约前**新增一步**：

```
1. 读 docs/施工队-临界占用.md（看实时占用）
2. 列出所有别队前缀的"在途任务卡"（无 .完工 .作废 后缀）
3. 提取别队卡的白名单 → 形成"友邻占用范围"
4. 对比本次需求范围 → 如有交集，立刻停下问用户：
   "你打算让 A 队和 B 队都改 X 文件，是否需要协调？"
5. 读 docs/施工日志/<今天>/<别队>.md，了解别队当日动作
```

## 八、新队加入流程

第二队首次接入项目时**不需要全量踏勘**，走"加入对齐"轻量流程：

### Step 1 · 读最新踏勘报告（信任主队）

读 `docs/工地踏勘报告.md`，作为本队世界观的入口。

### Step 2 · 扫多队当前状态

- 读 `docs/施工队-临界占用.md`
- 列所有未完工任务卡（含别队），按队分组
- 读 `docs/施工日志/<今天>/` 各队 + 总览（如有）
- 检查 AGENTS.md 的"已接入施工队"列表

### Step 3 · 输出《施工队加入对齐报告》

用 `templates/施工队加入对齐报告模板.md`，必填：

- 本队代号（用户给定）
- 本队预计的工作范围（文件域 / 子项目 / 模块）
- 别队当前在工范围
- 是否有冲突 / 风险
- 建议的分线方案

### Step 4 · 用户确认 → 写入 AGENTS.md

用户确认后，包工头：

- 在 AGENTS.md 的"已接入施工队"列表加入本队（代号 + 加入时间 + 范围）
- 在 `docs/施工日志/<今天>/<本队>.md` 创建当日日志（首条记"加入项目"）
- 进入正常接活流程

## 九、多队场景下文档对齐回写规则

每次完工后回写文档时，**多队模式额外要求**：

| 文档 | 多队规则 |
|---|---|
| 施工日志 | 写到本队当日日志；总览由主队同步 |
| 项目知识库 | 走临界占用打招呼协议 |
| 工地禁忌 | 走临界占用打招呼协议 |
| 路线图 | 走临界占用打招呼协议（一般只主队改） |
| 工地踏勘报告 | **只主队有权改** |
| AGENTS.md | 走临界占用打招呼协议 |
| 任务卡 | 自己的卡自己写，不动别队的卡 |

## 十、多队冲突的紧急处理

发现已经踩脚（git 冲突 / 文件被覆盖）：

1. **立刻停工**
2. 把当前任务卡置为 ⏸️ 阻塞，写明冲突情况
3. **不要自己 merge**（合并风险高）
4. 通知用户决策：
   - A 队回滚？
   - B 队回滚？
   - 协商 merge？
5. 用户决策后，把决策过程记入"工地禁忌"，避免重复

## 十一、退队 / 解散流程

某队完成阶段性工作后退出（用户告知或长时间无活动）：

- 在 AGENTS.md 的"已接入施工队"列表标注 ⏸️ 暂停 / ❌ 解散 + 时间
- 该队的在工任务卡需明确处置（继续 / 转交 / 作废）
- 该队的施工日志保留不删除

如果只剩一队 → 自动**回退到单队模式**，新卡可不带前缀。

## 十二、多队反例（工地禁忌候选）

- 单队模式下任务卡用了多队后缀 → 命名混乱
- 多队模式下不写当日施工日志 → 别队看不到你做了什么
- 改全局文档不打招呼 → 互相覆盖
- 自己合并 git 冲突而不通知用户
- 假装看不见别队的在工卡，硬上同样文件
- 占用临界文件后忘了释放（占着茅坑不出来）
- 别队代号与本队相同（命名冲突）
- 同一时间两队都在做"重新踏勘"
