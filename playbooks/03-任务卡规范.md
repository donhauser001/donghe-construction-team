# 03 · 任务卡规范

> 适用：M / L 级工作。S 级跳过任务卡，但仍须当日施工日志留痕。v0.4.9 起任务卡默认走"薄卡 + 机器证据"：写边界和验收，不写施工论文。

## 一、文件命名规则

### 单队模式

```
docs/任务卡/YYYY-MM-DD-NNN-任务名称[.状态].md
```

- `YYYY-MM-DD`：包工头签发日期
- `NNN`：当日单调递增三位编号，从 `001` 开始，跨日清零
- `任务名称`：6~16 字动宾短语，避免笼统词
- `.状态`：可选后缀，仅 `完工` / `阻塞` / `作废` 三种；默认（在途）无后缀

### 多队模式（详见 `playbooks/14-多施工队协同.md`）

```
docs/任务卡/YYYY-MM-DD-{队前缀}NNN-任务名称[.状态].md
```

- `{队前缀}`：单字母（如 `A` / `B` / `C`）或自由代号（如 `hotfix`）
- 编号空间分队，永不冲突

**示例**：

| 模式 | 状态 | 文件名 |
|---|---|---|
| 单队 · 在途 | 🔨 | `2026-05-06-001-接入用户登录页JWT.md` |
| 单队 · 完工 | ✅ | `2026-05-06-001-接入用户登录页JWT.完工.md` |
| 单队 · 阻塞 | ⏸️ | `2026-05-06-002-修复订单详情价格计算.阻塞.md` |
| 多队 · A 队 | 🔨 | `2026-05-06-A001-接入用户登录页JWT.md` |
| 多队 · B 队 | ⏸️ | `2026-05-06-B001-紧急修复支付回调.阻塞.md` |
| 多队 · 自由代号 | 🔨 | `2026-05-06-hotfix001-紧急修复支付回调.md` |

**重要**：文件名后缀仅为视觉标识；**权威状态以文件内 `## 当前状态` 字段为准**。包工头每次状态变更后既改文件名又改文件内字段。

## 二、必填段（用 `templates/任务卡模板.md`）

每张任务卡必须包含以下段（缺一不可），但内容要短，详细输出优先放到机器证据文件：

1. **元信息头**：编号、签发时间、半径、签发人（包工头）、**派给**（v0.4.3 强制 · 详见"二·2"）、**调用模型**（v0.4.3 强制 · 详见"二·2"）、**身份-模型匹配理由**（v0.4.3 强制 · 详见"二·2"）、**所属施工队**（多队模式）、**涉及子项目**（monorepo）、**`parent_focus`**（v0.4 必填，详见"二·1"）
2. **当前状态**（六态之一） + **状态变更日志**（追加式表格）
3. **上下文**：本卡在路线图 / 下一阶段工作的位置；关联的施工日志日期
4. **改动范围**：白名单（允许动的文件 / 目录）+ 黑名单（禁止动的文件 / 目录）+ **可读范围说明**（详见"四"）
5. **不变量**：必须保持的 SSOT、接口契约、数据形状、安全边界
6. **施工要点**：具体要做什么（只写 3-7 条关键动作）
7. **证据命令**：建议由 `scripts/collect_evidence.py` 执行的验证命令清单
8. **验收点**：必须满足的可测条件清单（带 checkbox，验收点必须能被证据命令或真实操作支撑）
9. **AI 工具同步**：本次改动需要新增 / 修改的 agent 工具清单（若无明确写"无"）
10. **工人回报区**（工人填写）：状态摘要 / 证据文件链接 / 阻塞原因 / 盲区 / 风险 / **本次读过的文件清单**
11. **监理审计区**（监理填写）：结论 + 证据链接 + P0/P1/P2 发现；P3 可进入后补队列

### 任务卡文档预算（v0.4.9）

| 半径 | 任务卡预算 |
|---|---|
| M | 正文建议 ≤ 80 行；超过必须写"为何不能拆小" |
| L | 单卡建议 ≤ 120 行；多卡用战情板串联，不把所有细节塞进一张卡 |
| 8-worker | 卡内只写写集、验收、证据命令、风险；战情板只放状态和证据链接 |

任务卡超过预算且没有跑通产品链路时，包工头必须先停写文档，回到施工 / 验证。

## 二 · 1 · `parent_focus` 字段（v0.4 强制 · 八大宪法第 8 条）

每张任务卡的元信息头必须有 `parent_focus` 字段，反向追溯到下一阶段工作的某个焦点条目。

格式：

```yaml
parent_focus: docs/下一阶段工作.md#focus-1
```

> `focus-1` 是下一阶段工作模板里给每条主线 / 次线打的 HTML 注释 `<!-- focus_id: focus-1 -->` 锚点。

**散落场景**（v0.4 情形 E）：如焦点散落在多份业务/链路文档（如本项目散在 23 条 .9 小节），`parent_focus` 可指向散落位置：

```yaml
parent_focus: docs/科科灵2.0-核心业务链路梳理-v1.md#L05-用户下单
```

**冻结场景**（v0.4 情形 B）：冻结期只允许接 S 级豁免活（详见 `playbooks/02` 四·1），任务卡可以省略（直接走施工日志），但若仍创建任务卡，`parent_focus` 字段值为：

```yaml
parent_focus: 冻结期 S 级豁免（无方向锚点）
```

**反向追溯检查**（监理姿态必查）：

- 任务卡 `parent_focus` 指向的下一阶段工作焦点条目，必须真实存在
- 焦点条目下的"关联任务卡"列表必须包含本卡的反向链接
- 焦点条目的"下放状态"必须切换为 `🔄 已下放`

**反例**：
- ❌ `parent_focus` 字段缺失或为 `TODO` / `xxx`
- ❌ `parent_focus` 指向的锚点在下一阶段工作里不存在
- ❌ 一张卡同时指向多个 `parent_focus`（违反"一卡一焦点"颗粒度铁律）

## 二 · 2 · 身份-模型 三元绑定（v0.4.3 强制 · 八大宪法配套铁律）

每张 M / L 任务卡的元信息头**强制三字段**（缺一即任务卡设计缺陷 · 不可派工）：

```yaml
派给:    工人#1                          # 三元字段 1
调用模型: composer-1.5                    # 三元字段 2 · 必须是具体 slug
身份-模型匹配理由: 文件白名单清晰 · 按详细 SOP 替换数据源 · composer-1.5 够用    # 三元字段 3
```

### 2·2·1 三字段填写规则

| 字段 | 合法值 | 反例（不可填）|
|---|---|---|
| **派给** | `工人#1` / `工人#2` / `工人#3` / `包工头亲自` / `包工头切监理姿态` | ❌ `工人 1` ❌ `worker A` ❌ `claude` ❌ 留空 |
| **调用模型** | 具体 slug（如 `composer-1.5` / `claude-4.6-sonnet-medium-thinking` / `gpt-5.5-medium` / `claude-4.6-opus-high-thinking`）| ❌ `便宜模型` ❌ `中档` ❌ `自动选` ❌ 留空 |
| **身份-模型匹配理由** | 一句话陈述（"按详细 SOP 替换 / API 契约清晰 / schema 是 SSOT 必须包工头亲自" 等）| ❌ `够用` ❌ `合适` ❌ `按惯例` ❌ 留空 |

### 2·2·2 模型档与任务半径不是同一回事（关键区分）

- **半径**（S / M / L）= 改动**影响范围**（单文件 / 单模块 / 跨层）
- **模型档**（composer / sonnet / opus）= 任务**推理难度**

→ M 半径 + 简单清单填空 = `composer-1.5`（半径 vs 难度二维 · 见 `playbooks/05-并行编排.md` §四·5·2）

### 2·2·3 双闸门（v0.4.3 核心机制）

**派工闸**（包工头派 Task 前自检 · 详见 `playbooks/05-并行编排.md` §四·5·3）：

- [ ] 卡上"派给"已填具体值（不是占位 / TODO）
- [ ] 卡上"调用模型"已填具体 slug（不是"自动" / "便宜"等抽象描述）
- [ ] 卡上"身份-模型匹配理由"已填一句话（不是"够用"这种水词）
- [ ] Task 调用里的 `model=` 与卡上"调用模型"完全一致
- [ ] Task 调用里 prompt 里**贴出**卡上三字段让工人核对

**开工闸**（工人接卡首动作 · 详见 `playbooks/04-施工工人协议.md` §一·1）：

- [ ] 工人读卡顶元信息
- [ ] 自检自己实际身份（"我是工人#几"）↔ 卡上"派给"是否一致
- [ ] 自检自己实际模型（agent 本体的当前 slug）↔ 卡上"调用模型"是否一致
- [ ] 任一不一致 → **立即罢工** · 在卡顶 `## 当前状态` 上方加一段"工人退卡"备注 · 状态保持 📋 不进入 🔨

### 2·2·4 包工头亲自 vs 派工的硬边界（红线 · 违反 = OPS-003）

- ❌ **任务卡半径 ≥ M + 工作量 ≥ 30 行业务代码**（不含 schema / 不含 smoke）→ **必派工** · 不许"派给：包工头亲自"
- ❌ 任务卡缺三元字段就派 Task → 派工闸违规
- ❌ 任务卡填了三元字段但 Task `model=` 与卡不一致 → 派工闸违规
- ❌ 工人接卡不核对身份-模型 → 开工闸违规

**包工头亲自的合法场景**（必须能在"匹配理由"里说清）：

| 场景 | 示例 | 必填理由 |
|---|---|---|
| schema 是 SSOT | Prisma schema 改动 / 数据库迁移 | "schema 是 SSOT · L 级铁律 · 包工头亲自" |
| smoke / 监理审计 | 端到端真实运行验证 | "监理姿态 · 不动代码 · 出可验收结论" |
| 治理类文档 | 工地禁忌沉淀 / skill 升级 | "治理类 · 包工头核心职能 · 不外包" |
| S 级单文件单函数 | 改 1 行配置 / 加 1 个常量 | "S 级 · 不写任务卡 · 当日施工日志留痕" |

**反例**（违反 OPS-003 的真实事件）：

- 2026-05-06 22:00：包工头亲手写 `miniapp/utils/sentry.js` 303 行 ES5 + cp 复制到 companion + smoke test → **跨度 M 级 · 应派 sonnet-medium worker · 包工头自留监理姿态**

## 三、颗粒度判定（软参考 vs 硬上限分层）

### 软参考（建议遵守，超出可解释）

| 维度 | 软参考 |
|---|---|
| 白名单文件数 | ≤ 5 |
| 施工要点条数 | ≤ 7 |
| 验收点条数 | ≤ 5 |
| 证据命令条数 | ≤ 5 |
| 预计施工时长 | ≤ 3 小时 |

超出软参考时，包工头只写一行"超阈值理由"，不得扩写成长段解释。

### 硬上限（不可破，破必拆）

| 硬上限 | 理由 |
|---|---|
| **单卡涉及子项目数 = 1** | 跨子项目 = 跨技术栈 / 跨上下文，必拆 |
| **单卡不能同时改 schema + 业务逻辑** | 数据迁移 + 业务变更必须分两卡，避免半失败 |
| **单卡不能同时涉及 P0/P1 安全边界 + 普通业务** | 安全审计粒度需要独立 |
| **单卡不能含 > 1 个核心 SSOT 模型变更** | 一卡只动一个 SSOT |

破硬上限的卡 = 包工头设计错误，必须拆开重派。

### 拆卡的常见切法

1. **按文件域**：前端 / 后端 / DB 各一卡
2. **按依赖层**：先动数据层 → 再动服务层 → 再动接口层（依赖在前的卡先做）
3. **按子项目**：A 子项目改动 / B 子项目改动 各一卡（**首选**，最干净）
4. **按改动类型**：迁移 / 新增 / 重构 各一卡

## 四、白名单与读 / 写边界

任务卡的"改动范围"实际上是**写白名单**，不是"读白名单"。

### 写边界（严格白名单）

工人**只能写**白名单内的文件。越界写 → ⏸️ 阻塞回报包工头。

### 读边界（按需放行，但要节制）

工人施工时**经常需要读相邻文件**才能正确实施（读类型定义、读父类、读相邻组件、读测试）。

**规则**：
- ✅ 允许读项目内任何文件（不需要事先声明）
- ✅ 但完工时必须在"工人回报区"列出**本次读过的关键文件清单**
- ⚠️ 大量读非白名单文件（>10 个）→ 提示包工头"任务卡上下文不足，建议把这些文件加入'上下文'段或拆卡"
- ❌ 禁止读 `.env` / 证书 / 任何明显敏感的文件

读清单的作用：
- 让监理评估盲区（没读到的部分就是盲区）
- 让包工头复盘任务卡的"上下文"段是否设计充分

## 五、永久禁词（任务卡内禁止出现）

任务卡的"施工要点"和"验收点"中不允许出现：

- `暂时` `临时` `先这样` `TODO 后续优化` `简单起见` `后面再说`
- `可能` `也许` `大概` `差不多`
- `修一下` `调一下` `优化下`（动作不明确）

出现即视为任务卡设计缺陷，包工头要重写。

## 六、状态机变更规则

```
📋 已派发 → 🔨 施工中：工人接卡时
🔨 施工中 → 🔍 待审计：工人完工自检并提交
🔨 施工中 → ⏸️ 阻塞：工人遇到任务卡矛盾 / 缺失 / 必须硬编码
⏸️ 阻塞 → 🔨 施工中：包工头处理后重派
🔍 待审计 → ✅ 已完工：监理验收通过
🔍 待审计 → ❌ 不可验收 → 🔨 施工中：监理打回
任意 → 🚫 作废：明确放弃此方案
```

**每次状态变更**必须在卡内"状态变更日志"表格追加一行：时间 / 操作者 / 新状态 / 简述。

**文件名后缀**仅在"完工 / 阻塞 / 作废"三态时变更：

- 进入 ✅ 已完工 → 加 `.完工`
- 进入 ⏸️ 阻塞 → 加 `.阻塞`（解除阻塞回到 🔨 时去掉 `.阻塞`）
- 进入 🚫 作废 → 加 `.作废`

## 七、关联与索引

- 任务卡里如果引用了其他文档，用相对路径：`../蓝图.md`、`../工地踏勘报告.md`、`./2026-05-06-001-xxx.md`
- 任务卡完工后**不转移目录**，留在原地仅改文件名后缀
- 当日施工日志会汇总当日所有任务卡的状态变化（链接到具体卡）
- 多队模式下：自己的卡只能自己写，**不动别队的卡**

## 八、生成任务卡的最小工作流

```
1. 复制 templates/任务卡模板.md → docs/任务卡/<命名规则>.md
2. 填元信息头（编号/签发时间/半径/派给谁/施工队/子项目）
3. 填上下文 / 改动范围（写白名单）/ 不变量 / 施工要点 / 验收点 / AI 工具同步
4. 当前状态置为 📋 已派发，状态变更日志加第一行
5. 派工：通过 Task tool 启动子 agent，详见 playbooks/05-并行编排.md
6. 工人接卡 → 改卡内当前状态为 🔨 施工中 → 开始施工
```

**包工头不要为了"快"省略任务卡**。M / L 级跳过任务卡 = 工地禁忌。
