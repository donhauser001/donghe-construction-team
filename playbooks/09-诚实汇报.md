# 09 · 诚实汇报

> 适用：每次给用户的最终回应、阶段性进度同步、阻塞通报。

## 一、汇报的固定结构（按半径分档）

### S 级 · 极简版（1~3 行）

```
✅ 已完成：<一句描述>
- 改了：<文件 1~2 个>
- 验证：<一行证据，如 tsc/lint pass · 简单运行时眼测>
- 当日施工日志已记一笔
```

### M 级 · 标准版（10 行内）

```
## 本轮结论
（可验收 / 不可验收 / 部分可验收 / 阻塞）

## 改了什么
- 文件清单（≤5）

## 已验证（带证据等级）
- ★★★★ xxx：（粘 1~2 行真实运行时片段）

## 未验证 + 原因
- xxx

## 涉及任务卡
- docs/任务卡/YYYY-MM-DD-NNN-xxx.完工.md
```

### L 级 · 完整版（无上限）

```
## 本轮结论
（可验收 / 不可验收 / 部分可验收 / 阻塞 / 未完成）

## 改了什么
（完整清单：文件 / 模块 / 接口 / 文档）

## 已验证什么（带证据等级）
- ★★★★★ xxx：（粘真实运行时输出片段）
- ★★★★ xxx：（粘 API + DB 证据）
- ★★★ xxx：（测试 PASS 输出）

## 未验证什么 + 原因
- xxx：环境受限 / 缺测试数据 / 超出本卡范围

## 已知盲区
- "我没读过 X，对 Y 行为不确定"
- "本次仅静态追踪 Z"

## 剩余风险
- xxx（建议下一步动作）

## 涉及的任务卡 / 施工日志
- docs/任务卡/2026-05-06-001-xxx.完工.md
- docs/施工日志/2026-05-06.md（多队场景：docs/施工日志/2026-05-06/<本队>.md）

## 文档同步
- 路线图：xxx 节点已标 ✅
- 项目知识库：追加了 xxx 条
- 工地禁忌：追加了 xxx 条
- 工地踏勘报告：是否需要重新踏勘？（按 13 评估）

## 元信息（L 级 / 里程碑必填）
- 本轮用了 N 个工人 / 总耗时 X / 工地禁忌新增 Y 条 / 项目知识库新增 Z 条 / 推进了哪个路线图节点
```

**口径**：S 不强写 M 的结构，L 不准用 S 的结构敷衍。半径标记的不可向下降。

---

## 二、标准句式 vs 禁说词

### ✅ 推荐使用

- "已验证：浏览器实操 + DB 查询，结果一致"
- "未验证：本机环境无 staging 数据，待你在 staging 跑一次"
- "我没读过 `<path>`，对 `<行为>` 不确定"
- "本次仅静态追踪，未真实运行验证"
- "build 通过 + unit test 通过，但浏览器未实测"
- "在 `<边界>` 下我没构造测试数据，行为未知"
- "本卡部分可验收：核心可用，xxx 待办（已开新卡）"

### ❌ 禁说

- ~~"搞定了"~~
- ~~"完成"~~（除非真的所有门槛都过）
- ~~"没问题"~~
- ~~"应该可以"~~
- ~~"大概率 OK"~~
- ~~"应该没问题"~~
- ~~"我跑了一下，没事"~~（无证据）
- ~~"通过了所有测试"~~（如果没说哪些测试、跑没跑、什么命令）

## 三、长任务的进度同步节奏

### M 级（单卡）
- 派出工人时：一句话告知"已派出工人 #N，预计 X 时间内回收"
- 收回时：完整汇报

### L 级（多卡 + 节点交付）
- 每完成一张卡：1~2 句进度
- 每完成 50% 卡：阶段性小汇报（已完 / 在做 / 风险）
- 全部完成 + 文档对齐 + 监理审完：完整汇报

### 阻塞时（任何时候）
- 立即停下来发汇报，不等节点
- 用 `playbooks/08-完工验收门槛.md` 的"阻塞回报"模板

## 四、错误 / 失误的汇报

如果在工作中发现自己之前的判断错了 / 之前的代码有问题 / 之前的承诺不能兑现：

**第一句必须直接承认错误**，不绕弯：

```
✅ "上一轮我说 X 已验证，刚才发现实际没验证 Y 场景，结果出错了。"
✅ "我刚才漏看了 `<file>`，导致 `<bug>`。"
✅ "我之前说 30 分钟能做完，实际遇到 X 阻塞，需要 Y 时间或换方案。"

❌ "可能上次有点小问题"
❌ "稍微调整一下"
❌ "情况有些变化"
```

承认错误是诚信的体现，不是失职。

---

## 五、与用户预期的对齐

汇报里可以问"是否接受当前状态"，但**不要把判断权完全推给用户**。包工头要主动给出建议：

```
我推荐：
  - A. xxx（成本/收益/风险）
  - B. xxx（成本/收益/风险）

我倾向 A，原因：xxx。等你拍板。
```

---

## 六、多队场景下的汇报增项

多队模式下，每次汇报额外附：

- **本队代号**（如 A 队 / hotfix 队）
- **本轮临界占用记录**（占用了哪些全局文档 / 何时释放）
- **对友邻队的提醒**（如"我修改了 schema，B 队需要重新生成 Prisma client"）
- **建议主队是否重新踏勘**（仅别队提建议，主队拍板）

---

## 七、汇报反例（工地禁忌）

下面都是会被记入工地禁忌的反例：

- 汇报里只说"完成了"不贴证据
- 汇报里说"测试通过"但没说跑的什么测试
- 汇报里没列出"未验证"项
- 工人发现的问题被包工头吞掉没汇报给用户
- 阻塞汇报但不说阻塞原因
- 长任务里中途 0 进度同步
- 出错时第一句不是承认错误而是解释
