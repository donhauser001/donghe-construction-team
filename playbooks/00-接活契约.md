# 00 · 接活契约

> 适用：每次接到新需求 / 新任务时的第一个动作。

## 零、优先识别"无守护触发词"（最高优先）

在做任何接活动作前，**先扫一眼用户原话**，识别以下任一类表达：

- "我要走了 / 出门了 / 不在 / 离开 / 离场 / 下班"
- "我要睡了 / 睡觉了 / 上床了"
- "你自己干 / 自动跑 / 无人值守 / 无守护 / 自动施工 / 自己看着办"
- "我去 X，X 小时后回 / 你先做着 / 你接着干"

**命中任一 → 立刻跳到 `playbooks/16-无守护自动施工模式.md` 的 Step 0**，**不要走本 playbook 的常规接活流程**。

理由：无守护模式的启动有专属 4 个必填确认（时长 / 范围 / 自动 commit-push 策略 / 阻塞策略）+ 时间契约 + 队列补给 + 临时分支推送状态机，**和常规接活契约不兼容**。如果走错流程会绕远路、可能漏护栏。

### 触发词识别精度（防误判）

自然语言识别有误差。下列情况**不应该**触发跳 16：

- "我刚走了一趟回来，给我做 X"（"我走了"是过去时）
- "上次睡觉时我想到 ..."（陈述往事）
- "你自己去看一下文档"（不是要自动施工，是要你查阅）
- "无人值守这个功能怎么实现"（在讨论实现方案，不是要进入该模式）

**包工头判断不确定时**，**不要直接跳 16**，而是**先复述一下**：

> 我注意到你提到"我要去 X"，请确认：
> - 是要现在进入"无守护自动施工模式"（我按用户给定时长持续施工，到点或熔断后停下）吗？
> - 还是只是顺口提了一下，本次还是常规接活？

用户明确确认后再跳 16；用户否认或忽略 → 走常规接活流程。

未命中触发词 → 继续下面的常规接活流程。

## 一、识别项目接入状态（三种情形）

按顺序判定：

```
1. 项目根 AGENTS.md 不存在，或不含 <!-- donghe-construction-team: initialized --> 标识
   → 情形 A · 首次进入   → 走"二、首次进入子流程"

2. AGENTS.md 含上述标识，但 docs/施工队-临界占用.md 存在或别队当日施工日志存在
   → 情形 B · 多队加入   → 走"三、多队加入对齐流程"（详见 playbooks/14-多施工队协同.md）

3. AGENTS.md 含上述标识，无别队痕迹
   → 情形 C · 单队常规接活   → 跳到"四、接活契约六栏"
```

## 二、首次进入子流程（情形 A）

> 关键原则：**改用户文件之前先告知**。**Step 5 之前不写任何项目文件**（除了纯读）。

### Step 1 · 扫描

- 列出项目根目录与 `docs/`、`backend/`、`frontend/`、`miniapp*/`、`admin*/` 等子项目
- Glob 所有 `*.md` 文档（README、规划、契约、日志类）
- 检查 `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` 等清单识别技术栈
- `git log --oneline -n 30` 看近期提交脉络
- `git status` 看工作区是否大量 untracked（可能需要工地清账）
- **项目根资产顺手扫一眼**：图片 / 压缩包 / 设计稿 / 临时下载文件等"非代码非文档"资产（`find . -maxdepth 1 -type f ! -name '.*'`），后面踏勘报告里要单独列

#### Step 1 · 中文文件名 / 路径陷阱

`git status --short` **默认会把含中文/空格的路径转成 `\xxx\xxx` 反斜杠转义**，会让你看不清真实路径，也容易误判文件不存在：

- 扫描含中文路径项目时**必须**用 `git -c core.quotepath=false status --short`
- 同理 `git -c core.quotepath=false ls-files`、`git -c core.quotepath=false diff --name-only`
- 如果跑过 `find` / `ls` 但显示的目录数量比 `git status` 列出的少，通常是因为这些目录已经被 `D` 删除（物理已删但未 commit），不是没扫到

#### Step 1 · 工作区清洁度判定（影响 Step 4 的 Q 列表）

跑完 `git status --short` 后**统计未追踪+已修改文件总数 N**：

| N | 等级 | 后续动作 |
|---|---|---|
| ≤ 5 | 🟢 干净 | 不必特别处理，Step 4 只在风险段顺手提一句 |
| 6 ~ 29 | 🟡 偏脏 | Step 4 报告"风险与建议"段需明确列出未追踪关键目录，但不必上 Q 列表 |
| ≥ 30 | 🔴 严重不洁 | **强制写入 Step 4 的 Q 列表**：让用户决定 commit 策略（A 现在分批 commit / B 起新分支分批 commit / C 暂不处理）。**包工头本轮不擅自 `git add` 任何东西** |

判定原则（**关键**）：

- **永远只摆事实，不擅自处置**。在用户对 Q 列表回复之前，**禁止运行 `git add` / `git commit` / `git stash` / `git restore` 等任何会改变工作区状态的命令**
- 如果 `≥30` 同时还含 **完整子项目整个 untracked**（如 `backend/` 整目录），这是更高级别的项目纪律风险，必须在 Step 4 风险段标 🔴 极高严重度

### Step 2 · 通读已存在的项目文档（不写）

- **必读**：所有 `docs/*.md`（按文件名优先级：含"蓝图""规划""路线图""契约""规范"等关键词的优先）
- **必读**：项目根 `README.md`、各子项目 `README.md`
- 大文件（>20KB）先读前 200 行了解大纲，按需深读
- 收集：领域术语、命名约定、SSOT、安全/合规约束、AI 工具能力

### Step 3 · 等价物启发式识别

按文件名 / 内容启发式识别"用户已有文档 → 东合标准角色"映射：

| 用户已有文档命名特征 | 候选东合角色 |
|---|---|
| 含"蓝图" / "建设方案" / "总体设计" | 蓝图 |
| 含"路线图" / "开发计划" / "里程碑" / "roadmap" | 路线图 |
| 含"PRD" / "功能规划" / "产品规划" | PRD（不在东合标准角色内，作为补充资料） |
| 含"代码规范" / "约定" / "convention" | 项目知识库（部分注入） |
| 含"API 契约" / "接口契约" / "contract" | 接口契约（不在东合标准角色内，作为补充资料） |
| 含"业务流程" / "链路梳理" | 业务流程图（补充资料） |
| 含"任务" / "TODO" / "待办" | 在途任务（建议转化为任务卡） |

输出**等价物映射建议**——但不要默认正确，让用户最终确认。

### Step 4 · 落盘《工地踏勘报告》并在聊天复述要点

用 `templates/工地踏勘报告模板.md` 的"首次踏勘"段写报告，至少包含：

- 项目快照（项目名 / 定位 / 技术栈 / 子项目分布 / git 状态 / **工作区清洁度** / **项目根资产清单**）
- 已存在文档清单（含字节数 + **是否被代码 SSOT 取代** + 等价物映射建议）
- **缺失的骨架清单**（要新建什么 + 不会动什么）
- 已识别的项目世界观（不变量 / 命名约定 / 安全约束 / SSOT）
- 原生 AI 应用素质粗评（可选）
- 当前阶段判断（**找不到时按"焦点缺失告警"处理，见下文**）
- **是否建议项目文档规范化**（如果文档布局明显不规范，详见 `playbooks/15-项目文档规范化.md`）
- 待用户拍板的关键决策（按本次实际场景列）
- **如果 Step 1 判定工作区"严重不洁"（≥30 文件未追踪/已修改）**，必须在 Q 列表中**第一条**就问 git commit 策略（见 Step 1 工作区清洁度判定）

**这一步的动作流（铁律）**：

1. **直接落盘**到 `docs/工地踏勘报告.md`（旧版若有，先 `git mv` 或 `cp` 归档到 `docs/踏勘历史/YYYY-MM-DD-<触发原因>.md`）
2. **同时在聊天界面完整复述要点 + Q 列表**（用户能直接在聊天回应，不必打开文件再读一遍）
3. 用户对 Q 列表逐项 / 一句话回应后，**回头做第二轮覆盖式重写**：去掉过时的待拍板段，把决策结果落到正文 + 触发 Step 5

**为什么踏勘报告本身默认直接落盘**：
- 报告是**独立中性产物**，不动其他文件、不动代码
- 不落盘 = 下次接活找不到上下文，违背 skill"工地档案必须留存"的核心精神
- 用户不满意可以让我覆盖式重写，零成本
- 落盘的"工地档案"是包工头交接、多队协同、监理审计的入口

#### Step 4 · 落盘的分级（替代旧版"硬约束铁律"）

落盘动作按"是否改变项目入口/工作流/工作区状态"分三级：

| 等级 | 动作 | 是否需要用户显式确认 | 例子 |
|---|---|---|---|
| **L1** | 踏勘报告本身 | ❌ **默认直接落** | 落盘 `docs/工地踏勘报告.md` / 把旧版归档到 `docs/踏勘历史/` |
| **L2** | 标准骨架 / 改文档结构 | ✅ 必须显式确认 | 新建 `AGENTS.md` 入口手册、`docs/任务卡/` `docs/施工日志/` `docs/工地禁忌.md` 骨架、薄壳指引、归档/重命名旧文档 |
| **L3** | git 写操作 | ✅ 必须显式确认 | `git add` / `git commit` / `git stash` / `git restore` / `git mv` / `git switch -c` |

L2 / L3 的"显式确认信号"口径：
- "落盘骨架" / "建" / "go" / "Q5=A" / "全要" / "按建议来" → **算授权**
- "嗯" / "好" / "看起来不错" → **不算授权**，必须反问一次："要我现在 X 吗？"
- 用户**默不作声 / 接着问别的问题** → **不算授权**，按兵不动

反例（违反 L2/L3）：
- ❌ "我先把 `AGENTS.md` 骨架建起来，方便你 review"（L2，必须等显式确认）
- ❌ "我先 `git add docs/` 把新报告加到暂存区"（L3，必须等显式确认）

正例：
- ✅ "我把踏勘报告落到 `docs/工地踏勘报告.md` 了，要点如下：... Q 列表如下：..."（L1 直接落 + 聊天复述）
- ✅ "Q5 你回复 'Q5=A' 之后我再建骨架"（L2 等确认）
- ✅ "Q2 你回复 'B' 之后我再起新分支分批 commit"（L3 等确认）

#### Step 4 · 方向链反向探查（**v0.4 重写：原"阶段焦点告警 A/B/C/D"已升级为完整四层探查**）

按 `playbooks/17-方向链反向递进.md` 的方法，**完整探查 4 层方向**并输出"📐 方向链快照"段（落入 `templates/工地踏勘报告模板.md` 六、）：

| 层 | 文件 | 探查的指标 |
|---|---|---|
| 1 任务卡 | `docs/任务卡/` | 在工 + 计划中数（不带 `.完工/.作废` 后缀的卡数） |
| 2 下一阶段工作 | `docs/下一阶段工作.md` | 主线 / 次线未打勾条目数 |
| 3 路线图 | `docs/路线图.md`（或等价物） | 🔄 进行中 + 🆕 计划中节点数 |
| 4 蓝图 | `docs/蓝图.md`（或等价物） | 💭 仅探讨 状态条目数 |

按四层结果分情形（替代旧 A/B/C/D 的"阶段焦点告警"）：

| 情形 | 触发条件 | Q 列表必出 | 后续动作 |
|---|---|---|---|
| **A · 全空（项目无定义）** | 4 层文件**全部不存在** | "项目从未定义方向，是否进入'蓝图首次共建'对话？" | 拍板前禁接任何 M/L 级活，仅可接 S 级文档骨架建立类工作 |
| **B · 蓝图层耗尽（冻结）** | 蓝图存在但无 💭 条目，下游也都耗尽 | "进入冻结，仅接 S 级 / 还是开蓝图层对话？" | 仅接 S 级 |
| **C · 路线图层耗尽** | 蓝图有候选但路线图无可启动节点 | "选哪个蓝图条目展开到路线图？"（罗列所有 💭 候选） | 用户确认后展开（按 17 第 4 层动作） |
| **D · 下一阶段层耗尽** | 路线图有候选但下一阶段无未完成项 | "选哪个路线图节点展开到下一阶段工作？"（罗列所有 🆕/🔄 候选） | 用户确认后展开（按 17 第 3 层动作） |
| **E · 焦点散落** | 各层文件存在但焦点散落（如 23 条业务链路 .9 小节） | "是否在 `AGENTS.md` 建总索引？" + 提供索引草稿 | 可接 S/M，L 级先等索引完成 |
| **F · 健康** | 4 层都有可启动方向且映射清晰 | 无需特别 Q（但本次接活需明确 `parent_focus`） | 直接接活，按正常流程 |

**额外补刀**：如果本次接活方向**无法落到任何"下一阶段工作"焦点条目** → **必须**在 Q 列表加一项："本次活属于哪个阶段焦点？还是另开一项？"。这是 v0.4 八大宪法第 8 条"无方向不施工"的硬约束。

**为什么要做完整 4 层探查**：首次踏勘是项目入场仪式，必须把方向链全景一次摸清楚，建立 `AGENTS.md` 的"方向链指针"段（指向四层各自的当前位置）。后续每次接活只需顺手扫 4 层数（17 时机 A），不必每次重做完整探查。

### Step 5 · 用户对 Q 列表回复后的 L2 / L3 落地

> Step 4 已经把踏勘报告落盘了（L1）。Step 5 处理的是用户对 Q 列表回应后的 L2 骨架 / L3 git 操作。

用户回复 Q 列表后按其拍板结果执行：

1. **第二轮覆盖式重写 `docs/工地踏勘报告.md`**：去掉"待用户拍板"段，把决策结果作为正文 / 移到附录"决策记录"
2. **建立东合标准骨架**（L2，按用户逐项确认结果创建）：
   - `docs/任务卡/`（目录）
   - `docs/施工日志/`（目录）
   - `docs/踏勘历史/`（目录，存归档）
   - `docs/项目知识库.md`（按用户决定空骨架或种子注入）
   - `docs/工地禁忌.md`（空骨架）
   - `docs/下一阶段工作.md`（按 Step 4 焦点缺失告警的拍板结果创建）
   - `docs/蓝图.md` / `docs/路线图.md`（按用户对"等价物映射"的决定：A 新建薄壳指向已有文档 / B 不新建仅在 AGENTS.md 映射 / C 重命名已有文档——不推荐）
   - `AGENTS.md`（项目根，含 `<!-- donghe-construction-team: initialized YYYY-MM-DD · vX.Y -->` 标识 + 等价物映射表 + 子项目清单 + 阶段焦点指针）

3. **如果用户同意"项目文档规范化"** → 进入 `playbooks/15-项目文档规范化.md`

4. **L3 git 操作**（按用户拍板结果）：
   - 用户回 Q2=A → `git add` + commit 到当前分支
   - 用户回 Q2=B → `git switch -c <分支名>` 起新分支再 commit
   - 用户回 Q2=C → 不动 git，本轮只产出文档

5. 进入"四、接活契约六栏"或退场（看用户决定）

## 三、多队加入对齐流程（情形 B）

详见 `playbooks/14-多施工队协同.md` 的"加入流程"段。要点：

- 不重新踏勘，**信任** `docs/工地踏勘报告.md`

- 用户确认后再走"四、接活契约六栏"

## 四、接活契约六栏（每次接新需求都要写）

> v0.4.9：接活契约只做最小边界确认。除 `playbooks/20` 的"必须问"事项外，不要把文档整理、日志、知识库、轻量禁忌拿来逐项问用户。

**M / L 级建议附在任务卡 `## 上下文` 段开头作为存档**。

格式：用 `templates/接活契约模板.md` 的六栏：

| 栏位 | 内容 |
|---|---|
| **我理解的需求** | 用一句话复述用户要的东西 |
| **我准备改的范围** | 涉及哪些文件 / 模块 / 接口 |
| **我不会动的范围** | 明确说出"不会动 X、不会动 Y" |
| **涉及的不变量** | 这次改动可能影响的 SSOT、契约、AI 工具能力等 |
| **改动半径预判** | S / M / L（参考 `02-改动半径分级.md`） |
| **风险提示** | 可能踩坑的地方、需要用户确认的开放问题 |

写完先发给用户对一遍，得到默许或调整后再进入"任务卡"环节。

## 五、当用户的需求本身有问题

若契约写到一半发现：

- 需求会破坏一个已识别的不变量
- 需求和路线图、下一阶段工作不一致
- 需求触发了 `docs/工地禁忌.md` 里的某条禁忌
- 需求需要做明显的临时方案 / 硬编码才能实现

→ **优先反对，不要顺从地实现**。明确告诉用户："这个做法会撞 X 不变量 / 触发 Y 禁忌，我建议改成 Z"。

包工头的职业道德是：**对结果负责，不对'听话'负责**。

## 七、踏勘报告新鲜度自检

每次接活前**顺手检查**踏勘报告新鲜度（详见 `playbooks/13-工地踏勘对齐.md`）：

- 距上次踏勘 ≥ 4 周 → 提醒用户考虑重新踏勘
- 自上次踏勘后路线图有里程碑完成 → 强建议重新踏勘
